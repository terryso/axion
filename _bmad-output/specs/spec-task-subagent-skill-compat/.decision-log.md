# Decision Log

## 2026-06-14

Created `SPEC-task-subagent-skill-compat` in express/headless mode from the user request to support Claude Code `Task(...)` skill prompts in Axion. Loaded BMad workflow config from `.agents/skills/bmad-spec/customize.toml` through the resolver. The configured persistent fact path `project-context.md` did not exist at repository root, so the run used `_bmad-output/project-context.md`, which is included as an adopted companion via `../../project-context.md`.

Key decisions:

- Use `_bmad-output/specs/spec-task-subagent-skill-compat/` because BMad spec workflow resolves to `{output_folder}/specs/spec-{slug}`.
- Treat `Task` compatibility as a subagent tool compatibility problem, not a slash routing problem, because Axion already routes `/skill-name args` to `executeSkillStream`.
- Require a real tool named `Task`; relying on the model to translate Claude Code `Task(...)` snippets to SDK `Agent` tool is not testable enough.
- Prefer SDK-level `createTaskTool()` plus spawner detection/filtering updates, with an Axion-local wrapper only as fallback.
- Add direct skill package context to close the `references/workflow-steps.md` path gap without changing existing skill files.

Self-validation:

- Coherence pass: each capability has intent and success; non-goals are explicit; constraints bend implementation choices; success signal is demonstrable with a real BMAD pipeline run.
- Preservation pass: all load-bearing findings from code inspection are captured in `brownfield-analysis.md`; architecture and test implications are captured in companions; wrapper-only details such as raw command outputs are intentionally omitted.

## 2026-06-14 Follow-up Review

Triggered by user review of the uncommitted spec. Rechecked Claude Code skills, subagents, Agent SDK subagents, and MCP docs, then compared against current Axion and local SDK code.

Corrections:

- Reframed `Task` as a Claude Code compatibility alias for SDK `Agent`, not a distinct child-agent abstraction.
- Expanded the spec from “make Task callable” to “preserve Claude Code tool semantics enough for real skills”: skill/subagent tools, MCP, WebSearch/WebFetch, ToolSearch, `allowed-tools`, and diagnostics must be part of the compatibility design.
- Added a shared tool profile requirement so direct skill execution does not silently lose MCP/search/web/skill tools just because it uses `buildSkillAgent()`.
- Recorded that current SDK `allowed-tools` parsing is enum-only and can silently drop unknown/MCP/custom tool names; this must be fixed before claiming Claude Code skill parity.
- Added filesystem subagent definitions (`.claude/agents/*.md`) as a deferred parity gap rather than MVP scope.
- Rechecked the local `~/.agents/skills/bmad-story-pipeline`: `references/workflow-steps.md` uses current `/bmad-*` names, but `SKILL.md` still contains old `/bmad-bmm-*` and `/bmad-tea-*` Task prompts, so manual acceptance needs both files synchronized or aliases.
