# Decision Log

## 2026-06-16

Created `SPEC-arch-upgrade-workflow` in express mode from the user discussion about extending `arch` from query-only into an upgrade-capable detail workflow. Loaded BMad workflow config through `_bmad/scripts/resolve_customization.py`. The configured persistent fact path `project-context.md` did not exist at repository root, so this run used `_bmad-output/project-context.md` and referenced it as adopted companion `../../project-context.md`.

Key decisions:

- Use `_bmad-output/specs/spec-arch-upgrade-workflow/` because BMad spec workflow resolves to `{output_folder}/specs/spec-{slug}`.
- Keep `axion arch` read-only and script-friendly; put upgrade entry points behind `/arch` detail or a future explicit command.
- Treat Homebrew formula/cask as the only MVP automatic upgrade sources because they can be mapped to deterministic commands and mocked in tests.
- Keep MacPorts, App Store, direct `.app`, system app and unknown sources as manual guidance for MVP.
- Separate scan facts, upgrade planning and upgrade execution so `AppArchitectureScanService` remains side-effect free.
- Require exact command preview, explicit confirmation and post-upgrade rescan before claiming the architecture issue is resolved.

Update after reviewing the detail UX question about uninstall:

- Add `x 卸载` as a detail-page disposal action for safe, non-system `.app` bundle matches.
- Reuse the existing `/apps` uninstall planner/executor path instead of adding a second uninstall implementation under `/arch`.
- Keep CLI package uninstall, `brew uninstall`, `port uninstall`, system app removal and permanent deletion out of MVP.
- Require existing support-data review, high-risk approval, typed confirmation, trash semantics and undo manifest for any `/arch`-initiated App uninstall.

Self-validation:

- Coherence pass: each capability has an intent and a testable success criterion; constraints rule out unsafe automatic upgrades and real package-manager calls in tests; non-goals explicitly exclude broad software-update behavior.
- Preservation pass: the load-bearing discussion points are captured in `SPEC.md`; source-specific handling, UX states, protocol seams, execution flow and test plan are captured in `upgrade-workflow.md`; wrapper-only chat context and raw command outputs are intentionally omitted.
