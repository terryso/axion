## Story 逐一分析

### Story 19.1: 统一 Agent 执行入口

**亮点：**
- **AgentBuilder 作为唯一构建入口** — 新建 `Sources/AxionCLI/Services/AgentBuilder.swift`，所有 Agent 构建逻辑集中到 `build()` 方法，RunCommand 和 AgentRunner 均通过 `BuildConfig.forCLI()` / `BuildConfig.forAPI()` 调用
- **两个关键 Bug 修复：** (1) `skillRegistry` 从未传给 AgentOptions → SDK ToolRestrictionStack 永远 nil，(2) AgentRunner 不传 `tools` → SDK agent 在 API 模式零注册工具
- **BuildConfig 工厂模式** — `forCLI()` 和 `forAPI()` 封装了 CLI/API 差异（Playwright MCP server 仅 CLI 需要），清晰分离关注点
- **MCP-prefixed SafetyHook 统一** — RunCommand 和 AgentRunner 原本各自用不同的 tool name 格式（MCP-prefixed vs bare），统一为 MCP-prefixed

**挑战：**
- **[HIGH] ApiRunner.runSkillAgent() 预解析永远失败** — skill 未注册到 SkillRegistry 就调用 resolveExplicitSlashSkillRequest。修复：Review 中发现并修复
- **[HIGH] runSkillAgent() 使用 forCLI() 包含 Playwright** — API skill 路径不应包含 Playwright MCP server。修复：新增 `forAPISkill()` 工厂方法
- **[HIGH] Story File List 不准确** — 声称 AgentRunner.swift 为 MODIFIED 但实际已重命名为 ApiRunner.swift
- **范围违规：** 19.1 Review 中做了 rename（19.3 scope）和部分 dead code deletion（20.1 scope）

**关键设计决策：**
- `buildFullSystemPrompt()` 和 `buildSafetyHookRegistry()` 从 RunCommand/AgentRunner 提取到 AgentBuilder
- `explicitSkill` 保留在 BuildConfig 中用于 model override 和 MCP server 排除决策
- `--fast` mode 的 maxSteps/maxTokens 限制保留在 RunCommand 调用方

### Story 19.2: Skill 处理对齐 SwiftWork 模式

**亮点：**
- **预解析范式转换** — 从"system prompt 注入 skill.promptTemplate"转为"预解析为 user message"。System prompt 始终是通用 planner prompt，skill 内容通过 SkillTool.call() 解析后作为 user message 传递
- **resolveExplicitSlashSkillRequest() 完整实现** — 镜像 SwiftWork AgentBridge 的模式：创建 SkillTool → 调用 call() → 提取 prompt → 返回 user message
- **allowedTools 彻底移除** — 不再由应用层手动设置 allowedTools，完全交给 SDK 的 ToolRestrictionStack
- **API 路径统一** — runSkillAgent() 也使用预解析模式，消除了 CLI/API 行为差异

**挑战：**
- **[MEDIUM] 测试断言不够精确** — pre-resolution 内容验证使用 contains() 而非精确匹配。修复：加强断言
- **[MEDIUM] allowedTools 审计测试需更新** — 现有测试未覆盖"不设置 allowedTools"的验证。修复：新增审计测试

**关键洞察：** 预解析 vs system prompt 注入的核心区别不在于"把 skill prompt 放哪"，而在于"谁管理 skill 生命周期"。System prompt 注入模式下应用层要自己处理 ToolRestriction、SkillTool 创建、prompt 组装；预解析模式下这些都交给 SDK。

### Story 19.3: API 路径对齐

**亮点：**
- **AgentRunner → ApiRunner 重命名** — 消除与 SDK `Agent` 类的语义冲突，所有 6 个调用点 + 14 个测试引用 + 3 个注释全部更新
- **processStream() 提取** — 将 runAgent() 和 runSkillAgent() 中 ~120 行重复的流处理逻辑提取为共享函数，两个入口函数变为薄层
- **StreamResult 统一返回** — processStream() 返回标准化的 StreamResult struct，包含 totalSteps、durationMs、stepSummaries 等所有下游需要的数据
- **Clean test update** — APITypesTests 的 14 个 inferResultKind 引用批量更新，零回归

**挑战：**
- **[HIGH] MCPServerRunner SafetyHook 缺少 MCP 前缀** — `buildSafetyHookRegistry()` 使用 bare tool names，SDK 传递 MCP-prefixed names，导致 shared-seat-mode 前台工具阻止完全失效。修复：添加 MCP prefix mapping
- **[MEDIUM] Completion Notes 声称 forCLI() 但实际用 forAPISkill()** — 文档与实现不一致。修复：更新文档
- **[MEDIUM] inferResultKind 接受未使用的 output 参数** — 函数签名有参数但从未读取。修复：移除参数

**关键设计决策：**
- `runSkillAgent()` 保留为独立入口（不同的 BuildConfig 参数），不合并到 `runAgent()`
- `BuildConfig.forAPISkill()` 使用 forCLI() 基础但不包含 Playwright
- processStream() 是 private static function，不暴露给外部

## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "单一构建入口"消除跨路径 Bug**
AgentBuilder.build() 是唯一创建 Agent 的入口。之前 RunCommand 和 AgentRunner 各自构建导致"修了 CLI 忘了 API"的 Bug（skillRegistry、tools、SafetyHook）。现在修改一处即可。
**启示：** 当两个入口共享核心逻辑时，共享函数 > 代码复制。即使初期提取成本高，避免 Bug 的收益远超成本。

**2. "参照实现驱动重构"降低设计风险**
每个 Story 都以 SwiftWork 的 AgentBridge 为参照实现。19.1 参照 `configure()` 的单一入口模式，19.2 参照 `resolveExplicitSlashSkillRequest()` 的预解析模式。这消除了"设计-实现差距"——重构目标明确、可验证。
**启示：** 重构时如果有已验证的参照实现，直接对齐比重新设计更安全。

**3. "BuildConfig 工厂"优雅处理入口差异**
`forCLI()` / `forAPI()` / `forAPISkill()` 三个工厂方法封装了入口差异，共享 build() 方法不包含任何 if-cli-if-api 分支。这是 Strategy 模式的轻量实现。
**启示：** 当共享函数需要处理不同调用方差异时，工厂方法比参数枚举更清晰。

**4. "Review 发现的系统性 Bug 跨 Story 传播"**
19.1 Review 发现 runSkillAgent() 的预解析失败和 Playwright 泄漏，19.3 Review 发现 MCPServerRunner 的 SafetyHook MCP 前缀缺失。这些 Bug 存在已久但直到重构统一路径才暴露——重构的"发现 Bug"价值。
**启示：** 重构的价值不仅在消除重复，更在暴露隐藏 Bug。统一路径让历史 Bug 无处藏身。

### 挑战模式（需要改进）

**1. "范围违规"在连续两个 Epic 出现**
19.1 Review 中完成了 AgentRunner → ApiRunner 重命名（19.3 scope）和部分 dead code deletion（20.1 scope）。这和 Epic 18 中 18.2 的"声称完成但实际未覆盖"是同类问题——Story 边界模糊导致越界执行。
**根因：** AI Dev Agent 在 Review 阶段发现"顺手就能修"的问题时，倾向于立即修复而非标记留到后续 Story。
**改进：** Review 发现的修复如果超出 Story scope，应标记为 TODO 并创建后续 Story，而非直接实施。

**2. "SafetyHook MCP 前缀"是第三次出现同类 Bug**
- Epic 17（17.1）: RunCommand 的 SafetyHook 使用 MCP-prefixed names ✓
- Epic 18（18.3）: AgentRunner.runSkillAgent() 使用 bare names → 安全检查失效
- Epic 19（19.1）: 统一为 MCP-prefixed names
- Epic 19（19.3）: MCPServerRunner.buildSafetyHookRegistry() 仍然使用 bare names

同一类 Bug 出现了 4 次。每次修复一处就漏掉另一处。
**根因：** SafetyHook 的 tool name 格式（MCP-prefixed vs bare）没有统一文档化，只有 AgentBuilder.swift 中的注释提及。
**改进：** SafetyHook 的 tool name 格式规则应写入 `project-context.md`，作为 persistent fact 在每个 Story 加载。

**3. "Story File List 与实际不一致"再次出现**
19.1 的 File List 声称 AgentRunner.swift 是 MODIFIED 但实际已重命名为 ApiRunner.swift。19.3 的 Completion Notes 声称使用 forCLI() 但实际用 forAPISkill()。文档与代码不一致。
**根因：** Completion Notes 在开发过程中编写，但后续 Review 修改了实现而未同步更新 Notes。
**改进：** Review 修改实现后必须同步更新 Completion Notes 和 File List。

**4. "测试计数波动"持续存在**
19.1 报告 1601 tests, 19.2 报告 1444 tests, 19.3 报告 1605 tests，最终 Review 后 1311 tests pass。测试总数在同一代码库中不应有如此大波动。
**根因：** 不同 Story 运行测试时的 filter 参数不同（AxionCLITests vs AxionHelperTests），导致计数不一致。
**改进：** Completion Notes 应记录完整的测试命令和 filter 参数，使计数可复现。Epic 18 行动项 3（使用 `grep -c "@Test"` 统计）仍未执行。

### 突破时刻

**1. ToolRestrictionStack 生效——Skill 系统真正可用**
Epic 17-18 建立了 Skill 系统的完整功能，但 ToolRestrictionStack 从未生效（skillRegistry 未传递）。Epic 19.1 的修复让 SDK 的工具限制真正执行——skill 声明 `toolRestrictions: [.bash]` 时，MCP 工具（screenshot、type_text）确实被过滤了。这是 Skill 系统从"看起来能用"到"真正可靠"的关键转折。

**2. 预解析范式简化了整个 Skill 调用链**
从"应用层构建 skill-specific system prompt + 手动设 allowedTools + 手动管理 MCP servers"简化为"预解析 skill 为 user message → SDK 管理一切"。删除了 buildSkillSystemPrompt()、移除了 allowedTools 设置、简化了 MCP server 条件逻辑。代码量减少，正确性提高。

**3. processStream() 提取验证了"薄层"架构**
runAgent() 和 runSkillAgent() 各自从 ~180 行减少到 ~60 行，核心逻辑（SSE、CostTracker、RunTracker）全部集中在 processStream()。这验证了 phase6-refactor-architecture.md 中的"应用层薄层"设计——入口只负责参数准备和调用共享处理器。

## 关键经验

1. **重构的真正价值在于暴露隐藏 Bug** — Epic 19 的核心任务是消除代码重复，但发现了 3 个比重复更严重的问题：skillRegistry 从未传递、API 路径零注册工具、SafetyHook MCP 前缀不一致。这些 Bug 存在已久但直到统一路径才暴露。教训：重构不是"让代码更好看"，而是"让隐藏 Bug 无处藏身"。

2. **"参照实现驱动"是重构的最安全策略** — 每个 Story 都对齐 SwiftWork 的已知正确模式，而非重新设计。这消除了"设计阶段觉得完美、实现阶段发现漏洞"的常见重构陷阱。教训：当有可参照的正确实现时，直接对齐 > 重新设计。

3. **SafetyHook 的 tool name 格式是"隐性知识"** — MCP-prefixed vs bare names 的规则从未文档化，只存在于 AgentBuilder.swift 的注释中。这导致同类 Bug 在 4 个位置独立出现、独立修复。教训：当同一个 Bug 反复出现时，根因通常不是"遗漏"而是"知识未文档化"。

4. **BuildConfig 工厂模式优雅解决了"统一 + 差异"矛盾** — 共享 build() 方法无分支，差异通过 BuildConfig 工厂方法封装。这比在 build() 内部加 if-cli-if-api 判断更清晰、更易扩展。教训：Strategy 模式的轻量实现（工厂方法 + 共享函数）是处理入口差异的最佳实践。

5. **Review 阶段的范围自律是持续交付的保障** — 19.1 Review 越界完成了 rename 和 dead code deletion，虽然修复了实际问题，但模糊了 Story 边界，使得进度跟踪不准确。教训：Review 的职责是发现问题并记录，不是在当前 Story 中解决所有问题。
