## Story 逐一分析

### Story 16.1: launchd Daemon 支持

**亮点：**
- DaemonService 使用注入 `@Sendable` 闭包实现 launchctl 调用的可测试性——纯 Swift 函数式注入，不依赖 protocol/mock 框架
- plist XML 生成包含完整配置：Label、ProgramArguments、RunAtLoad、KeepAlive（Crashed=true）、ThrottleInterval=10、StandardOutPath/StandardErrorPath
- DaemonCommand 嵌套子命令设计清晰：install/status/uninstall 三个子命令，参数解析完整
- ServerCommand authKey 优先级链：CLI `--auth-key` > 环境变量 `AXION_AUTH_KEY` > nil——优雅的环境变量 fallback
- XML 转义辅助方法 `escapeXML()` 处理 &, <, >, ", ' 五种特殊字符
- LaunchctlCallCollector 使用 NSLock 实现线程安全的调用记录——解决 Sendable 闭包捕获问题

**挑战：**
- **HIGH: AC3 ThrottleInterval 缺失** — plist 初始版本未包含 ThrottleInterval，不满足 AC3 "10秒后重启"。修复：添加 `<key>ThrottleInterval</key><integer>10</integer>`。
- **MEDIUM: install() 目录创建顺序错误** — `~/Library/LaunchAgents/` 目录创建在 plist 写入之后，可能导致首次安装失败。修复：移到写入之前。
- **MEDIUM: install() 无回滚机制** — bootstrap 或 kickstart 失败时 plist 文件残留。修复：失败时自动清理已写入的 plist。
- **MEDIUM: DaemonInstallCommand 缺少端口验证** — 未限制 port 范围。修复：添加 `validate()` 方法，限制 1-65535。
- **LOW: 测试计数不准确** — Dev Notes 声称 17 tests 但实际有 22 个。更新文档。

**已知限制：**
- AC3 "连续崩溃5次后停止重启并记录日志"：launchd 不原生支持此行为。使用 `ThrottleInterval: 10` + `KeepAlive: { Crashed: true }` 满足核心意图。crash counter 作为后续优化。

### Story 16.2: API Server 持久化运行恢复

**亮点：**
- RunPersistenceService 使用 Sendable struct（非 actor），所有方法为纯函数调用——设计轻量，无并发开销
- 存储路径分离：API 运行状态用 `~/.axion/api-runs/`，CLI trace 用 `~/.axion/runs/`——避免两个系统互相干扰
- 原子写入策略：先写 `.tmp` 文件再 rename，POSIX 保证同文件系统下 rename 原子性——防止崩溃时文件损坏
- PersistedSSEEvent Codable wrapper 巧妙桥接 SSEEvent enum 到 JSONL 格式——解决了 enum 不可直接 Codable 的问题
- RunRecoveryService 作为 enum（无实例），只提供 static 方法——无状态恢复逻辑的恰当抽象
- 恢复状态映射完整覆盖 7 种 APIRunStatus，intervention_needed 状态正确保持不变
- SSE 历史重放支持磁盘回退——内存 buffer miss 时从 api-events.jsonl 加载

**挑战：**
- **MEDIUM: subscribeWithReplay 磁盘回退不缓存** — 每次新订阅者都触发磁盘读取而非缓存已加载事件。修复：首次从磁盘加载后写入 replayBuffer，后续订阅者使用缓存。
- **MEDIUM: 缺少 resuming/userTakeover 恢复测试** — AC4 要求 resuming 和 userTakeover 标记为 failed，但测试仅覆盖 running/queued。修复：新增 2 个测试。
- **LOW: Dev notes 测试计数不准确** — 声称 17 tests 但实际 20 个，Review 后 22 个。更新文档。

## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "注入闭包"模式统一了系统调用的可测试性**
DaemonService 的 launchctl 调用通过 `@Sendable (String) throws -> String` 闭包注入，测试中使用 LaunchctlCallCollector 记录调用。这与 TakeoverIO 的 readLine/writeLine 注入模式一脉相承，已成为 Axion 项目处理系统调用的标准模式。
**启示：** 当外部依赖（launchctl、stdin/stdout、文件系统）不可在单元测试中直接调用时，注入闭包比创建 protocol + mock 更轻量、更 Swift 惯用。

**2. "Sendable struct 纯函数"模式优于 actor 过度使用**
RunPersistenceService 是 Sendable struct，所有方法操作 FileManager（线程安全），无并发状态需要隔离。相比使用 actor，避免了不必要的 await 开销和死锁风险。
**启示：** 不是所有服务都需要 actor 隔离。当服务无内部可变状态且底层 API（如 FileManager）已线程安全时，Sendable struct 是更好的选择。

**3. "存储路径分离"策略防止系统间干扰**
`~/.axion/api-runs/` 与 `~/.axion/runs/` 的分离确保 RunPersistenceService 和 TraceRecorder 互不干扰。两个系统有不同的写入模式（覆写 vs 追加）、不同的文件格式（JSON vs JSONL）、不同的清理策略。
**启示：** 当两个子系统需要持久化到同一根目录但有不同的文件结构和生命周期时，子目录分离比共享目录更安全。

**4. 原子写入在守护进程场景下是必须的**
daemon 模式下进程可能在任意时刻崩溃（OOM、SIGKILL、内核 panic）。RunPersistenceService 的原子写入（write-to-tmp + rename）确保 `api-output.json` 永远处于一致状态——要么是旧版本，要么是新版本，不会出现半写。
**启示：** 在 daemon/服务场景下，文件写入的原子性不是优化，而是正确性要求。

**5. Review 发现问题模式保持稳定**
16.1 和 16.2 的 Review findings 分别为 5 个和 3 个，与 Epic 15 的"每 Story 5 个 findings"模式一致。但 16.2 的 findings 数量减少（3 vs 5），且没有 HIGH severity——说明代码质量在 Epic 内逐步提升。

### 挑战模式（需要改进）

**1. plist 生成中的遗漏配置项**
16.1 的 HIGH finding 是 ThrottleInterval 缺失——AC3 明确要求"10秒后重启"，但初始实现只设置了 KeepAlive 而未设置重启间隔。这是对 AC 需求的解读不够精确。
**根因：** KeepAlive 和 ThrottleInterval 是 launchd plist 中两个独立的配置项，开发者对 launchd 配置语义理解不完整。
**改进：** 涉及外部系统配置（launchd、Hummingbird、Network）的 AC，Dev Notes 应增加配置项与 AC 的映射表，确保每个 AC 都有对应的配置项覆盖。

**2. install() 的错误处理链不完整**
16.1 的两个 MEDIUM findings（目录创建顺序和回滚缺失）都是 install 流程的错误处理问题。install 涉及多个步骤（创建目录 → 写 plist → bootstrap → kickstart），任何一步失败都需要正确的回滚。
**根因：** 多步骤操作的事务性保证不足——缺少"全部成功或全部回滚"的语义。
**改进：** 多步骤操作应在 Dev Notes 中增加操作序列图，明确每步失败时的回滚策略。

**3. 磁盘缓存的首次加载遗漏**
16.2 的 subscribeWithReplay 磁盘回退不缓存——首次加载后未写入 replayBuffer，导致每个新订阅者都重复读取磁盘。这是典型的"忘记缓存已加载数据"问题。
**根因：** SSE 订阅路径中，磁盘加载被视为一次性的 fallback 而非正式的数据源。
**改进：** 任何"从外部源加载数据"的操作（磁盘、网络、数据库），加载后都应考虑是否需要缓存到内存，并在 Dev Notes 中明确缓存策略。

**4. 测试计数偏差仍是低频问题**
16.1 和 16.2 的 Dev Notes 都有测试计数不准确的问题。这是从 Epic 14 延续的模式——Completion Notes 中的测试计数基于"估计"而非"实际文件统计"。
**根因：** 测试计数来自开发过程中的估计，而非最终文件扫描。
**改进：** Completion Notes 应使用 `grep -c "@Test" Tests/...` 命令统计实际测试数量，而非手动估计。

### 突破时刻

**1. LaunchctlCallCollector 解决 Sendable 闭包捕获问题**
DaemonService 的 launchctl 闭包是 `@Sendable`，但测试需要记录调用参数。LaunchctlCallCollector 使用 NSLock + mutable array 在 Sendable 闭包中安全捕获调用参数——这是一个精巧的并发安全模式。

**2. 原子写入与 JSONL 追加的混合策略**
`api-output.json` 使用覆写（每次状态更新都是完整快照），`api-events.jsonl` 使用追加（每个事件是独立行）。两种策略分别适用于不同的数据特性：状态是"当前值"（覆写），事件是"历史序列"（追加）。

**3. 恢复状态映射的"不干预"原则**
RunRecoveryService 对 intervention_needed 状态保持不变——这是一个关键的"不干预"设计决策。在 daemon 崩溃恢复场景下，过度干预（如自动标记为 failed）可能丢失用户的干预上下文。

## 关键经验

1. **原子写入是守护进程场景下的正确性要求，不是性能优化** — daemon 模式下进程可能在任意时刻崩溃。RunPersistenceService 的 write-to-tmp + rename 策略确保 `api-output.json` 永远处于一致状态。教训：在 daemon/服务场景下，所有关键文件的写入都必须使用原子操作。

2. **"Sendable struct 纯函数"是 actor 的正确替代方案** — RunPersistenceService 使用 Sendable struct 而非 actor，因为 FileManager 本身线程安全且服务无内部可变状态。教训：不要过度使用 actor——当服务无并发状态且底层 API 已线程安全时，Sendable struct 更简洁高效。

3. **注入闭包已成为 Axion 处理系统调用可测试性的标准模式** — TakeoverIO（stdin/stdout）、DaemonService（launchctl）、DaemonService（resolveAxionBin）都使用 `@Sendable` 闭包注入。这比 protocol + mock 更轻量，更 Swift 惯用。教训：当外部依赖是单个函数调用（而非复杂接口）时，闭包注入优于 protocol 抽象。

4. **Phase 4 证明了"安全优先、质量驱动"的策略有效** — 先在 Epic 13 建立安全基础设施（运行锁、视觉增量、成本控制），再在 Epic 14-16 构建生产级 API。安全基础设施不需要额外推广——API 层自然复用了 RunTracker、EventBroadcaster、StandardTaskOutput。这种"基础设施先于功能"的策略确保了每个 Epic 的实现都建立在稳定的基础上。
