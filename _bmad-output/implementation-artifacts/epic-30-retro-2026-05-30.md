## What Could Be Improved

### C1: Critical Bugs in Review Point to Incomplete Self-Checking

Story 30.3's CRITICAL bug (idle timing) and Story 30.2's HIGH bug (EventBus lifecycle) were both fundamental logic errors that should have been caught during implementation, not review. The idle timing bug in particular — setting `_lastTaskAt` before checking `shouldCurate()` — is the kind of "off-by-one" error that a quick mental walkthrough would catch.

**Recommendation:** Before marking a story for review, implementers should walk through each acceptance criterion step-by-step in code, verifying the logic path actually works. Add this to the story template as a self-check step.

### C2: EventBus Lifecycle Not Documented as an Architectural Constraint

The discovery that per-request EventBus doesn't span detached Task execution was found during Story 30.2 implementation, not during design. This is a fundamental architectural constraint that should have been documented when the EventBus was designed.

**Recommendation:** Add a "Detached Task Constraints" section to architecture.md documenting that: (a) per-request EventBus lifecycle ends before detached Tasks complete, (b) detached Tasks should use direct callbacks for cross-component communication, (c) EventBus is only reliable for synchronous-within-request-path communication.

### C3: Test Coverage Masked Timing Bugs (Story 30.3 L1)

All initial CuratorScheduler handle() tests used `curatorIdleHours: 0.0`, which meant the idle check always passed — masking the critical timing bug. Tests should use realistic default values to catch timing-related logic errors.

**Recommendation:** Scheduler tests should default to non-zero interval values that force the test to explicitly set up conditions matching the check logic.

## Lessons Learned

### L1: Detached Tasks Must Use Direct Callbacks, Not EventBus

Story 30.2's HIGH finding proved that EventBus is request-scoped. When a detached Task outlives the request, the EventBus it captured may be stopped. Direct callbacks (wired at construction time in GatewayCommand) are the reliable pattern for cross-component communication from background tasks.

**Key Insight:** The architectural rule is: **EventBus for synchronous event propagation within a request lifecycle. Direct callbacks for anything that runs in detached Tasks.**

### L2: Scheduler "Now" vs "Previous" Timing Is Tricky

Story 30.3's CRITICAL bug showed that updating `_lastTaskAt = now` before checking `shouldCurate()` resets the idle timer to 0, making the idle condition always false. The fix was to capture the previous `_lastTaskAt` value before updating. This is a common off-by-one pattern in scheduler implementations.

**Key Insight:** Scheduler state updates should happen AFTER condition checks, not before. Or: capture previous state, check against previous, then update.

### L3: Test Defaults Should Match Production Defaults

Story 30.3's tests used `curatorIdleHours: 0.0` which made every check pass. Tests should use realistic defaults (e.g., 2.0 hours) and explicitly set conditions to exercise both the trigger and non-trigger paths.

**Key Insight:** When testing time-based conditions, use non-zero defaults that force explicit condition setup. Zero-value defaults mask logic errors.

### L4: GatewayCommand Is Becoming a God Object

GatewayStartCommand now assembles: AxionRuntime, AxionAPI, TelegramAdapter, TaskSerialQueue, TGEventHandler, TGCommandRouter, ReviewScheduler, CuratorScheduler, IntelligentCurator (with 6 sub-dependencies). The file is ~400+ lines and growing.

**Key Insight:** Consider extracting GatewayServiceFactory or similar to manage the construction of the Gateway component graph, separating wiring from command logic.

### L5: ReviewScheduler ↔ CuratorScheduler Independence Works Well

Both schedulers listen to AgentCompletedEvent independently, check different conditions, and execute different background tasks. There's no coupling between them. This validates the "independent EventHandler actors" architectural pattern.

**Key Insight:** Multiple EventHandlers subscribing to the same event is safe and clean when they have completely independent responsibilities.
