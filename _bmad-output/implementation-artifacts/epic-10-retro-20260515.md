## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "独立进程 + HTTP API" 通信架构清晰可靠**
- AxionBar 作为独立 macOS App 通过 HTTP API 与 CLI 后端通信
- 两个进程完全解耦：AxionBar 崩溃不影响后端，后端崩溃 AxionBar 自动检测并提示重启
- 这种松耦合架构使得 UI 层可以独立演进、独立分发
- **启示：** GUI 壳与核心引擎应通过 API 通信，而非进程内调用

**2. "Bar 前缀模型" 解耦策略**
- AxionBar 定义自己的 API 模型（BarCreateRunRequest、BarRunStatusResponse 等）
- 不 import AxionCLI，仅依赖 AxionCore
- UI 层只需后端 API 的字段子集，不需要完整的后端模型
- **启示：** 不同进程/target 应定义自己的 API 视图模型，通过 JSON 序列化通信

**3. NSPanel/NSWindow + SwiftUI hosting 窗口模式统一**
- 所有弹出窗口（QuickRunWindow、TaskDetailPanel、RunHistoryWindow、SettingsWindow）使用同一模式
- NSWindow 提供精细控制（floating、level、styleMask），SwiftUI 提供声明式 UI
- **启示：** 菜单栏 App 的窗口管理应统一使用 NSWindow + SwiftUI hosting，而非 WindowGroup

**4. Review 发现并修复了关键遗漏**
- Story 10.1 的两个 CRITICAL 问题（LSUIElement 未实现、AC5 不工作）都是功能性的遗漏
- Story 10.2 的 SSE 监控集中化缺失是架构层面的遗漏
- **启示：** 涉及 AC 验收标准的 Story 完成后必须逐条核对 AC

### 挑战模式（需要改进）

**1. AC 验收标准未逐条验证**
- Story 10.1 的 AC5（后端断连处理）在初始实现中完全不工作
- Story 10.2 的 AC2（进度显示）和 AC4（完成通知）依赖 SSE 监控集中化，但初始实现遗漏了
- **根因：** 开发者关注主路径（happy path）实现，未系统性地逐条验证每个 AC
- **改进：** 每个 Story 完成后建立 AC checklist 逐条验证

**2. 两套菜单实现增加维护负担**
- MenuBarBuilder（NSMenu）和 AxionBarMenuContent（SwiftUI）需要同步更新
- 三个 Story 中每次修改菜单都需要同时更新两个文件
- **根因：** Story 10.1 同时实现了两种方案"提供灵活性"，但实际只使用 SwiftUI 版本
- **改进：** 确定主要菜单实现方式，将另一个标记为 deprecated 或移除

**3. Review 中反复出现测试质量问题**
- HotkeyConfigManagerTests 写入真实文件系统（~/.axion/hotkeys.json）
- 初始测试计数不准确（10.1 声称 17 tests 实际 47 tests）
- **根因：** 测试实现时未考虑隔离性，计数来源不明确
- **改进：** 测试必须使用临时目录，不接受真实文件系统写入

**4. SSE 事件处理需要更早的架构设计**
- Story 10.2 的 SSE 监控应该是 Story 级别的架构决策，而不是 review 中才发现的遗漏
- SSE 连接生命周期管理在三个地方分散（StatusBarController、TaskDetailPanel、SSEEventClient）
- **根因：** 初始设计未考虑 SSE 订阅的中心化管理
- **改进：** 涉及实时事件流的 Story 应在设计阶段明确"谁来管理连接生命周期"

### 突破时刻

**1. 技能执行复用 SSE 管线 — 架构复用的典范**
Story 10.3 决定技能执行走现有 RunTracker + EventBroadcaster 管线，而非新建通信通道。这意味着：
- 技能执行结果通过 SSE 推送到 AxionBar
- 复用 TaskDetailPanel 显示技能执行进度
- 复用 StatusBarController 的 running 状态管理
- 零新增通信基础设施

**2. "Bar 前缀模型"成为跨进程 API 集成的标准模式**
AxionBar 的 API 模型命名约定（Bar 前缀）解决了跨 target 模型重名的问题。这种模式可以推广到未来的其他 GUI 客户端。

**3. NSEvent 全局监听方案选择精准**
放弃 CGEvent Tap（需要硬性 Accessibility 权限依赖）选择 NSEvent.addGlobalMonitorForEvents（更轻量、权限要求更灵活），体现了对 macOS API 的深入理解。

## 关键经验

1. **AC 逐条验证是防止功能遗漏的最后防线** — Epic 10 最深刻的教训：Story 10.1 的 AC5（后端断连处理）在初始实现中完全不工作，直到 review 才发现。Story 10.2 的 AC2（进度显示）和 AC4（完成通知）同样遗漏。根因是开发者专注主路径实现，未系统性地逐条验证每个 AC。每个 Story 完成后必须建立 AC checklist，逐条确认每个 AC 有明确的验证证据。

2. **"独立进程 + HTTP API" 是 macOS GUI 壳与 CLI 引擎的理想架构** — AxionBar 通过 HTTP API 与 axion server 通信，两个进程完全解耦。这种架构带来三大好处：(1) UI 崩溃不影响正在运行的任务；(2) 后端崩溃 UI 可以检测并提示重启；(3) UI 和后端可以独立迭代和分发。这种模式值得推广到所有"GUI 壳 + CLI 核心"的场景。

3. **两套实现的"灵活性"是隐性的技术债务** — Story 10.1 同时实现了 NSMenu 和 SwiftUI 两套菜单，初衷是"提供灵活性"。但三个 Story 中每次修改都需要同步更新两个文件，实际增加了开发成本。结论：除非有明确的"两种方案都需要"的需求，应选择一种方案并坚持，而非同时维护两套。

4. **SSE 事件流的连接生命周期需要中心化管理** — Story 10.2 的核心遗漏是 SSE 订阅散落在多个组件中（StatusBarController、TaskDetailPanel），没有统一的生命周期管理。Review 后引入 startRunMonitoring() 集中管理 SSE 订阅，但这一架构决策本应在设计阶段明确。教训：涉及实时事件流的架构应早期明确"谁管理连接"。

5. **keyCode 映射等"看似简单"的代码最容易出错** — Story 10.3 的 keyCodeToString 映射有 8 个 key code 映射错误。这类映射表代码看似简单，实际上是"代码中的数据"，需要逐项与 Apple 官方 kVK_* 常量核对。教训：涉及平台 API 常量映射的代码应参考官方文档逐项验证，不能"凭记忆编写"。
