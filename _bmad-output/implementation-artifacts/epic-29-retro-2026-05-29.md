## What Went Well

### W1: Single-Model Execution Produced Highly Consistent Code

All 5 stories were implemented by GLM-5.1, resulting in consistent patterns across the entire epic: actor isolation, protocol extraction for testability, closure-based dependency injection, and `nonisolated(unsafe)` for cross-actor reads. Unlike Epic 28 which mixed GLM-5.1 and Claude Opus 4.7, this single-model approach eliminated the "discovery cost" of cross-model handoffs.

### W2: Protocol Extraction for Testability Is Now Automatic

Every new component extracted a protocol for test injection: `TGAPIClientProtocol`, `TaskSerialQueueProtocol`, `TGEventHandlerProtocol` (later removed as dead code). The `DaemonRuntimeManaging` protocol was extended with backward-compatible `extraHandlers` parameter. This practice is now deeply internalized — no review findings about "missing protocol abstraction" for the 2nd consecutive epic.

### W3: Sequential Story Dependencies Were Perfectly Ordered

Stories 29.1 → 29.2 → 29.3 → 29.4 → 29.5 built naturally on each other with zero rework caused by ordering mistakes. Story 29.2 (queue) correctly modified the adapter from 29.1. Story 29.3 (event handler) correctly modified the queue from 29.2. Story 29.4 (commands) correctly extended the adapter without breaking queue integration. This is the gold standard for epic story sequencing.

### W4: Closure-Based Dependency Injection Avoided Circular Dependencies

The `replyHandler` closure pattern in TaskSerialQueue, `sendMessage` closure in TGEventHandler, and `statusProvider`/`skillsProvider` closures in TGCommandRouter all avoided direct references between actors. This eliminated potential circular dependency issues and made every component independently testable. The weak reference pattern (`[weak adapter]`) in GatewayCommand was correctly applied to prevent retain cycles.

### W5: Security Rules Were Consistently Applied

Unauthorized message silencing (no reply, no content logging), token-via-environment-variable (never in config.json), API key exclusion from error messages, and authorization-before-routing ordering were all correctly implemented across all stories without any review findings.

### C2: Review Finding Rate Elevated — ~23 Total Across 5 Stories

Average 4.6 findings per story, with HIGH findings in 3 out of 5 stories. Common patterns:

- **Duplicate message sending** (29.3 HIGH): TGEventHandler and TaskSerialQueue both sent completion/error messages to the user
- **Recursive stack risk** (29.2 HIGH): `processNext()` was recursive before review caught it
- **Weak assertions** (29.1, 29.4 MEDIUM): `#expect(true)` placeholders, overly broad `#expect(text.contains("0"))`
- **Missing edge cases in tests** (29.4 MEDIUM): No test for commands with trailing arguments or @botname suffix

**Recommendation:** Add a pre-review self-check: "Does every test assert something specific? Are all message sending paths unique (no duplicates)?"

Story 29.5 (TG image support) has no Dev Agent Record, no review section, no completion notes, and its status in the story file header says "in-progress" despite being marked "done" in sprint-status.yaml and having a completed git commit (`0b84a6a`). **Impact:** Medium — future developers (or AI agents) have no context on what was actually implemented, what challenges arose, or what review findings existed.

**Recommendation:** Ensure every completed story has at minimum: completion notes, file list, and review results.

### C4: Recursive processNext Despite Explicit Warning From Epic 28

Epic 28 retro L1 stated: "When an anti-pattern is identified in Dev Notes, provide the correct implementation path as a task, not just a warning." Story 29.2's Dev Notes did not explicitly warn against recursion, but the review caught a recursive `processNext()` implementation and converted it to a while loop. The pattern of "implementation-first, review-catches" continues.

**Recommendation:** Story Dev Notes should include implementation constraints (e.g., "use while loop, not recursion") for known risky patterns.

## Lessons Learned

### L1: Single-Model Execution Is More Consistent Than Cross-Model

Epic 29 used only GLM-5.1 across all 5 stories. The resulting code is more internally consistent than Epic 28's GLM + Opus mix. Cross-model handoffs require extra Dev Notes context to bridge model-specific patterns; single-model avoids this overhead.

**Key Insight:** When stories are tightly sequential (each modifies files from the previous), single-model execution reduces friction. Reserve cross-model collaboration for parallel or independent story tracks.

### L2: Duplicate Message Paths Are a Systemic Risk in Event-Driven Architectures

Story 29.3's HIGH finding (duplicate completion/error messages from both TGEventHandler and TaskSerialQueue) revealed a systemic risk: when multiple components can send messages to the same channel, it's easy to create duplicate paths. The fix required removing message sending from TaskSerialQueue and delegating entirely to TGEventHandler.

**Key Insight:** Designate a single "message owner" for each communication channel. Document which component is responsible for which message type.

### L3: Protocol Extension With Default Parameters Enables Backward Compatibility

The `DaemonRuntimeManaging` protocol extension pattern (adding `extraHandlers: [any EventHandler] = []` with a default implementation) allowed Story 29.3 to modify the protocol without breaking existing callers. This is a validated pattern for evolving protocols in a Swift codebase.

**Key Insight:** Protocol evolution should use default-parameter overloads rather than modifying existing method signatures.

### L4: Story 29.5's Minimal Documentation Is a Warning Signal

Story 29.5 was the last story in the epic, implemented quickly, and left with minimal documentation. This is a common "end-of-epic rush" pattern. The story was functionally complete but lacks the context richness of earlier stories.

**Key Insight:** Last stories in an epic need the same documentation rigor as first stories. Consider a documentation completeness check before marking a story "done."

### L5: While-Loop > Recursion for Async Task Processing

Story 29.2 initially used recursive `processNext()` calls for queue processing. Review correctly identified this as a stack depth risk and converted to a while loop. This is the second time this pattern appeared (also caught in Epic 28).

**Key Insight:** Queue/task processing should always use while loops, not recursion. Add this to the project coding conventions.
