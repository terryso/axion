## What Could Be Improved

### C1: High-Severity Issues Found in Code Review for Every Story

Every story had at least 3 HIGH-severity issues found during code review. While it's good they were caught, the pattern suggests the initial implementation quality could be higher. Common categories:

- **Incomplete wiring:** Features implemented but not connected (CostTracker budget check never called, stop() was no-op)
- **Lost data in mapping:** Tool names dropped during SDKMessage → SSEEvent mapping
- **Missing error recovery:** Agent stream failures left state machines stuck

**Root Cause:** The development agent focused on getting individual components working in isolation but didn't consistently verify end-to-end integration paths.

**Recommendation:** Add an explicit "integration path walkthrough" step to the dev workflow — trace the data flow from entry point to exit for each acceptance criterion before marking implementation complete.

### C2: M1 Fix from Story 20.2 Should Have Been Caught Earlier

The `ISO8601DateFormatter` being allocated per `record()` call was identified as a medium-severity performance issue in Story 20.2's review. This same pattern should have been checked when TraceRecorder was first written. The fix (making it an instance property on the actor) is straightforward, but the issue consumed review cycles.

**Recommendation:** Add a "Performance Anti-Patterns" checklist to story dev notes for stories involving actors with frequently-called methods. Items: date formatter allocation, string interpolation in hot paths, unnecessary copies of large collections.

### C3: Test Count Accuracy in Completion Notes

Story 20.2 had a discrepancy: completion notes said 25 new tests, review found 39 actual tests. Story 20.4 had a similar issue (34 vs 44). This indicates the dev agent isn't accurately counting tests before writing completion notes.

**Recommendation:** Make test counting a mandatory step — `grep -c "func test" <file>` or `swift test 2>&1 | grep -c "Test case"` — before writing completion notes.

### C4: No E2E Tests for HTTP Server

All 4 stories explicitly state "No E2E tests" in their testing strategy. While this is justified for utility components, the HTTP server is a user-facing integration point. The unit tests cover individual components well (RunTracker state machine, EventBroadcaster fan-out, etc.), but there's no test that starts a server, submits a real run, and verifies SSE events arrive correctly.

**Risk:** Integration issues between Hummingbird routing, RunTracker, EventBroadcaster, and Agent streaming are not caught by unit tests.

**Recommendation:** Consider adding a minimal integration test that starts an AgentHTTPServer on a random port with a mock agent and verifies the POST → SSE → GET lifecycle works end-to-end.

## Technical Debt Incurred

### TD1: Tool Name Lost in Trace Events

Story 20.2 review noted (M3) that `toolResult` trace events don't include the tool name because `ToolResultData` doesn't carry a `toolName` field. The `toolNameMap` pattern was added to the HTTP server (Story 20.1) to work around this, but TraceRecorder doesn't have the same workaround.

**Impact:** Trace files show step_done events without tool names for the toolResult step, making trace analysis less useful.

**Priority:** Medium — affects observability but doesn't break functionality.

### TD2: AgentOptions Deferred Fields Growing

Epic 17 added 5 deferred fields to AgentOptions. Epic 20 added 2 more (`traceEnabled`, `traceBaseURL`). Each addition follows the same pattern: declared in the type but not fully wired in all agent execution paths. The CostTracker integration (Story 20.2) demonstrated this risk — the budget check was implemented but never actually called in the stream path.

**Impact:** Risk that options are declared but not honored in all code paths.

**Priority:** Low — each story's review catches wiring gaps, but the pattern is recurring.

### TD3: Legacy Memory Files Not Cleaned Up

Story 20.3's lazy migration creates `{domain}-facts.json` files alongside legacy `{domain}.json` files but doesn't delete the old files. Over time, this doubles the memory directory size and creates confusion about which file is authoritative.

**Impact:** Minor storage waste, potential confusion during debugging.

**Priority:** Low — can be addressed when Memory system gets wider adoption.

## Lessons Learned

### L1: Struct vs Actor Decision Has Architectural Consequences

The CostTracker was deliberately designed as a `Sendable struct` (not an actor) because each `stream()` call creates its own instance mutated in a single-threaded loop. This decision simplified testing but meant budget checks had to be explicitly called in the loop — they weren't automatically triggered by state changes. The review caught the missing calls, but the pattern is worth documenting: **structs with mutating methods need explicit call-site discipline that actors enforce automatically.**

### L2: "Purely Additive" Stories Are Lower Risk

Stories 20.3 and 20.4 modified zero existing files. This meant zero regression risk and zero blast radius. Future epics should aim for this pattern where possible — new modules with clean boundaries are safer than modifying shared code.

### L3: Hummingbird 2.x API Requires Specific Knowledge

Story 20.1's dev notes document several Hummingbird 2.x API discoveries (parameter access via `context.parameters`, flat route path strings, `ResponseBody` buffer requirements). These are now documented in the story file, but they're not in any shared reference. Future stories using Hummingbird will benefit from a consolidated reference.

### L4: Deterministic Hashing (djb2) for Entity Identity

Story 20.3's `MemoryFact` uses djb2 hash of `kind + normalized description` for fact IDs, ensuring the same observation produces the same ID across process restarts. This is a reusable pattern for any entity that needs stable identity across sessions.

### L5: Test Count Tracking Should Be Automated

Manual test counts in completion notes were wrong in 2 out of 4 stories. This is a systematic issue that should be fixed with tooling rather than process.
