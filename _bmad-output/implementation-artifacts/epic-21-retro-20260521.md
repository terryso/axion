## Challenges & Growth Areas

### 1. SDK Spec Describes Non-Existent Features

Stories 21.2 and the original spec referenced `traceEnabled`, `traceBaseURL`, `onRunComplete`, and `TraceRecorder` as SDK features. None exist in OpenAgentSDK. This caused confusion during implementation and required keeping Axion's TraceRecorder despite the plan to delete it.

**Impact:** Story 21.2 could only partially deliver -- CostTracker was deleted but TraceRecorder remains. The spec-to-implementation gap wasted analysis time.

**Systemic Issue:** SDK specs document planned features alongside implemented features without distinction.

### 2. Type Name Collisions Between SDK and Axion

SDK and Axion share many type names (`Task`, `Event`, `MemoryFact` vs `AppMemoryFact`). Swift's `struct AxionCLI: AsyncParsableCommand` shadows the module name, preventing `AxionCLI.TypeName` qualification. This was the #1 recurring pain point across all 6 stories.

**Impact:** Required targeted imports, private typealiases, and careful namespace management in every story.

### 3. SDK Types Are Slimmer Than Axion Equivalents

SDK's `MemoryFact` lacks scope/cause/evidence fields. SDK's `RunCompletedData` lacks `replanCount`. SDK's `StepCompletedData` lacks `purpose`. Each gap required adapter work or data loss acceptance.

**Impact:** Adapter files (352 lines for HTTP API alone) represent permanent overhead. `replanCount` is hardcoded to 0.

### 4. Story Scope Creep During Execution

Stories 21.4 and 21.5 both exceeded their estimated scope. Story 21.4 required updating 8+ test files beyond the spec's scope. Story 21.5 discovered that RunLockService is used by 3 independent callers, making simplification infeasible.

**Impact:** None critical, but highlights that refactoring stories have inherently unpredictable scope.

### 5. O(n^2) Algorithm in Existing Code

Story 21.3 review caught an O(n^2) demotion loop in `RunMemoryProcessor.preRunCleanup` -- this existed before Epic 21 but was discovered during the refactor.

## Key Insights & Learnings

1. **Adapter Pattern is the Right Abstraction for SDK Consumption.** When Axion needs behavior beyond what SDK provides, a thin adapter wrapping the SDK component preserves both correctness and extensibility. This pattern should be documented as the standard approach for future SDK integrations.

2. **Pre-Deletion Audits Prevent Silent Data Loss.** The bugs caught in Stories 21.1 and 21.3 were both "silent failures" -- code compiled and tests passed, but behavior was wrong. Only careful review of data flow caught them. The audit-table pattern from Epic 20 proved its value again.

3. **SDK Specs Need an "Implemented vs Planned" Distinction.** The gap between what the spec describes and what the SDK actually exports caused wasted effort and partial delivery. Future epics should verify SDK exports before writing stories.

4. **Type Collision is a Structural Problem, Not a One-Off.** With SDK exporting `public struct Task`, every Swift file that imports both OpenAgentSDK and uses concurrency needs workarounds. This should be raised with the SDK team.

5. **Documentation Stories Deserve Dedicated Stories.** Story 21.6 caught 8 doc inaccuracies that accumulated across 5 implementation stories. Without a dedicated doc story, these would have compounded.

6. **Deferred Work Can Age Out.** The ToolRegistrar split was deferred for 19 epics. When finally tackled, the approach was straightforward. Deferred items should be periodically reassessed -- some become easier with context.

## Significant Discoveries

### Discovery 1: SDK Lacks Trace Infrastructure

**Finding:** OpenAgentSDK does not export `traceEnabled`, `traceBaseURL`, `TraceRecorder`, or any trace recording API. These features were documented in the SDK spec but never implemented.

**Impact:** Axion's `TraceRecorder.swift` (308 lines) must be maintained indefinitely. Story 21.2 could only deliver partial scope (CostTracker deleted, TraceRecorder kept). `AgentOptions.runId` is wired but always nil.

**Action:** Flag with SDK team. Document as permanent Axion-specific code.

### Discovery 2: SDK Memory Types Lack Axion-Specific Fields

**Finding:** SDK's `MemoryFact` has no `scope`, `cause`, or `evidence` fields. These are critical for Axion's desktop-specific memory (skill-scoped facts, takeover learning).

**Impact:** `AxionFactStore` actor is a permanent adapter, not a temporary bridge. Every SDK boundary requires conversion helpers. The dual type system (`AppMemoryFact` + SDK `MemoryFact`) adds complexity to all memory operations.

**Action:** Consider proposing field extensions to SDK team. Document conversion layer as permanent.

### Discovery 3: SDK Exports `public struct Task` Shadowing Swift Concurrency

**Finding:** OpenAgentSDK exports `public struct Task` which shadows Swift's `_Concurrency.Task`. This requires `Swift.Task { }` qualification or targeted imports in every file that uses both SDK and concurrency.

**Impact:** Affects all 6 stories and will affect all future SDK-consuming code. This is a structural friction point.

**Action:** Raise with SDK team. Consider SDK namespacing change.
