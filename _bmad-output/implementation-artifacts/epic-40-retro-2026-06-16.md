## 做得好的地方（Successes）

### W1：线性链 epic 的风险控制样本——"最小爆炸半径"成为贯穿纪律

Epic 40 是 Axion 至今**最长的一次性线性链 epic**（10 story，严格 40.1→40.10 依赖）。风险表明确警告"Story 40.2 重新变成大而全的 refactor"，而每个 story 都用**显式红线表**守住边界：

- 40-2 红线：不做 40.3（注册 Agent/Task）、不改 `excludedToolNames`（40.5）、不让 `buildSkillAgent` 复用 helper（40.4/40.5）、不加未来字段（YAGNI）

- 40-4 红线：不碰工具集/权限/提示（留给 40.5/40.6/40.7）
- 40-8 仅改聊天路径，刻意不动 `SDKTerminalOutputHandler`（`axion run`）/ `SDKJSONOutputHandler`（API）
- 40-9 **零 `Sources/` 生产代码改动**，40-10 **零源码/测试改动**（纯验证记录）

> **Amelia (Developer):** "线性链 epic 最大的风险是 story N 把 story N+1 的活干了一半，破坏后续 story 的可验证性。这次每个 story 都画红线表，确认没有越界才提交。"
> **Charlie (Senior Dev):** "40-9/40-10 两个 story 故意不写新代码——一个是确定性证据，一个是真实性证据。这种克制本身就是交付。"

### W2：纯可注入 helper 家族成为整个 epic 的可测性骨架

Epic 40 把 Epic 37（L4）/38/39 的"纯函数 + DI"模式**体系化**为一组对称的静态 helper 家族：

- `buildToolProfile(...)`（40-2）→ `buildSkillToolProfile(registry:)`（40-3）→ `makeDiscoveredSkillRegistry(ensuring:discoveryDirectories:)`（40-4）
- `effectiveExcludedToolNames(allowingToolSearch:)` / `resolveSkillMcpServers(from:helperPath:)`（40-5）→ `diagnoseToolAvailability(...)` / `effectiveSkillToolPool(...)`（40-6）→ `slashSkillAndTaskGuidance(noSkills:dryrun:)`（40-7）

每一个都是：纯函数、带可注入 default-arg seam（临时目录 / `helperPath:"/usr/bin/true"` / `apiKey:"sk-test"` / `discoveryDirectories`）、无副作用、被直接单测（**从不调用真实 `build()`/`buildSkillAgent()`**）。这让 9 个中端模型 story 仍能逐 story 增量交付且可隔离测试。

## 挑战与成长空间（Challenges）

### C1：child 侧 slash guidance 在不编辑 SDK 的前提下不可实现（架构决策精炼）

Epic 架构 §4 写"该指令可加到**父** agent system prompt（当 Skill+Task 都注册时）**或**子 agent system prompt（在 subagent tool factory 中）"。40-7 的实现发现：**child agent 的 system prompt 由 SDK `DefaultSubAgentSpawner` 决定，而 `SubAgentInheritanceContext` 没有 `systemPrompt` 字段**，且 `BUILTIN_AGENTS` 只有 "Explore"/"Plan"，`subagent_type:"general-purpose"` → `agentDef == nil` → `systemPrompt: nil`。

因此 40-7 只能走"父侧 prompt 注入"分支，child 侧被显式归类为 **SDK follow-up**（而非 story 内任务）。这是对 spec §4 "parent OR child"的精炼——当前 SDK 形状下 child 分支不可达。

> **Winston (Architect):** "spec 给的是逻辑选择，实现撞上的是 SDK 形状约束。诚实归因到 SDK follow-up，比硬塞一个编辑 `.build/checkouts/` 的临时补丁更对。"

### C2：File-List 透明度——drive-by 测试修复反复漏报（最高频 review 发现）

`TaskSerialQueueTests.swift` 的 `.serialized` + 10s 超时 + 收窄断言的**串行 `--no-parallel` 稳定性修复**，出现在 40-3/40-4/40-5/40-6/40-7/40-8 **至少 6 个 story** 的工作树里，且**反复不在 File List / Change Log 中**，每次都由 review 兜底补文档。这是 Epic 40 最高频、最一致的 review 发现（40-3 MEDIUM、40-4 MEDIUM、40-5 review 回填、40-6 LOW、40-7 M1、40-8 M1）。

根因：`make test`（4000+ 测试）在串行模式下时序不稳定，迫使这个可靠性修复渗进每个 story 的树；而 dev 在提交时倾向于只列"本 story 功能文件"，遗漏了 drive-by 修复。

> **Charlie (Senior Dev):** "同一个稳定性修复漏报 6 次，说明这不是个人疏忽，是流程缺口——drive-by 修复必须在提交时就进 File List，而不是等 review 发现。"

### C3：DesktopNotifier tmux OSC-9 噪声——多 epic 持续未处理

每个 story 的 `make test` 都固定报 **7 个 `DesktopNotifier` 套件失败**（tmux OSC 9 被 wrap 成 DCS `Ptmux;...` 透传）。这是从 Epic 39（TD-Dana）延续的既有环境问题，**与 Epic 40 无关**（diff ∩ DesktopNotifier = ∅），但持续污染测试输出、削弱"全绿"信号可信度。每个 story 都要花笔墨证明"非本 story 引入"。

### C4：编排/API 流层的脆弱性（非模型能力问题）

40-9 的 story 文件**两次被重建**：一次是 create-story 会话**连续 4 次 API 流式传输失败**后由 orchestrator 回退创建；一次是文件在实现中被外部进程删除（git 未跟踪），按首次读取内容重建。这暴露的脆弱性在**编排/API 流层**，与模型能力无关——是 story-automator 与 API 流的稳定性问题，对 LLM 能力评估是干扰项。

### C5：AC1 正面完成路径未在真实运行中端到端捕获（诚实性披露）

40-10 为遵守"不对真实生产冲刺中的活跃 story 运行完整 5 步流水线"的红线，使用了安全/可丢弃的 story ID（`99-1` 不存在、`1-1`/`33-1` 无写入）和损坏副本。后果：**AC1 的"第一步成功 → 第二步"正面完成路径未在真实运行中端到端捕获**。dev 在 story 中主动添加"诚实性说明"披露此点，正面路径的正确性由 40-9 的 fixture 确定性等价物证明。这是污染控制约束下的运行时取舍。

### C6：并行 workflow 污染 git 工作树（40-10 MEDIUM-1）

40-10 的工作树包含未提交的无关更改——`ChatCommand.swift`、`AppArchitectureFormatter.swift`、新增 `AppArchitectureSelectionPrompt.swift` + 测试、`_bmad-output/specs/spec-arch-upgrade-workflow/`——来自**并行的 `app-architecture` 升级 workflow**，非 40-10 产生。当前 `git status`（本回顾生成时）仍可见这些未提交项。在此污染树上运行 `make test` 仍显示 Epic-40 套件零回归，但工作树卫生需要单独提交处理。

## 关键洞察与教训（Key Insights）

### L1（精炼 Epic 39 L4）：模型选型的真正杠杆是"分解质量 + 对抗性 review"，而非模型等级本身

**这是本次回顾最重要的发现。** Epic 38/39 连续两个 epic 把"复杂 story 必须用最高能力模型"作为 flagged 教训，Epic 39 甚至把它**升级为规则**（行动项 #1）。但 Epic 40 用**中端模型（GLM-5.2[1m]）实现了 9/10 story，10 个 story 全部 0 CRITICAL / 0 HIGH**。

这说明原规则**过于粗放**。真正决定 review 负担的不是模型单价，而是**返工系数**；而返工系数主要由**分解质量**决定：

- 细粒度 AC + 预验证的 SDK 事实 + 显式红线 → 中端模型返工系数很低
- 纯函数 helper + 可注入 seam → 错误在单测层就被抓，不流到 review
- 强制全新上下文对抗性 review（3 层：Blind/Edge/Acceptance）→ 兜底剩余风险

> **Key Insight：** "模型成本 = 单价 × 返工系数"仍成立，但**充分分解 + 强制对抗性 review 能把中端模型的返工系数压到接近最高能力模型**。规则应从"复杂 story 用最高模型"修订为"**复杂 story 用最高模型，或确保充分分解 + 强制对抗性 review**"——决策依据是返工系数，不是单价。

### L2（强化）：线性链 epic 的首要风险控制是"最小爆炸半径 + 显式红线表"

Epic 40 的 10 story 零回滚、零 CRITICAL，首要归因不是模型，而是**每个 story 都画红线表、确认不越界才提交**（W1）。这条纪律应上升为线性链 epic 的标准实践。

### L3（强化）：纯可注入 helper 家族是"宿主层组装"变更的可测性骨架

Epic 37/38/39 的"纯函数 + DI"在 Epic 40 体系化为 `buildToolProfile` 家族（W2）。任何未来涉及 AgentBuilder 工具组装的 epic 都应延续这个模式，而不是回到 `build()` 热路径。

### L4（新）：端到端 real-LLM story 用"确定性证据 + 真实性证据"双轨验收

涉及真实 LLM 的端到端 story（如 BMAD pipeline、多步编排）应**强制拆出确定性 fixture story**（无网络、可重复、CI 可跑）作为正确性主证据，真实运行 story 作为真实性补充证据（W4）。二者不可互替，避免 epic 被一次性日志的人工签字 block。

### L5（新）：behavior-faithful AC over literal AC

当 spec 断言字面与真实运行时行为冲突时，忠实行为 + 记录偏差（W3）。绝不交付注定失败的测试；spec 与实现的出入应显式记录为 follow-up，而非用脆弱断言掩盖。

### L6（强化）：宿主层只做"组装与策略"，不重写 SDK

Epic 40 多次选择**用 SDK 公开 helper 在边界重算**而非编辑 `.build/checkouts/`：40-2 字节级 Parity 提取（零重写）、40-6 用公开 helper 重算 `ToolFilterDiagnostics`（SDK 在 `DefaultSubAgentSpawner` 边界丢弃诊断）。这条边界守得很干净，使 Epic 40 的所有 SDK 依赖都可通过版本升级获得，而非维护 fork 补丁。
