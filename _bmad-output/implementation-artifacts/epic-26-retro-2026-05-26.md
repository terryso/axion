This extends the interface-first pattern validated across Epics 21–25. Six consecutive epics now confirm this as a proven architectural principle.

### W2: Consistent Composition Pattern Across All 16 Event Types

Every event type follows the identical pattern:
- `struct` with `let` properties (immutable, value semantics)
- `AgentEvent` + `Equatable` conformance
- `base: BaseAgentEvent` composition for `id` and `timestamp`
- Flat JSON serialization with snake_case CodingKeys
- Explicit `init(from:)` and `encode(to:)` for SSE-ready flat JSON

This consistency means Epic 27's emit code will be mechanical — construct event, call `eventBus.publish()`. No per-type special handling needed.

### W3: EventBus Actor Design Is Clean and Robust

- `removeSubscriber` calls `continuation.finish()` to unblock consumer `for await` loops

The debug log revealed a critical insight: initially `removeSubscriber` didn't call `continuation.finish()`, causing `for await` loops to hang indefinitely. This was caught during development and fixed.

### C3: Mixed Agent Models Produced Slightly Different Patterns

Stories 26.4 and 26.1 used Claude Opus 4.7 while others used GLM-5.1. The results were functionally equivalent, but there were minor style differences:
- Claude Opus stories had slightly more detailed dev notes
- GLM-5.1 stories had more documentation inaccuracies in completion notes

**Impact:** Low — both produced working code. But consistency in dev agent selection would reduce review overhead.

### C4: Story 26.6 Initially Hung Due to Missing `continuation.finish()`

The EventBus story had a subtle concurrency bug: `removeSubscriber` did not call `continuation.finish()`, causing `for await` loops in tests to block indefinitely. This was caught during development, but it highlights a risk: Swift's `AsyncStream` termination semantics require explicit `finish()` calls.

**Recommendation:** When working with `AsyncStream.Continuation`, always pair `removeValue(forKey:)` with `continuation.finish()` to unblock consumers. Document this as a pattern rule.

## Lessons Learned

### L1: Interface-First Pattern Confirmed for 6th Consecutive Epic

Epics 21–26 all used the same pattern: define protocols and base types in Story 1, implement concrete types in subsequent stories. Zero interface changes needed after Story 1 in every epic. This is now firmly established as a validated architectural principle.

**Key Insight:** When building layered type hierarchies, invest heavily in Story 1's protocol design. The payoff is zero interface churn across the remaining stories.

### L2: Flat JSON Serialization Is the Right Choice for SSE

All 16 event types use flat JSON (id/timestamp at top level, no nested "base" key). This makes SSE mapping in Epic 28 trivial — no JSON transformation needed. The explicit `init(from:)`/`encode(to:)` approach gives full control over snake_case key mapping.

**Key Insight:** When types will be consumed by external systems (SSE, HTTP API), design serialization for the consumer, not for Swift ergonomics.

### L3: Swift AsyncStream Termination Requires Explicit finish()

The EventBus bug (missing `continuation.finish()`) revealed that `AsyncStream` doesn't automatically terminate when you remove a continuation from a dictionary. Consumers blocking on `for await` will hang forever unless you explicitly call `finish()`.

**Key Insight:** When managing `AsyncStream.Continuation` lifecycles, always call `continuation.finish()` when removing a subscriber, not just `removeValue(forKey:)`.

### L4: Test Count Accuracy Requires Post-Review Verification

Despite being flagged in 4 previous retrospectives, test count documentation accuracy remains an issue. The pattern is consistent: dev agent writes correct code but doesn't update the story file after code review adds/changes tests.

**Key Insight:** The story automator should extract test counts from `swift test` output after the review step, not from the dev agent's completion notes.
