## Story 逐一分析

### Story 17.1: RunCommand 集成 SkillRegistry

**亮点：**
- **仅修改一个文件**（RunCommand.swift）就完成了完整的 Skill 系统集成——SDK 提供的基础设施（SkillLoader、SkillRegistry、SkillTool、formatSkillsForPrompt）功能完备，Axion 只需"接线"
- `--no-skills` flag 设计与 `--no-memory` 正交，两者独立控制，保持 CLI 参数设计一致性
- SkillRegistry 作为 `run()` 的局部变量，每次运行重新扫描——避免跨运行缓存导致技能文件变更不生效
- 14 个单元测试使用临时目录，不依赖真实 `~/.claude/skills/`，测试隔离性良好

**挑战：**
- 无 HIGH severity 问题
- 无重大技术障碍——SDK API 表面设计清晰，集成工作流畅

### Story 17.2: 双轨技能查找

**亮点：**
- SkillLookupService 作为无状态 struct 设计——每次 lookup 从文件系统读取最新状态，不缓存，符合技能文件可能被用户在运行间修改的使用场景
- RecordedSkillRunner 成功从 SkillRunCommand 提取可复用的 Helper 启动 + SkillExecutor 执行 + 元数据更新逻辑，消除了代码重复
- 16 个单元测试覆盖全部 6 个 AC 及边界情况（空技能名、特殊字符、无效 JSON、URL-like 参数）

**挑战：**
- **[MEDIUM] Helper leak on throw** — RecordedSkillRunner 的 `withTaskCancellationHandler` 在 `executor.execute()` throw 时不触发清理。修复：改用显式 try/catch + guaranteed `helperManager.stop()`
- **[MEDIUM] Duplicate SkillRegistry** — `registerDiscoveredSkills()` 被调用两次。修复：将 registry 创建提升到 `run()` 入口
- **[MEDIUM] Success message swallowed** — 打印在 metadata 写入之后，写入失败时用户看不到结果。修复：移到写入之前
- **两种 Skill 类型的桥接** — `OpenAgentSDK.Skill`（prompt 技能）和 `AxionCore.Skill`（录制技能）是完全不同的类型，`SkillLookupResult` enum 巧妙地统一了两种查找路径

### Story 17.3: 显式 `/skill-name` 触发

**亮点：**
- 显式触发路径的设计决策清晰——promptTemplate **替换**（不追加）baseSystemPrompt，因为用户明确选择了技能
- `explicitSkill` 作为局部变量在 `run()` 中流转，避免了全局状态污染
- `allowedTools` 为 nil 表示全部可用——与 SDK 约定一致，避免显式列举所有工具
- 录制技能参数校验前移到 helper 启动之前，节省资源

**挑战：**
- **[HIGH] toolRestrictions 生效时 systemPrompt 仍列出所有 MCP 工具** — 当技能通过 `allowedTools` 限制了可用工具集，但 systemPrompt 的 Available Tools section 仍列出所有 MCP 工具名，误导 LLM。修复：条件分支，有 restrictions 时只列 restricted 工具名
- **[MEDIUM] 参数缺失错误消息用 print() 输出到 stdout** — 错误信息应输出到 stderr。修复：改用 `fputs(..., stderr)`
- **[MEDIUM] 缺少 toolRestrictions 限制工具列表的测试** — 新增 `testToolRestrictionsLimitsToolList` 测试
- AC5 和 AC6 已由 Story 17.2 实现，本 Story 无需额外修改——体现了 Story 间的良好分层

### Story 17.4: 隐式技能触发

**亮点：**
- **变更量极小**（仅修改 buildFullSystemPrompt 中的 skillsPrompt 注入逻辑），正确识别了隐式触发的本质——不需要修改代码逻辑流程，只需让 LLM 知道何时及如何调用 Skill 工具
- 指引文本简洁（两句话约 50 tokens），不浪费 system prompt 空间
- 显式触发 vs 隐式触发的对比表清晰，两个路径正交不冲突

**挑战：**
- **[HIGH] 测试误报风险** — `testGuideMentionsParameters` 使用 `contains("skill")` 会匹配标题中的 "Skills"。修复：改用 backtick 精确匹配（`` `skill` `` / `` `args` ``）
- **[MEDIUM] AC4 测试同义反复** — 测试直接传空字符串而非模拟 noSkills 代码路径。修复：改为模拟 `cmd.noSkills ? "" : "dummy"` 验证 flag 控制
- **[MEDIUM] 指南文本 args 可选标注缺失** — 添加 `required`/`optional` 标注

## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "SDK 集成 = 只接线不造轮子"**
Epic 17 最突出的模式是 Axion 只做集成层，所有核心逻辑（SkillLoader、SkillRegistry、SkillTool、formatSkillsForPrompt）都由 SDK 提供。4 个 Story 中 Axion 新增的源代码总共仅 138 行（3 个文件），但实现了完整的技能系统。
**启示：** 当 SDK 提供了完整的 API 表面时，集成层的代码量应远小于 SDK 本身。如果集成层代码量接近 SDK，说明可能存在职责越界。

**2. "局部变量流转"避免全局状态**
SkillRegistry 和 explicitSkill 都是 `run()` 的局部变量，在函数内通过条件分支传递，不存储到全局状态或单例。这确保了并发安全和状态隔离。
**启示：** 在 CLI 工具中，每次运行创建新的局部状态比使用全局单例更安全、更可测试。

**3. "enum 统一多态查找"模式**
`SkillLookupResult`（`.promptSkill(SDK Skill)` / `.recordedSkill(AxionCore Skill)` / `.notFound`）用 enum 优雅地统一了两种完全不同的技能类型查找路径，避免了继承或 protocol 复杂性。
**启示：** 当需要统一两种不相关的类型时，enum + associated value 比 protocol 更清晰——每种情况的处理逻辑完全不同。

**4. "Story 间分层清晰"避免重复工作**
17.3 的 AC5/AC6 已由 17.2 实现，17.4 的 AC2/AC3/AC4 已由 17.1 实现。每个 Story 明确标注了"Note: 已由前序 Story 实现"，避免了重复开发和测试。
**启示：** Story 规格中的 AC 归属标注（哪个 Story 实现了哪个 AC）对团队协作至关重要。

**5. "显式替换 vs 隐式追加"的 systemPrompt 策略**
显式触发时 promptTemplate **替换** baseSystemPrompt（用户明确选择），隐式触发时技能描述**追加**到 baseSystemPrompt 末尾（用户未指定）。两种策略各有适用场景。
**启示：** system prompt 的构建策略应根据用户意图的明确程度选择——明确选择用替换，模糊匹配用追加。

### 挑战模式（需要改进）

**1. Review findings 数量仍在 3-6 个/Story**
17.3 有 6 个 findings（1 HIGH），17.4 有 5 个 findings（1 HIGH）。HIGH severity 的两个问题分别是：toolRestrictions 与 Available Tools 不一致、测试匹配模式误报。
**根因：** 两个 HIGH 都是"部分实现与整体上下文不一致"——17.3 修改了 allowedTools 但忘记同步修改 Available Tools section；17.4 修改了指引文本但测试仍用模糊匹配。
**改进：** 当修改影响多个关联组件（如 allowedTools + Available Tools section、指引文本 + 测试断言）时，Dev Notes 应列出关联组件清单，确保修改完整性。

**2. RunCommand 持续膨胀**
RunCommand.swift 已达 1152 行。Epic 17 新增了 SkillRegistry 创建、SkillLookup 解析、explicitSkill 处理、条件 systemPrompt 构建等逻辑，进一步增加了 RunCommand 的复杂度。
**根因：** RunCommand 是 Axion CLI 的主入口，所有 Agent 创建管线逻辑集中在此。Epic 16 回顾中标记的"RunCommand 提取 MessageStreamProcessor"技术债务仍未处理。
**改进：** 在 Epic 18 启动前，应完成 RunCommand 的重构——至少将 Skill 相关逻辑提取到独立的服务层。

**3. 测试计数偏差仍然存在**
17.3 的 Dev Notes 声称"16 个单元测试"但实际有 17 个（Completion Notes 后更正）。这是从 Epic 14 延续的模式。
**根因：** 测试计数来自开发过程中的估计，而非最终文件扫描。
**改进：** Completion Notes 应使用 `grep -c "@Test" Tests/...` 命令统计实际测试数量。

### 突破时刻

**1. 双轨查找的 "prompt 优先" 简化决策**
SkillLookupService 的查找顺序（先 registry 后 JSON 文件）看起来简单，但这个设计决策避免了复杂的冲突解决策略。prompt 技能（LLM 驱动、灵活）天然优于录制技能（固定步骤、无适应性），"先到先得"的优先级策略是最简洁正确的选择。

**2. RecordedSkillRunner 提取消除了 SkillRunCommand 的代码孤岛**
Story 17.2 将 Helper 启动 + SkillExecutor 执行 + 元数据更新提取为 RecordedSkillRunner，使得 RunCommand 和 SkillRunCommand 都能复用录制技能执行逻辑。这是 Epic 17 中最有价值的重构。

**3. 隐式触发的"最小变更"洞察**
Story 17.4 的核心洞察是：隐式触发不需要修改任何执行逻辑——SkillTool 已注册（17.1），formatSkillsForPrompt 已注入（17.1），LLM 只需要知道"何时及如何"调用 Skill 工具。两句话的指引文本就完成了整个隐式触发能力。这是 Epic 17 中变更量最小但价值最高的 Story。

## 关键经验

1. **SDK 集成的核心是"不造轮子"** — Epic 17 新增源代码仅 138 行（3 个文件），但实现了完整的技能系统（发现、注册、查找、显式触发、隐式触发）。核心原因是 SDK 的 SkillLoader、SkillRegistry、SkillTool 已经提供了 90% 的功能，Axion 只做"接线"。教训：在集成 SDK 功能时，首先评估 SDK API 表面的覆盖度，只补充 SDK 未提供的集成逻辑。

2. **显式触发替换 systemPrompt vs 隐式触发追加** — 两种触发方式对 systemPrompt 的处理策略完全不同。显式触发时用户明确选择了技能，技能的 promptTemplate 应**替换**默认 planner-system.md（避免通用指令干扰专业技能）；隐式触发时用户未指定技能，技能列表应**追加**到默认 prompt 末尾（保持通用能力的同时增加技能发现）。教训：systemPrompt 的构建策略不是一成不变的，应根据用户意图的明确程度灵活调整。

3. **"最小变更"洞察的价值** — Story 17.4（隐式触发）仅通过在 system prompt 中添加两句话的指引文本就实现了完整的隐式技能触发能力。正确的分析发现了核心需求已被前序 Story 覆盖（SkillTool 注册、formatSkillsForPrompt 注入），只缺少"告诉 LLM 如何使用"。教训：在实现新功能前，先评估已有基础设施的覆盖度，识别真正的增量需求。

4. **RunCommand 膨胀是项目最大技术风险** — RunCommand 从 Epic 1 的约 200 行增长到 Epic 17 的 1152 行，承担了 Agent 创建管线、Memory 注入、Skill 集成、显式/隐式触发等多个职责。连续 5 个 Epic 的回顾都标记了"RunCommand 提取 MessageStreamProcessor"为 High 优先级技术债务，但从未执行。教训：当关键文件的技术债务在每个 Epic 都被重新标记但从未处理时，应触发强制重构。

5. **Epic 17 的 Review findings 模式反映了"集成 vs 实现"的质量差异** — 17.1 和 17.4 的 findings 较少（17.1 未记录, 17.4 有 5 个但主要是测试问题），17.2 和 17.3 的 findings 较多（17.2: 4 个, 17.3: 6 个）。17.1 和 17.4 是"纯集成"（使用 SDK API），17.2 和 17.3 涉及"新逻辑"（双轨查找、显式触发增强）。教训：纯集成 Story 的代码质量天然高于新逻辑 Story——集成时 SDK 的 API 约束了实现方式，减少了设计空间和出错概率。
