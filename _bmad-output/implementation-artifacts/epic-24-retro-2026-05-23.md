## What Went Well

### W1: Interface-First Pipeline Assembly Works Perfectly

Story 24.1 defined `ReviewAgentConfig`, `ReviewAgentResult`, `ReviewPromptBuilder`, and `Agent.createReviewAgent(config:)` as the public interface. Stories 24.2–24.4 consumed this interface without modification:

- Story 24.2 created tools matching the `allowedTools` names defined in 24.1's `ReviewAgentConfig` defaults
- Story 24.3 composed `createReviewAgent()` + `createReviewTools()` + `ReviewPromptBuilder` without touching any earlier types
- Story 24.4 enhanced `createReviewAgent()` with cached system prompt without breaking 24.3's usage

**Key Evidence:** Zero interface changes needed after Story 24.1. The dependency graph (24.1 → 24.2 → 24.3, with 24.4 enhancing 24.1's output) assembled cleanly.

### W2: Review Pipeline Is Complete and Functional

The full review pipeline now works end-to-end:

```
sessionEnd hook → ReviewOrchestrator.shouldReview()
  → interval check (memory: every 4th session, skill: every 6th)
  → ReviewOrchestrator.executeReview()
    → ReviewPromptBuilder.selectPrompt()
    → Agent.createReviewAgent(config:) — fork isolated Agent
    → createReviewTools() — inject 4 review tools
    → Task.detached { reviewAgent.prompt() } — non-blocking
    → summarizeActions() — extract human-readable summary
    → Logger.shared.info() — log review results
```

Every piece of Hermes's `background_review.py` has a Swift counterpart.

### W3: Cost Optimization via Prefix Cache Sharing

Story 24.4 implemented a sophisticated prefix cache sharing mechanism:

- Parent Agent builds system prompt (with git context, project instructions, session memory)
- `cachedSystemPrompt` stores the fully-built prompt
- Review Agent reuses the cached prompt verbatim (via `_rawSystemPromptMode`)
- Both Agents share the same `LLMClient` reference
- Result: ~26% cost reduction on review API calls (matching Hermes PR #17276 analysis)

This required careful attention to `buildSystemPrompt()` internals — nilling out dynamic context fields on the review Agent so the cached prompt isn't re-modified.

### W4: `agentLabel` Enables Cost Attribution

Story 24.4 added `agentLabel: String?` to `AgentOptions` and wired it through `CostTracker`, `CostSummary`, and `CostBreakdownEntry`. Review Agents get `agentLabel: "review"`, making it trivial to distinguish main agent costs from review agent costs in cost reports.

This is a clean, non-breaking addition — `nil` by default, only set for review agents.

### W5: Safety Boundaries Are Well-Designed

The review Agent has multiple safety layers:

- **Tool restriction**: Only 4 review tools available (no file system, no bash, no MCP)
- **Hook isolation**: `hookRegistry` set to nil — review Agent never triggers user hooks
- **Permission bypass**: `permissionMode: .bypassPermissions` — review tools don't need approval
- **Cost cap**: `maxBudgetUsd` inherited from parent — review can't exceed parent's budget
- **No nested review**: `reviewScheduleConfig` nilled out on review Agent — prevents recursive reviews
- **Skill registry isolation**: Review Agent gets its own skill registry — no cross-contamination
- **Background execution**: `Task.detached` — review never blocks the parent Agent's return

### C2: Story 24.3 Had 3 HIGH Issues in Code Review

Story 24.3 (ReviewOrchestrator) had the most review issues of the epic:

- **H1: Dead `IntervalTracker` class and unused `tracker` property** — The interval tracker was designed in the story spec but the actual `shouldReview()` implementation uses modulo arithmetic instead. The dead code was left behind.
- **H2: `executeReview` wrapped `prompt()` in pointless `Task.detached` + `.value`** — The method was supposed to run the review in a detached task, but it wrapped `Task.detached { await prompt() }.value`, which is equivalent to just `await prompt()` directly. The detachment was redundant.
- **H3: Missing failure detection** — `executeReview` didn't check if `prompt()` returned an error status, always returning a `ReviewAgentResult` even on failure.

**Root Cause:** Story 24.3 was the most complex story (scheduling + orchestration + hook registration + tools injection). The spec was detailed but the implementation deviated from the spec in subtle ways. The dead code suggests the developer started implementing the spec literally, then simplified, but didn't clean up.

**Recommendation:** For complex orchestration stories, consider a mid-implementation checkpoint to validate approach before filling in all the details.

### C3: Signal Type Mismatch Between Spec and Implementation

Story 24.2 used `.refinement` signal type instead of `.update` (which doesn't exist in `SkillSignalType`). Similarly, it used `.conversation` source instead of `.review` (which doesn't exist in `SkillEvolutionSource`).

The spec referenced enum values that don't exist. The developer correctly found the closest matching values, but this is a spec-code mismatch that could have been caught earlier.

**Recommendation:** Story specs should verify enum values against the actual codebase, not against Hermes Python conventions.

### C4: `_rawSystemPromptMode` Is an Internal Mechanism on a Public Type

Story 24.4 added `_rawSystemPromptMode: Bool` to `AgentOptions` — a public struct. While the underscore prefix signals "internal", it's still accessible to consumers. This is a leaky abstraction: the prefix cache sharing mechanism bleeds into the public API.

**Impact:** Low — consumers shouldn't set `_rawSystemPromptMode` directly, and it defaults to `false`. But it's a design smell.

**Recommendation:** Consider making this `fileprivate` or using a different mechanism (e.g., a separate `ReviewAgentOptions` type that wraps `AgentOptions`).

### C5: Review Results Are Fire-and-Forget — No Callback Mechanism

The `ReviewOrchestrator` runs reviews in `Task.detached` and logs results via `Logger.shared`. There's no callback, no event emission, no way for consumers to know when a review completes or what it found.

**Impact:** Medium — developers who want to display review results in their UI or trigger follow-up actions have no hook point.

**Recommendation:** Add an optional `onReviewComplete: ((ReviewAgentResult) -> Void)?` callback to `ReviewScheduleConfig` or emit a dedicated hook event.

### TD2: Review Agent Tool Injection via Mutable `options.tools`

Story 24.3 injects review tools by mutating `reviewAgent.options.tools` after creation. This works because `Agent.options` is `internal var` and `AgentOptions.tools` is `public var`, but it's a fragile pattern:

- The review Agent is briefly in an invalid state (created with `tools: []`, tools injected later)
- If `Agent.options` becomes `let` in the future, this breaks
- The pattern is not obvious to future maintainers

**Impact:** Low — works today, but could be fragile.
**Priority:** Low — consider adding a `withTools()` method on `Agent` for a cleaner pattern.

### TD3: No Review History or Deduplication

The review orchestrator doesn't track what was previously reviewed. Each review starts fresh with no memory of past extractions. This means:

- The same memory could be extracted repeatedly across sessions
- Skill updates could oscillate between sessions
- No "already known" filtering

**Impact:** Medium — repeated extractions waste tokens and could create duplicate/conflicting memories.
**Priority:** Medium — the `summarizeActions` dedup against `priorSnapshot` is a partial mitigation, but a review history store would be more effective.

### TD4: E2E Tests Continue to Require ANTHROPIC_API_KEY

Epic 22 retro (C5) flagged this. Epic 24 adds more E2E tests for review tools that also require the API key. Tests for `ReviewOrchestrator.shouldReview()` and `ReviewScheduleConfig` could run without an API key since they test scheduling logic, not LLM calls.

**Impact:** Low — CI can skip these, but local development loses coverage.
**Priority:** Low — separate E2E tests into "LLM-required" and "no-LLM-required" categories.

### TD5: `ReviewAgentResult` Never Propagated to Consumers

The `ReviewAgentResult` struct is returned from `executeReview()` but only logged. The `sessionEnd` hook handler returns `nil` (fire-and-forget). Consumers have no way to access review results programmatically.

**Impact:** Medium — the review system works but is invisible to SDK consumers.
**Priority:** Medium — add callback or event mechanism in a follow-up.

## Lessons Learned

### L1: Complex Orchestration Stories Benefit from Mid-Implementation Checkpoints

Story 24.3 (ReviewOrchestrator) was the most complex story — it wires together factory, tools, scheduling, and hook registration. It had the most review issues (3 HIGH). The simpler stories (24.1, 24.4) had fewer issues.

**Key Insight:** For stories that compose 3+ components, consider implementing the happy path first, running tests, then adding error handling and edge cases. The "implement everything at once" approach led to dead code and redundant patterns.

### L2: Prefix Cache Sharing Requires Deep Understanding of Prompt Construction

Story 24.4's `cachedSystemPrompt` mechanism only works because the developer understood exactly how `buildSystemPrompt()` constructs the prompt (base + git context + project instructions + session memory). The `_rawSystemPromptMode` flag bypasses all dynamic additions. Without this understanding, the cache would miss because the review Agent would add its own git context.

**Key Insight:** Cache sharing isn't just "copy the string" — it requires understanding the full prompt construction pipeline to ensure byte-level identity.

### L3: AgentOptions as a "God Struct" Is Becoming a Concern

`AgentOptions` now has fields for: basic config, tools, hooks, skills, MCP, stores, permissions, memory, review scheduling, agent label, and raw system prompt mode. It's a 500+ line struct with ~80 fields. Each epic adds 1-3 fields.

**Key Insight:** The review system alone added 3 fields (`reviewScheduleConfig`, `agentLabel`, `_rawSystemPromptMode`). Consider whether specialized configuration types (e.g., `ReviewConfig`, `MemoryConfig`) should be composed into `AgentOptions` rather than adding flat fields.

### L4: Fire-and-Forget vs. Return-Result Is a Design Decision, Not an Afterthought

Story 24.3 chose fire-and-forget (`Task.detached` + `Logger.shared`) for review execution. This is correct for the default use case (background review shouldn't block session end), but it means review results are invisible to consumers. This trade-off should have been explicitly designed, not emerged from implementation.

**Key Insight:** For any background process that produces results, decide upfront: who consumes the results? How? If the answer is "nobody, just log it," that's fine — but document it.

## Team Agreements

1. **Fire-and-forget requires explicit documentation:** When a background process produces results that are not returned to the caller, document the decision and the intended consumer (even if it's just "logged for debugging").
2. **Interface-first continues:** Define types and protocols in Story 1 of a pipeline, implement in subsequent stories. This pattern has now worked reliably across Epics 21–24.
3. **Prefix cache sharing requires prompt construction understanding:** When sharing cached prompts between Agent instances, ensure the full prompt construction pipeline is understood — byte-level identity is required for cache hits.
4. **Review Agent safety boundaries are non-negotiable:** Review Agents must always have: nil hookRegistry, nil stores, restricted tools, cost cap, and no nested review scheduling.
