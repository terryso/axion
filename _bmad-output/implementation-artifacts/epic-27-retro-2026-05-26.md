## What Went Well

### W1: Zero-Overhead Injection Pattern Proven in Production

The inline `if let eventBus = options.eventBus` guard pattern was adopted consistently across all 5 stories. Every emit point uses the same 3-line pattern:

```swift
if let eventBus = options.eventBus {
    await eventBus.publish(EventType(...))
}
```

When `eventBus == nil`, no event struct is constructed, no publish is called. This zero-overhead guarantee was validated by all 5,922 existing tests passing without modification. The pattern proved simple, mechanical, and reliable — a direct payoff from Epic 26's clean EventBus design.

### W2: Story 27.1 Was the Perfect Setup Story

Story 27.1 (AgentOptions EventBus parameter) was a minimal, low-risk addition — one optional field with nil default. It created the injection point that all subsequent stories consumed. The separation of "add the parameter" from "use the parameter" is the interface-first pattern applied at the story level, and it worked flawlessly.

### W3: Smart Architecture Decision on Tool Event Injection

Story 27.3 diverged from the epic spec's recommendation to emit tool events "in Agent.swift around the ToolExecutor.executeTools() call." The dev notes identified a critical flaw: `executeTools()` processes a batch of tools (possibly concurrent), making per-tool durationMs impossible to measure accurately.

Instead, `EventBus?` and `sessionId` were injected through `ToolContext` into `executeSingleTool()`, following the existing 17-field optional injection pattern. This gave:
- Precise per-tool timing (durationMs reflects actual execution time)
- Independent event sequences for concurrent read-only tools
- No architectural changes — just adding 2 more optional fields to ToolContext

The review caught and fixed a HIGH-severity bug where `durationMs` was computed after `firePostToolHook()`, inflating the reported duration with hook execution time. The fix moved the timing computation to immediately after `tool.call()`.

### W4: Story 27.5's Timeout-Protected Test Helpers

Story 27.5 introduced `collectEventsWithTimeout`, `collectFirstMatching`, and similar helpers that wrap `AsyncStream` consumption with timeouts. This refactored all existing EventBus tests to use the safer pattern, eliminating potential indefinite hangs. This is a significant improvement to test reliability.

### W5: Cross-Story Consistency Across Mixed Agent Models

Three different agent models were used: GLM-5.1 (27.1, 27.2, 27.4), Claude Opus 4.7 (27.3), and Claude Sonnet 4.6 (27.5). Despite this, the emit pattern was perfectly consistent. The dev notes and code patterns from each story were carried forward to the next, creating a cumulative knowledge transfer that minimized risk.

### W6: No Sendable Conformance Issues

Despite `EventBus` being a `public actor` and `AgentOptions` being `Sendable`, no conformance issues arose. The design correctly leverages Swift's implicit `Sendable` conformance for actors. The `capturedEventBus = options.eventBus` pattern in `stream()` handled the Sendable boundary cleanly.

### C2: Story 27.4 Sprint-Status Discrepancy

This inconsistency between the story file and sprint-status suggests the story file wasn't updated after the final review completion.

### C3: Resume Fire-and-Forget Pattern Is a Subtle Risk

Story 27.2 used a `Task { await eventBus.publish(...) }` fire-and-forget pattern in `resume()` to avoid changing the method signature from sync to async. This means:
- The event may be published after `resume()` returns
- In rare cases, the event may be lost if the process terminates between `resume()` and the Task executing
- The event ordering relative to subsequent events is non-deterministic

This was the right tradeoff (preserving sync API), but it's worth documenting for consumers.

### C4: Emit Ordering Inconsistency Between promptImpl and stream

The code review for Story 27.2 noted an ordering inconsistency: `promptImpl` emits `AgentStartedEvent` before `SessionCreatedEvent`, while `stream` could have a different ordering. This wasn't flagged as a bug but creates an observable difference between the two execution paths. If consumers rely on event ordering, this could be a problem.

## Lessons Learned

### L1: Emit-Point Precision Matters for Timing Events

Story 27.3's HIGH-severity review finding (durationMs inflated by hook execution time) demonstrates that when emitting timing-related events, the measurement must be as close to the actual operation as possible. The fix moved `durationMs` computation to immediately after `tool.call()`, before `firePostToolHook()`.

**Key Insight:** For events carrying timing data, measure the core operation only. Post-operation hooks should not be included in the reported duration.

### L2: ToolContext Injection Pattern Is the Right Approach for Cross-Layer Events

The decision to inject `EventBus` through `ToolContext` rather than emitting at the `Agent.swift → executeTools()` boundary proved correct. It gave per-tool precision, preserved the existing batch execution model, and followed an established pattern (ToolContext already had 17 optional fields).

**Key Insight:** When injecting cross-cutting concerns into static utility methods, prefer parameter injection over caller-side wrapping. It's more precise and follows existing patterns.

### L3: Test Refactoring During Implementation Pays Dividends

Story 27.5's refactoring of all EventBus tests to use timeout-protected helpers (`collectEventsWithTimeout`, `collectFirstMatching`) improved reliability across all event test suites, not just the new ones. This proactive refactoring during feature development prevented the accumulation of fragile test patterns.

**Key Insight:** When adding tests to an existing test file, refactor shared patterns if the existing tests are fragile. The investment pays off in reduced flakiness.

### L4: Story-Level Interface-First Pattern Works

Story 27.1 (add parameter only) followed by 27.2–27.5 (use parameter) is the interface-first pattern at the story level. It reduced risk by ensuring the injection point was solid before any emit code was written. This mirrors the epic-level interface-first pattern validated across 7 consecutive epics.

**Key Insight:** Apply interface-first at both epic and story levels for maximum predictability.

**Epic 28 Stories (3):**

| 28.3 | Token Streaming Event (P2) | Medium | High-frequency events, default off |

**Key Risks for Epic 28:**

1. **SSE backward compatibility** — HTTP API's SSE push behavior must remain identical to the current manual emit approach. Any difference is a regression.
2. **Event ordering guarantees** — SSE events must arrive in the same order as the current manual emit. EventBus's `AsyncStream` with `.bufferingNewest(100)` preserves order within a single publisher, but the bridge must ensure this.
3. **Performance under load** — Adding a subscriber to EventBus creates a background Task for type filtering. Multiple SSE clients means multiple subscribers.
4. **Gradual migration risk** — Epic 28 spec suggests running EventBus and manual emit in parallel before removing manual code. This doubles event traffic during migration.

**Preparation Needed:**

1. Review current SSE event types in `APITypes.swift` and map to AgentEvent types
2. Understand `EventBroadcaster` lifecycle and how it's created/destroyed per SSE connection
3. Plan for testing SSE backward compatibility (snapshot existing SSE output before migration)
4. Consider whether `cost_update` is a new SSE event type or maps to an existing one

## Team Agreements

1. **Inline guard pattern for zero-overhead event emission**: Use `if let eventBus = ... { await eventBus.publish(...) }` inline at every emit point. No helper methods, no pre-constructed event structs when eventBus is nil. This is the confirmed pattern for all event emission code.
2. **ToolContext injection for cross-layer events**: When emitting events from within static utility methods (like `ToolExecutor.executeSingleTool()`), inject `EventBus` through existing parameter structs (like `ToolContext`) rather than wrapping at the call site.
3. **Duration measurement excludes post-operation hooks**: For events carrying `durationMs`, measure only the core operation. Post-operation hooks (like `firePostToolHook()`) must not be included in the reported duration.
4. **Technical debt 5-epic carry-over is the escalation threshold**: If a debt item survives 5 epics, it becomes HIGH priority and must be scheduled in the next sprint. `stripCodeFences()` has reached this threshold.
5. **Timeout-protected test helpers for AsyncStream consumption**: When writing tests that consume `AsyncStream`, always use timeout-protected collection helpers to prevent indefinite hangs.
