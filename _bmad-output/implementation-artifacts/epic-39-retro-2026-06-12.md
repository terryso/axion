## 做得好的地方（Successes）

### W1：安全红线在最高风险 Epic 中被严格守住

Epic 39 是 Axion 至今**误删风险最高**的 epic——直接操作用户文件、移动、卸载 App。回顾全部 4 个 story：

- **零 `delete` 动作**：所有破坏性操作均为 `trash`（`FileManager.trashItem`），全代码库无永久删除路径。
- **无 sudo**：所有路径限定在用户可读写范围，系统级卸载只提示不处理。
- **纵深防御 re-validation**：39-2 执行器采用 "draft-first + 逐项再次校验"——即使在 plan 已确认后，执行每个 item 前仍重新校验路径与状态，防止 plan 与执行之间的 TOCTOU 漏洞。
- **undo 仅清理空目录**：`removeItem` 仅用于撤销时移除新建的空目录，绝不触碰用户文件。

> **Amelia (Developer):** "Epic 38 我们定了'安全优先'的基调，这次是真正用删除/卸载来考验它。结果证明红线没有被任何 story 突破。"
> **Charlie (Senior Dev):** "纵深防御那次 re-validation 是关键——plan 确认到执行之间如果有文件变化，逐项复检能兜住。"

### W2：模块边界一致性达到新高

Epic 39 严格遵循 `AxionCore（纯模型，零外部依赖）← AxionCLI（services/tools/commands）← AxionHelper（不碰文件系统逻辑）` 的分层：

- 39-1：模型进 `AxionCore/Models/Storage/`，扫描服务 + 2 个只读 tool 进 AxionCLI。
- 39-3：8 个 App 模型进 AxionCore，10 个服务 + 2 个 tool 进 AxionCLI。
- 39-4：8 个 Approval 模型进 AxionCore（`StorageApprovalDecision` 为纯函数），6 个服务进 AxionCLI。

**AxionHelper 在整个 epic 中未被触碰用于文件系统逻辑**——这是 CLAUDE.md anti-pattern 的正面体现。模型层纯函数化使 approval 决策可被单元测试覆盖而无需任何运行时副作用。

### W3：纯函数 + DI 模式在跨 epic 中持续生效

这是从 Epic 37（L4）确立、Epic 38 强化的模式，在 Epic 39 再次证明价值：

- `StorageApprovalDecision` 为纯函数，所有副作用通过 `StorageApproving` protocol 注入。
- `AppUninstallExecutor` 通过 `TrashPerforming` protocol 注入 trash 行为，测试用 Mock 替换，不碰真实文件系统。
- 通知类副作用（`NotificationHandler`）通过 `notify` 闭包注入为 no-op mock——严格遵守 CLAUDE.md "单元测试禁止系统副作用" 规则。

> **Winston (Architect):** "纯函数 + DI 让 approval 决策逻辑 100% 可测，且 run/chat/telegram 三入口共享同一套决策函数，差异只在 SurfacePolicy。"

### W4：SurfacePolicy 干净地实现"远程入口更保守"

Epic 定义要求"高风险操作在远程入口中必须更保守"。39-4 用单一 `SurfacePolicy` 枚举表达了三入口差异：

- **run / chat**：允许全部动作（含 trash）。
- **telegram**：`scanOnly + trash only`，永不 `delete`，且 MVP 阶段统一 `.cancel`（无 inline 按钮、无 `Telegram/` 代码改动）。

这既满足了 epic 的安全约束，又把 Telegram 实现严格限制在"预留"范围（不引入未规划的 Telegram UI 工作），是一个干净的范围控制决策。

### W5：契约模型的前向兼容解码一致应用

全部 contract 模型采用 `snake_case CodingKeys` + `decodeIfPresent`——这是 Axion 跨 epic 的既定模式，Epic 39 在 Storage/App/Approval 三组共 20+ 个模型上 100% 一致。意味着未来字段增补不会破坏已序列化的 manifest/plan。

### W6：Epic 38 教训被部分有效贯彻

- **L1（wiring 验证 via `// AC#` 注释）**：每个 story 在集成点标注 `// AC#`，39-1 主动把 "输出管线 wiring（AC#5）" 作为 LOW follow-up 显式记录而非隐藏。

### C2：~/Library 排除逻辑与 Support 数据扫描的张力（架构洞察）

`StorageExclusions.evaluate()` 会排除整个 `~/Library`（合理——避免误操作系统库）。但 39-3 的 `SupportDataScanService` **必须**扫描 `~/Library/Application Support`、`Containers`、`Group Containers` 等。两者直接冲突。

解决方案：SupportDataScanService **刻意不调用** `StorageExclusions.evaluate()`，改用精确的 bundle-id keyed 路径探测（只探测与目标 App bundle id 精确匹配的路径）。

> **Winston (Architect):** "这是一个真实的架构张力：通用排除（保护）与定向纳入（功能）必须共存。我们不能让通用排除吃掉定向功能。"

**这条洞察对任何未来涉及 `~/Library` 的功能（包括潜在 TG 清理增强）都成立**——已记为文档更新候选。

### C3：多入口审批集成的表面广度

39-4 触及的表面极广：`AgentBuilder.swift`、`AgentBuilder+Config.swift`（新增 `jsonOutput`）、`AxionRuntime.swift`、`RunCommand.swift`、`PermissionHandler.swift`、`ChatCommand.swift`、`TaskSerialQueue+Execution.swift`。`canUseTool` hook 的生效条件有多个微妙约束（仅 `desktopAutomation && !dryrun && !emitTokenStream`；非 storage tool 永远 `.allow()`）。

ESC 监听与 stdin `readLine` 的时序问题（HIGH bug）说明：**审批 UI 与终端原始模式输入的交互是 Axion 持续脆弱点**——与 Epic 38 TD6（"approval canUseTool 用 readLine 而非 raw mode 按键"）是同一区域。39-4 修了具体的 ESC 时序 bug，但底层 "raw mode 按键 vs readLine" 的重构仍未完成。

### C4：状态机结果组合的边界枚举

39-2 的 executor 需要处理 "成功/跳过/失败/被拒" 的组合状态。Review 发现 "全部 item 被拒" 应映射为 `partiallyFailed` 还是完全拒绝存在歧义。这类"多 item 操作的状态聚合"在 Storage 域是首次出现，缺乏先例模式，靠 Review 兜底。

## 关键洞察与教训（Key Insights）

### L1（强化）：安全红线的最好守护是"机制而非纪律"

Epic 39 没有 `delete` 动作不是靠人记住"别删"，而是因为**契约层根本没有 delete 这个 action case**（只有 `move`/`trash`/`createDirectory`/`scanOnly`）。把不可能性编码进类型系统，比靠 review 抓更可靠。

> **Key Insight：** 高风险域的安全，优先靠"让危险操作在类型/契约层不可表达"实现，而非靠流程审查。

### L2（强化）：通用排除与定向纳入必须显式解耦

`~/Library` 张力的解法（SupportDataScanService 绕过通用排除、改用 bundle-id 精确探测）应上升为模式：**当一个通用保护规则与一个定向功能需求冲突时，定向功能应使用自己的精确匹配逻辑，而不是放宽通用规则。**

### L3（强化）：SurfacePolicy 是跨入口差异的正确抽象

run/chat/telegram 三入口共享决策函数、差异收敛到一个 `SurfacePolicy` 枚举——这是"DRY 决策 + 单点差异"的干净实例。未来任何新入口（HTTP API、menubar）只需新增一个 policy case。

### L4（重复，需真正落地）：复杂 story 的模型选型应优先质量

Epic 38 已提，Epic 39 再次确认：状态机/多入口集成 story 用中端模型（GLM-5.1）省的模型费，被 Senior Review 返工成本抵消甚至超出。39-1 用 Codex 的零返工是正面对照。

> **Key Insight：** 模型成本不是单价，而是"单价 × 返工系数"。复杂 story 的返工系数高，应直接上最高能力模型。

### L5（新）：可撤销 manifest 是破坏性操作的信任基础

`StorageManifest` + `undo_storage_op` 让"先废纸篓、可回滚"成为可承诺的产品特性。manifest 记录每项 outcome（`StorageItemOutcome`），undo 从废纸篓恢复。这让 Axion 从"敢执行"升级为"敢执行且能负责"。

### L6（新）：MVP 边界用 policy + reserve 双重锁定

39-4 的 Telegram 既用 `SurfacePolicy` 锁定动作范围（scanOnly+trash），又用"统一 `.cancel`、无 inline 按钮、零 `Telegram/` 改动"锁定实现范围。**用两层锁定（行为层 + 实现层）来守住 MVP 边界**，避免预留工作失控膨胀为完整功能。
