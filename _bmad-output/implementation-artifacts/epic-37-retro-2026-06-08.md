## What Went Well

### W1: Story Dependency Chain Was Perfectly Ordered

The story sequence 37.0 → 37.1 → ... → 37.9 had a clean, natural dependency chain:
- 37.0 established the coding agent mode branch (`AgentMode.codingAgent`) and prompt template → all subsequent stories build on this mode
- 37.1 slash commands created the REPL command infrastructure → 37.2 Ctrl+C extended the REPL signal handling
- 37.3 banner used the session context → 37.4 output formatting consumed the same message stream
- 37.5 permissions plugged into the agent build pipeline → 37.6 multiline input replaced `readLine()` → 37.9 CJK input extended multiline

Zero rework was needed across the entire chain. Each story's deliverables were cleanly consumed by subsequent stories.

### W2: Desktop Automation / Coding Agent Mode Isolation

The `AgentMode` enum (`.desktopAutomation` / `.codingAgent`) in `BuildConfig` created a clean branch point in `AgentBuilder.build()`. The coding agent path correctly:
- Skips Helper process requirement (Story 37.0 H1 fix)
- Skips MCP server connections (Story 37.0 H2 fix)
- Uses its own prompt template (`coding-agent-system.md`)
- Has independent `maxTokens: 131072` (128K)
- Has independent `permissionMode` via `canUseTool` callback

All 10 stories maintained the invariant: `axion run "task"` behavior completely unaffected.

### W3: Chat/ Directory Architecture

All new code lives in `Sources/AxionCLI/Chat/` — a self-contained module:
- `SlashCommand.swift` — command parsing (pure function, Foundation only)
- `SlashCommandHandler.swift` — command logic
- `SignalHandler.swift` — SIGINT management
- `BannerRenderer.swift` — startup/exit formatting
- `MarkdownTerminalRenderer.swift` — ANSI rendering
- `SpinnerRenderer.swift` — progress indicator
- `ChatOutputFormatter.swift` — SDKMessage output handler
- `PermissionHandler.swift` — tool approval
- `MultiLineInputReader.swift` — bracket paste + continuation
- `ContextManager.swift` — context usage tracking
- `SessionResumeManager.swift` — session restore
- `CJKInputHandler.swift` — raw mode UTF-8 input

This separation means the Chat feature can be reasoned about independently from the rest of AxionCLI. No existing files were modified beyond `ChatCommand.swift` and `AgentBuilder.swift`.

### W4: Permission Mechanism Design Bypasses SDK Limitation

The discovery that SDK's `.default` `permissionMode` directly blocks non-readonly tools (without waiting for user input) was caught early. The solution — use `canUseTool` callback with `permissionMode: .bypassPermissions` — gives Axion full control over the approval flow. This is a reusable pattern for any future interactive mode.

### W5: SDK Auto-Compact Discovered and Leveraged (Story 37.7)

Story 37.7 discovered that OpenAgentSDK's `stream()` already includes auto-compact at the `contextWindowSize - 13,000 buffer` threshold. Instead of reimplementing compaction, the story correctly leveraged the SDK's built-in mechanism by listening for `compactBoundary` system events. The `/compact` command shows current status and the auto-trigger works transparently.

### W6: CJK Input Fix Addresses Real User Pain Point

Story 37.9's raw-mode UTF-8 character boundary detection solves a fundamental usability issue for Chinese users. The `utf8CharLength()` → `processBackspace()` chain correctly handles ASCII (1 byte), Chinese (3 bytes), and emoji (4 bytes). This was the right final story — it closes a UX gap that affects the primary user base.

## What Could Be Improved

### C1: Story 37.8 Had 3 CRITICAL Bugs — Session Resume Was Almost Non-Functional

Story 37.8 (会话恢复) had the highest bug density of the epic:
- **C1:** No session existence validation — `formatSessionNotFound()` was dead code, invalid session IDs silently failed
- **C2:** No running state check — `formatSessionAlreadyRunning()` was dead code, restoring current session was possible
- **C3:** `messageCount: 0` hardcoded — restore banner always showed "0 条消息"

All 3 were caught by adversarial review, not by tests. The story's tests passed because they tested the formatting functions in isolation, not the integration path.

**Root Cause:** The story implemented all formatting helpers first, then wired them into ChatCommand. The wiring step was treated as trivial but was the source of all 3 bugs.

**Recommendation:** For stories that connect multiple components (session validation + formatting + agent rebuild), add an explicit "wiring verification" task: "For each formatting function, trace its call site in ChatCommand and verify it's called with correct arguments."

### C2: Story 37.9 CJK Path Lost Continuation Support — Most Complex Integration

Story 37.9's initial implementation only replaced `readLine()` with `CJKInputHandler.readRawLine()`, losing:
- Bracket paste detection in raw mode
- Backslash continuation in raw mode
- The CRITICAL fix required implementing `readCJKInput()` and `readCJKContinuation()` private methods that duplicate MultiLineInputReader's continuation logic in raw mode

**Root Cause:** `CJKInputHandler` was designed as a drop-in replacement for `readLine()`, but `MultiLineInputReader` has multi-step input logic (continuation, paste detection) that doesn't compose with a simple function replacement.

**Recommendation:** When replacing a lower-level component, explicitly audit all callers' usage patterns. CJK input isn't just replacing `readLine()` — it's replacing the terminal I/O layer that MultiLineInputReader builds on.

### C3: AC10 Unknown Slash Command Was Dead Code (Story 37.1)

`handleUnknown()` was implemented and tested but never called from ChatCommand. `/foo` was sent directly to the agent. The review caught this as HIGH severity.

**Root Cause:** The REPL's slash command handling used `if let cmd = SlashCommand.parse(trimmed)` without an `else if trimmed.hasPrefix("/")` fallback for unknown commands.

**Recommendation:** Every `if let` pattern for command parsing should have an explicit `else if` for the "matched prefix but no valid command" case. This is a structural pattern, not a one-off mistake.

### C4: MarkdownTerminalRenderer Built But Not Wired Into Streaming (Story 37.4)

Story 37.4 built the `MarkdownTerminalRenderer` with bold, code, and code block ANSI rendering, but the streaming output path (`ChatOutputFormatter.handle(.partialMessage)`) directly `fputs` raw text without rendering. The renderer exists but is unused in practice.

**Root Cause:** Streaming Markdown rendering requires buffering (partial tokens may split `**bold**` syntax). The story chose the simpler path of direct output to avoid buffering complexity.

**Recommendation:** Mark this as explicit tech debt. The renderer can be wired in when buffering architecture is added (e.g., after `.assistant` events that contain the complete text).

## Lessons Learned

### L1: Wiring Verification Is a Separate Task, Not an Afterthought

Stories 37.1 (AC10 dead code) and 37.8 (3 CRITICALs) both failed at the wiring step — functions existed and passed tests but weren't connected to the REPL. The pattern: implement helper functions → test in isolation → "trivially" wire into ChatCommand → wiring has bugs.

**Key Insight:** For any story that connects new components to the REPL loop, add an explicit wiring verification task: "Trace each new function's call site in ChatCommand. Verify correct invocation, correct arguments, and correct branch handling (if/else if/else)."

### L2: Component Replacement Requires Caller Pattern Audit

Story 37.9 replaced `readLine()` with `CJKInputHandler.readRawLine()`, but `MultiLineInputReader` doesn't just call `readLine()` once — it has multi-step logic (continuation, paste detection, Ctrl+C handling). The replacement broke all of these.

**Key Insight:** When replacing a lower-level dependency, audit all callers' usage patterns. The caller may depend on behavior that isn't obvious from the replaced function's signature alone.

### L3: SDK Behavior Discovery During Implementation Is Valuable

Stories 37.0 (SDK default permission mode blocks without waiting) and 37.7 (SDK has built-in auto-compact) discovered important SDK behaviors during implementation. Both discoveries changed the implementation approach significantly.

**Key Insight:** Time spent reading SDK source code during story implementation is not wasted. Discovering "the SDK already does this" saves implementation effort and avoids reinventing the wheel.

### L4: Pure Function + Dependency Injection Pattern Scales Well

All Chat/ components use pure functions or injected closures:
- `SlashCommand.parse()` — Foundation only, no SDK dependency
- `SlashCommandHandler.handle()` — agent/config as parameters, not stored
- `BannerRenderer` — all static methods, no state
- `MarkdownTerminalRenderer` — all static methods
- `PermissionHandler.createCanUseTool()` — closure injection for I/O
- `MultiLineInputReader` — `readLineFn` / `isTTYFn` injection

This pattern made all components testable without mocks for SDK types.

**Key Insight:** The pure function + dependency injection pattern, established in Story 37.0, scaled cleanly through all 10 stories. It should be the default pattern for new CLI components.

### L5: Review Continues to Be Essential — All CRITICAL Bugs Were Caught

All 4 CRITICAL bugs across the epic (37.8's 3 + 37.9's 1) were caught by adversarial AI review, not by tests. Tests passed because they tested individual components, not integration paths.

**Key Insight:** This reinforces the lesson from Epic 32: tests verify component correctness, review verifies composition correctness. Both are essential.

### L6: Single-Day Epic Delivery Is Possible With Sequential Stories

All 10 stories were completed on 2026-06-07. The sequential dependency chain, pure function architecture, and comprehensive story specs (each ~200-400 lines of Dev Notes) enabled rapid, correct implementation.

**Key Insight:** Well-specified stories with clear dependency chains and consistent architectural patterns enable extremely fast delivery. The time invested in detailed story specs pays back in implementation speed.
