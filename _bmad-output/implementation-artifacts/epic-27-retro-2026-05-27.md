## What Could Be Improved

### C1: Test Count Claims Still Inconsistent — 3rd Consecutive Epic

Counts decrease then increase across stories within the same epic, which is impossible without removing tests between stories. This is the same issue flagged in Epic 25 (C4) and Epic 26 (C1). The dev agents appear to count from different compilation targets or intermediate builds.

**Root Cause:** Dev agents run `swift test` with different filter arguments or at different stages of the build, getting different total counts.

**Recommendation:** Standardise on `swift test --filter "AxionCLITests" 2>&1 | tail -1` for AxionCLI test count claims in story files.

### C2: Review Finding Rate Remains Elevated — ~18 Total Across 4 Stories

Epic 27 had ~18 review findings (1 CRITICAL, 8 HIGH, 7 MEDIUM, 2 LOW) across 4 stories (avg 4.5/story). This is consistent with Epic 26 (5.0/story) and Epic 25 (5.5/story). Common patterns:

- **Missing protocol abstraction** (27.1 HIGH): First-pass code directly instantiated concrete types
- **Missing parameter forwarding** (27.4 HIGH): `maxSteps` not passed to `buildSkillAgent()`
- **Builder injection seam bypassed** (27.4 HIGH): Direct class reference instead of protocol method
- **Dead code after refactoring** (27.4 HIGH): `processStreamFromAsyncStream()` left behind after migration

**Recommendation:** Add a pre-review self-check: "Does this code use the protocol seam or the concrete type? Are all parameters forwarded? Is any code no longer called?"

### C3: AC #3 in Story 27.3 Not Literally Met — Acceptable Deviation

The epic spec stated "AxionRuntime is created once with API handler set" but the implementation uses per-request AxionRuntime instances. The code review noted this as a deliberate architectural choice for true concurrency (Option A in dev notes). The handler registration IS centralised in `DaemonRuntimeManager`, but the runtime instances are per-request.

**Impact:** Low — the spirit of the AC (consistent handler setup, no per-request handler registration) is met. The literal wording (single instance) was a premature optimisation assumption.

**Recommendation:** When epic specs make architectural implementation assumptions, flag them as "suggested approach" rather than AC-level requirements.

### C4: ApiRunner SSE Wiring Incomplete for Skill Execution

Story 27.4's known limitation: `ApiRunner.runSkillAgent()` completion callback returns minimal data (no step summaries, no cost telemetry), and the `eventBroadcaster` SSE wiring is not connected through EventBusBridge. Skill execution through AxionRuntime works for EventBus handlers, but SSE consumers don't get skill events.

**Impact:** Medium — API consumers monitoring skill runs via SSE won't see real-time events. They'll only get the completion callback.

**Recommendation:** Schedule a follow-up story to wire EventBusBridge for skill SSE events.

## Lessons Learned

### L1: Two AI Models Can Collaborate Effectively With Strong Architecture Docs

GLM-5.1 and Claude Opus 4.7 produced consistent code when guided by the same architecture document, epic spec, and story conventions. The key enabler was the detailed Dev Notes sections in each story file, which provided model-specific context about existing patterns and constraints. Stories 27.2 and 27.4 (both GLM-5.1) correctly built on patterns established in 27.3 (Opus 4.7).

**Key Insight:** Detailed Dev Notes with "Patterns to Follow" and "Key Files to Touch" sections are the primary mechanism for cross-model consistency.

### L2: Actor Concurrency Requires Explicit Design Decisions

Story 27.3's discovery that AxionRuntime's actor isolation serialises all method calls was not obvious from the API surface. The decision to use per-request instances for concurrency was a correct and non-trivial architectural choice. Future actor-based components should document their concurrency expectations explicitly.

**Key Insight:** When designing actor-based APIs, document whether callers should expect serialisation or create separate instances for parallelism.

### L3: Dead Code Identification After Migration Needs Proactive Checking

Story 27.4 correctly removed `processStreamFromAsyncStream()` (HIGH issue found in review), but `RunOrchestrator.executeSkillDirectly()` was kept. The review process caught one case but missed the other. The lesson: after any "migration" story, explicitly enumerate all former callers of the old path and verify they're updated.

**Key Insight:** Migration stories should include a "Caller Audit" section that lists every call site of the old path and its migration status.

### L4: Epic Spec ACs Should Separate Requirements From Implementation Suggestions

Story 27.3's AC #3 ("AxionRuntime is created once") was an implementation suggestion, not a user-observable requirement. The code review correctly identified this as a deliberate deviation, but the mismatch between AC and implementation created confusion.

**Key Insight:** ACs should describe user-observable behavior. Implementation approaches should be in Dev Notes, not ACs.
