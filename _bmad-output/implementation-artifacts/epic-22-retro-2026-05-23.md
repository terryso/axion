## What Could Be Improved

### C1: Test Count Inaccuracy Persists — Third Consecutive Epic

Epic 20 retro (C3) identified test count inaccuracy. Epic 21 retro (C2) flagged it again. Now in Epic 22:

| Story | Claimed Count | Actual Count | Delta |
|-------|---------------|--------------|-------|
| 22.1 | 5117 | 5596 | -479 |
| 22.2 | 5160 | 5125 | +35 |
| 22.3 | 5202 | — | — |
| 22.4 | 5240 | 5241 | -1 |

Stories 22.1 and 22.2 have significant count discrepancies. The story automator agent consistently miscounts, likely because it counts at different points in the pipeline (after its own additions but before or after ATDD test additions).

**Root Cause:** The story automator runs tests at story completion time, but ATDD/QA tests are added in a separate phase. The automator's count is stale by the time QA is done.

**Recommendation:** The retrospective itself should always verify with `swift test` at the start, making the retrospective count authoritative. Stop relying on story completion notes for test counts.

### C2: Story 22.1 Had 2 HIGH Issues in Code Review Despite "Pure Types" Story

Story 22.1 was a protocol-only story (no I/O, no LLM calls), yet had 2 HIGH-severity issues:
- H1: `SkillSignal` id not normalized — diverges from `ExperienceSignal` pattern
- H2: Missing Codable test for `createdAt` round-trip

The normalization bug (H1) means two signals with the same logical content but different casing would get different IDs, breaking deduplication.

**Recommendation:** When mirroring patterns from other epics (e.g., `ExperienceSignal` → `SkillSignal`), do a line-by-line comparison to ensure all normalization logic is carried over.

### C3: Story 22.2 Had Redundant Signal Serialization — Caught in Review

Story 22.2's `LLMSkillEvolver` called `serializeSignals()` twice — once in the system prompt and once in the user message — doubling token usage with identical content. This was caught in review (H1) and fixed.

**Root Cause:** The developer followed the `LLMExperienceExtractor` pattern but didn't adapt the message structure for the different domain. `LLMExperienceExtractor` puts context in the system prompt; `LLMSkillEvolver` should do the same but the developer duplicated it into the user message as well.

**Recommendation:** When adapting patterns across domains, verify that message structure is purposefully adapted, not blindly copied.

### C4: `currentLifecycleState` Computed Property Uses Hardcoded Thresholds

Story 22.3 added `SkillUsageData.currentLifecycleState` as a computed property, but it uses hardcoded 30/90 day thresholds instead of the configurable `SkillUsageTrackerConfig` thresholds. The actual lifecycle evaluation in `SkillUsageTracker` uses the config properly.

**Impact:** The computed property gives a quick approximation but may diverge from the tracker's actual evaluation when custom thresholds are configured.

**Recommendation:** Either make `currentLifecycleState` accept a config parameter, or document clearly that it uses default thresholds and the tracker should be used for actual evaluations.

### C5: E2E Tests Continue to Require ANTHROPIC_API_KEY

All E2E tests (42 total across Epics 21-22) are skipped when `ANTHROPIC_API_KEY` is not set. Stories 22.3 and 22.4 added E2E tests for store persistence and lifecycle transitions, which could potentially run without an API key since they test file I/O, not LLM calls.

**Recommendation:** Separate E2E tests into "LLM-required" and "no-LLM-required" categories. Tests for stores and trackers can run in CI without API credentials.

## Lessons Learned

### L1: Pattern Mirroring Across Epics Is Highly Efficient but Requires Verification

The `ExperienceExtractor` → `SkillEvolver` pattern mirroring was very efficient — each story in Epic 22 had a direct counterpart in Epic 21:
- 22.1 ↔ 21.1 (protocol + types)
- 22.2 ↔ 21.2 (LLM implementation)
- 22.3 had no direct counterpart (new concept: usage tracking)
- 22.4 had no direct counterpart (new concept: curation)

However, the normalization bug (H1 in 22.1) and redundant serialization (H1 in 22.2) show that blind pattern copying introduces subtle bugs. **Mirroring is efficient, but line-by-line verification of normalization, validation, and message structure is essential.**

### L2: Atomic File Writes Need `removeItem` Before `moveItem` on Apple Platforms

Story 22.3 discovered that `FileManager.moveItem(at:to:)` fails when the destination file already exists on macOS. The fix is to `removeItem` the existing file first, then `moveItem`. This differs from the Hermes Python pattern where `os.replace()` atomically overwrites.

**Key Insight:** Atomic file replacement is platform-specific. On Apple platforms, you need `removeItem` + `moveItem`. On Python/POSIX, `os.replace()` handles it in one call.

### L3: Sidecar Files Separate Operational Data from User Content

The `.usage.json` sidecar pattern (from Hermes) cleanly separates operational telemetry (view counts, timestamps) from user-authored skill content (SKILL.md). This is a good pattern for any SDK feature that tracks metadata about user-defined content.

### L4: `precondition()` for Config Validation Catches Bugs in Release Builds

Story 22.3 introduced config validation via `precondition()` (not `assert()`), ensuring that invalid configs (`archiveAfterDays <= staleAfterDays`, negative thresholds) crash immediately in both debug and release builds. This is the right approach for config validation — fail fast and loud.

### L5: Value Types Eliminate Concurrency Concerns for Stateless Services

`SkillUsageTracker` and `SkillCurator` are both structs with no mutable state. They delegate persistence to actors (`SkillUsageStore`, `SkillCuratorStore`). This eliminates the need for locks, actor isolation, or synchronization in the business logic layer.
