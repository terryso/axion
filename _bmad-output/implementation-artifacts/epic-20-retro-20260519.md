## Cross-Story Patterns

### Success Patterns (Worth Preserving)

**1. "Pre-Deletion Audit Table" Prevents Catastrophic Mistakes**
The dev agent built a comprehensive reference verification table listing every symbol, its references outside dead code dirs, and verdict (DEAD/ALIVE). This caught 4 incorrectly classified files before any deletion occurred. Without this table, the build would have broken silently on multiple files.
**Lesson:** When deleting code, grep every symbol before deleting. "Delete first, fix later" is reckless for dead code cleanup.

**2. "Dependency Chain Analysis" for Protocol Conformance**
The OutputProtocol → TerminalOutput/JSONOutput dependency chain was analyzed: live files conformed to a dead protocol. The solution was surgical — remove conformance declaration, remove dead-typed methods, then delete the protocol. No shotgun approach.
**Lesson:** When live code depends on dead abstractions, remove the dependency surgically rather than keeping the abstraction alive.

**3. "Extract Before Delete" for Shared Types**
The Value enum was defined inside Step.swift (dead) but used by MCPClientProtocol and SkillExecutor (alive). Extracting it first, then deleting Step.swift, avoided compilation errors.
**Lesson:** Shared types embedded in dead files must be extracted before the file is deleted.

### Challenge Patterns

**1. "Epic Spec Inaccuracy" in Dead Code Classification**
The epic's story description listed 4 live files as dead. This is the same pattern as Epic 19's "Story File List 不准确" — planning artifacts diverge from actual code state.
**Root Cause:** The epic was written based on architectural assumptions (these files belong to the self-built Agent Loop) without verifying actual references via grep.
**Improvement:** Dead code cleanup stories should require a pre-implementation reference audit as an AC, not just a task.

**2. "Scope Expansion" During Story Execution**
Story 20.1 was planned as pure deletion but ended up creating new files (Value.swift, VisualDeltaTracker.swift) and rewriting tests. The actual work was more complex than "delete files and verify build."
**Root Cause:** Dead code cleanup is never just deletion — live code depends on types defined alongside dead code.
**Improvement:** Future dead code cleanup stories should budget for extraction and test rewrite work, not just deletion.

## Key Lessons

1. **Dead code cleanup is never just "delete files"** — Live code depends on types defined alongside dead code. Every deletion must be preceded by a reference audit and may require extraction of shared types. Lesson: Budget for extraction and test rewrite, not just deletion.

2. **The pre-deletion audit is the most valuable artifact** — The symbol-by-symbol reference verification table documented in Dev Notes is more valuable than the deletions themselves. It provides auditable proof that every deletion was safe. Lesson: In dead code cleanup, the audit document is the deliverable, not just the deletion commit.

3. **Epic specs can classify live code as dead** — The Epic 20 story description incorrectly marked 4 live files for deletion. Without the pre-deletion audit, the build would have broken. Lesson: Never trust the epic's dead code classification without grep verification.

4. **Phase 6 is complete** — Epic 19 (SDK alignment) and Epic 20 (dead code cleanup) together accomplish the Phase 6 goals: Axion's SDK usage is now correct (AgentBuilder + BuildConfig), and the self-built Agent Loop is fully removed. The codebase is cleaner, smaller, and more honest about what code is actually used.

5. **Carried technical debt is accumulating** — 10 debt items carried from previous epics remain unaddressed. While none are blocking, the accumulation means each subsequent phase will carry more baggage. Lesson: A dedicated "debt reduction" sprint may be needed before Phase 7.
