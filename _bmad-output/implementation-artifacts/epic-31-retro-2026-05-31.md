## What Went Well

### W1: Clean Story Dependency Chain — Zero Rework Between Stories

All 5 stories had a clean sequential dependency: 31.1 (storage) → 31.2 (agent tool) → 31.3 (review tool) → 31.4 (security integration) → 31.5 (CLI). Each story's "Files that must NOT change" section was respected across all subsequent stories. Zero stories needed to revisit a predecessor's implementation. This is the cleanest story chain in recent memory.

### W2: Security Scanner Design — Write-Time + Load-Time Dual Defense

The MemorySecurityScanner's dual-layer approach (write-time rejection for new content, load-time filtering for existing content) proved robust across Stories 31.1–31.4. The entry-level `scanEntry()` added in Story 31.4 correctly reused the existing `scan()` patterns, and the filtering in `buildUniversalMemoryContext()` was verified by dedicated tests. No security gaps were found in review.

### W3: Frozen Snapshot Emerged Naturally From Architecture

Story 31.1 discovered that frozen snapshot behavior was already achieved by the existing architecture — `buildSystemPrompt()` runs once at agent construction, and the SDK caches the result. No special snapshot caching mechanism was needed. Story 31.4 confirmed this with explicit frozen snapshot verification tests. This is a case where understanding the existing system eliminated unnecessary work.

### W4: Two Distinct Tools (MemoryTool + ReviewSaveUniversalMemoryTool) Was the Right Decision

The separation between agent-facing `MemoryTool` (add/replace/remove/read) and review-agent-facing `ReviewSaveUniversalMemoryTool` (add/replace only) proved clean. Different consumers, different capabilities, different registration paths. No tool was bloated with features only one consumer needed.

### W5: GLM-5.1 Delivered Consistent Quality Across 4 Stories

GLM-5.1 implemented Stories 31.1–31.3 and 31.5 with consistent patterns, proper test coverage, and correct use of existing codebase conventions (actor for I/O, struct for pure functions, ToolProtocol via class for reference capture). The review findings were mostly edge cases (missing test coverage, response format nuances) rather than fundamental design issues.

## What Could Be Improved

### C1: CRITICAL SDK Change Committed Only Locally (Story 31.3)

The `additionalReviewTools` parameter added to `ReviewOrchestrator` in the SDK was committed to `.build/checkouts/open-agent-sdk-swift/` but NOT pushed upstream. If `swift package resolve` runs, the change is lost. This is the most dangerous finding of the entire epic — the entire review-agent memory pipeline depends on a local-only SDK modification.

**Recommendation:** Immediately push the SDK change, tag as v0.7.0, and update `Package.swift` dependency version. This is a blocking prerequisite before any production use of Epic 31 features.

### C2: `replace()` and `add()` Return Bool — Cannot Distinguish Error Types

Carried from Story 31.2 (M2): `UniversalMemoryStore.replace()` returns `false` for both "keyword not found" and "char limit exceeded". The MemoryTool and ReviewSaveUniversalMemoryTool cannot provide meaningful error messages. The review noted this is because modifying `UniversalMemoryStore` was out of scope, but the user experience degradation is real.

**Recommendation:** Change `UniversalMemoryStore` methods to return a `Result` type or an enum distinguishing `success`, `notFound`, `charLimitExceeded`. This is a small refactoring that would significantly improve error feedback.

### C3: `§` Delimiter Is Fragile

Noted in Story 31.1 and carried forward: if user content contains `§`, parsing breaks. This is an acknowledged limitation that affects both storage parsing and entry filtering.

**Recommendation:** Consider using a more robust delimiter or escaping strategy in a future story. Low priority but worth documenting.

### C4: Review Findings Show Pattern of Missing Edge-Case Test Coverage

Across Stories 31.1–31.5, the most common review finding was missing test coverage for specific edge cases: `target: "user"` tests in 31.2, security scan response format in 31.3, stderr output in 31.4. This suggests a systematic gap in the AI agent's test generation where it covers the happy path and primary error paths but misses secondary paths.

**Recommendation:** Add a self-check step: "For each action parameter combination, verify all targets are tested." This aligns with the Epic 30 retro action item about pre-review self-checks.

## Lessons Learned

### L1: SDK Changes Must Be Upstreamed Before Depending on Them

Story 31.3's CRITICAL finding: the `additionalReviewTools` SDK parameter was committed locally and is at risk of being lost on `swift package resolve`. Any SDK modification that Epic 31 depends on MUST be upstreamed immediately.

**Key Insight:** SDK changes should be upstreamed as a separate commit/PR BEFORE starting the stories that depend on them. Do not accumulate local-only SDK changes.

### L2: Frozen Snapshot Doesn't Need Special Code — Architecture IS the Snapshot

The frozen snapshot behavior (Story 31.1 AC #2, Story 31.4 AC #4–5) was naturally achieved by the existing `buildSystemPrompt()` once-at-construction pattern. No caching, no snapshot objects, no invalidation logic needed.

**Key Insight:** Before implementing "caching" or "snapshot" features, verify whether the existing architecture already provides the desired behavior. The simplest frozen snapshot is one you don't write.

### L3: Dual-Layer Security (Write-Time + Load-Time) Is the Right Pattern for LLM Memory

Write-time rejection prevents bad content from entering storage. Load-time filtering catches content that was injected by other means (manual edit, disk corruption, SDK bug). Both layers are independently testable.

**Key Insight:** For any data that flows into LLM prompts, always have two independent security layers: one at the write boundary and one at the read boundary.

### L4: Review Agent Only Needs Add/Replace — Not Full CRUD

The `ReviewSaveUniversalMemoryTool` intentionally provides only `add` and `replace` actions (no `remove` or `read`). The review agent's job is to save discoveries, not manage existing memory. This simplification reduced implementation complexity without sacrificing capability.

**Key Insight:** Design tools based on the consumer's actual needs, not by mirroring the full CRUD surface. Review agents save; they don't browse.

### L5: Multiple Actor Instances Writing to Same Files Is Safe with Atomic Writes

The decision to create separate `UniversalMemoryStore` actor instances for `MemoryTool`, `ReviewSaveUniversalMemoryTool`, `MemoryContextProvider`, and CLI commands — all writing to the same MEMORY.md/USER.md files — was validated as safe because underlying `FileManager` writes use `atomically: true`.

**Key Insight:** Swift's `actor` + atomic file writes provides sufficient isolation for concurrent file access. No external locking needed when each write is atomic.

### L6: Error Response Format Matters for SDK Integration

Story 31.3 discovered that the tool's success response used `{"status":"ok"}` instead of `{"success":true}` — invisible to SDK's `summarizeActions()`. This is a subtle integration contract that's easy to miss.

**Key Insight:** When implementing `ToolProtocol`, match the SDK's expected response format exactly. Check how the SDK consumes tool results before defining response schemas.
