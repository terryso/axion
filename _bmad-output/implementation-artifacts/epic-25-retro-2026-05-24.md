## What Went Well

### W1: Two-Phase Pipeline Design Matches Hermes Architecture Perfectly

The two-phase execution model (mechanical → intelligent) mirrors Hermes `run_curator_review()` faithfully:

- Phase 1: `SkillCurator.run()` handles time-based state transitions (already existed from Epic 22)
- Phase 2: `IntelligentCurator.execute()` forks a curator agent for LLM-driven consolidation

Each phase is independently testable. Phase 1 results are preserved even if Phase 2 fails. This resilience pattern (AC8) worked correctly.

### W2: Interface-First Pattern Continues to Work Reliably

Zero interface changes needed after Story 25.1. This pattern has now worked reliably across Epics 21–25.

### TD2: YAML Parsing Uses String Matching Without Robust Error Reporting

`IntelligentCurator.parseYAMLSummary()` uses regex-based string matching to extract consolidations and prunings from the LLM's YAML output. If the LLM produces malformed YAML, the parser silently returns empty arrays.

**Impact:** Medium — malformed YAML from the LLM would result in a report showing no consolidations/prunings even if the curator agent actually performed operations via tool calls.
**Priority:** Medium — consider logging a warning when YAML parsing returns empty results but tool calls were made.

### TD3: No Curator Scheduling — IntelligentCurator Requires Manual Invocation

`IntelligentCurator.execute()` must be called explicitly by the application. There's no automatic scheduling (unlike `ReviewOrchestrator` which hooks into `sessionEnd`). Applications must decide when and how often to run curation.

**Impact:** Low — the SDK is a library, not a framework. Scheduling is the application's responsibility.
**Priority:** Low — document the recommended scheduling interval (Hermes runs after every session that passes the review interval threshold).

### TD4: `absorbedInto` on SkillUsageData Is Only Set at Archive Time

The `absorbedInto` field added in Story 25.2 is only populated when `CuratorArchiveTool` runs. There's no way to see the merge history without scanning all usage records.

**Impact:** Low — the data is queryable, just not aggregated.
**Priority:** Low — consider a `mergeHistory()` query method on `SkillUsageStore` if consumers need it.

### TD5: E2E Tests Require ANTHROPIC_API_KEY — Continued from Epic 24

## Lessons Learned

### L1: Interface-First Pipeline Assembly Is Now a Proven Pattern

For the 5th consecutive epic (21–25), the interface-first approach has worked flawlessly: define types and interfaces in Story 1, implement and compose in subsequent stories. Zero interface changes needed after Story 1 in every epic.

**Key Insight:** This pattern is no longer experimental — it's a validated architectural principle for this codebase.

### L2: Two-Phase Execution with Error Resilience Is a Valuable Pattern

The mechanical → LLM two-phase pattern (with Phase 1 preserved on Phase 2 failure) worked well. This is the same pattern used by Hermes and translates cleanly to the SDK. The key insight is: always guarantee that deterministic results (Phase 1) are never lost due to probabilistic failures (Phase 2).

**Key Insight:** For any pipeline that combines deterministic and non-deterministic phases, wrap the non-deterministic phase in error handling that preserves deterministic results.

### L3: Rendering Methods Need Explicit Output Format Examples in Specs

Story 25.4's AC4 specified a summary line format ("Skills: {before} → {after} ({delta})") that the initial implementation didn't match. The dev notes described the format textually but didn't include an expected output example. When specs describe output formats (Markdown, YAML, console), an example of the expected output would prevent mismatches.

**Key Insight:** For rendering/presentation stories, include a concrete example of expected output in the spec or dev notes.

### L4: Test Count Claims Need Verification — Accuracy Has Regressed

Epic 24 retro celebrated improved test count accuracy, but Epic 25 shows regression (Story 25.3 claims 6,120 vs actual 5,640). The story automator or dev agent may be counting tests from a different test target or including non-test compilations.

**Key Insight:** Test count claims should be verified by `swift test` output, not intermediate build counts.
