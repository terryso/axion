## What Could Be Improved

### C1: HIGH-Severity Issues in Code Review for Every Implementation Story

Stories 21.2, 21.3, and 21.4 each had HIGH-severity issues found during code review. Common categories:

- **Incomplete wiring:** Config fields defined but not used in handler logic (Story 21.3 `domains`)
- **Missing init paths:** Alternate initializers not updated for new fields (Story 21.3 `init(from config:)`)
- **Silent failure masking:** Errors swallowed without logging (Story 21.2 malformed JSON)

**Root Cause:** The development agent implements the happy path first and doesn't systematically check all initialization paths, config field usage, and error logging points.

**Recommendation:** Before marking implementation complete, verify: (1) all init paths assign new fields, (2) all config fields are actually read in handler logic, (3) all error paths log before returning empty/nil.

## Lessons Learned

### L1: Protocol-First Stories Are Low Risk and High Value

Story 21.1 (protocol + types only) was the safest story in the epic — purely additive with zero modified files and no runtime behavior. It established the vocabulary for all subsequent stories. This pattern should be preferred for any epic introducing new abstract concepts.

### L2: @unchecked Sendable + SharedMockState Is the Standard Test Pattern for Swift 6

Three consecutive stories discovered the same solution for test mocks under Swift 6 strict concurrency: wrap mutable mock state in a `final class SharedMockState<T>: @unchecked Sendable` and access it via shared reference. This pattern should be documented as the canonical approach.

### L3: IntervalTracker Reference Wrapper Pattern for Sendable Closures

Story 21.3 needed mutable interval tracking inside a `@Sendable` closure. The solution was an `IntervalTracker` reference wrapper (class with `@unchecked Sendable`). This is a reusable pattern for any hook or callback that needs to maintain mutable state across invocations.

### L4: Provider Guard Prevents Cross-Provider Feature Leakage

Story 21.3's review added a provider guard — `LLMExperienceExtractor` uses the Anthropic API, so the hook should only be registered when using the Anthropic provider. This prevents confusing errors when using OpenAI or other providers. Future stories integrating provider-specific features should include provider guards from the start.

### L5: Deterministic First-Match-Wins for Security Scanning

Story 21.4's scanner checks rules in a fixed order (content length → blocked domain → blocked pattern → confidence ceiling). This makes behavior deterministic and testable. When building multi-rule validators, establish a fixed priority order.
