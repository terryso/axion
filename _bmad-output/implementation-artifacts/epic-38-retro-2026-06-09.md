## What Went Well

### W1: Story Dependency Chain Was Flawlessly Planned and Executed

The story sequence 38.0 → 38.1 → ... → 38.7 had a clean dependency chain that required zero rework:

- 38.0 established `ChatComposer` (raw mode + `ComposerMode` state machine + `ComposerDraft` snapshot) → all subsequent stories build on this foundation
- 38.1 built `ChatTheme` + `TranscriptRenderer` → consumed by 38.2 (slash popup), 38.3 (approval renderer), 38.6 (file search popup)
- 38.2 created `SlashCommandContext` + `SlashPopup` pattern → reused as template by 38.4 (history search), 38.6 (file search)
- 38.3 established `ApprovalDecision` + `SessionAllowList` → independent, clean module boundary
- 38.4 completed `ComposerMode.historySearch` + `ComposerDraft.snapshot/restore` lifecycle → validated the composer architecture
- 38.5 added `InputQueue` → proved ChatComposer's extensibility for new Ctrl+ shortcuts
- 38.6 created `FileSearcher` + `/diff`/`/status` → demonstrated slash command extensibility
- 38.7 added `/new`/`/fork`/`/archive` → completed the session lifecycle story

Zero stories required going back to modify a predecessor's architecture. The dependency graph was acyclic and correctly topologically sorted.

### W2: ComposerMode State Machine Scaled Elegantly

Story 38.0 defined 5 `ComposerMode` cases (`normal`, `slashPopup`, `historySearch`, `fileSearch`, `approval`). Every subsequent story only **filled in** the implementation of an existing mode — no new modes were needed, no existing modes needed modification:

This validated the upfront design: pre-defining all modes in 38.0 and having subsequent stories only implement them was a resounding success.

### W3: Pure Function + DI Pattern Consistently Applied

Epic 37 established the pattern (L4). Epic 38 took it to the next level across all 8 stories:

- **25+ new structs/enums** are pure (zero I/O, zero external dependencies)
- **All I/O** routed through injected closures (`writeStdout`, `writeStderr`, `readUserInput`, `confirmFn`, `processLauncher`, `envVar`, `launchProcess`, `fileSearcher`)
- **All terminal interactions** testable via `MockKeyReader` injection
- **Zero test files** depend on a real terminal, real file system, or real process

This consistency is remarkable for 8 stories implemented by 3 different AI models in a single day. The story specs' "绝对禁止" sections enforcing this pattern were extremely effective.

This conscious adaptation (rather than blindly copying) kept the implementation focused and appropriate for the REPL context.

## What Could Be Improved

### C1: Story 38.5 "Busy-turn Input Queue" Implemented as Phase 1 Half-Concurrency

Story 38.5's AC1 explicitly requires "agent 执行时先输入下一条消息并排队". The implemented solution is a Phase 1 half-concurrency approach: users can only queue messages via Ctrl+Q during the readInput phase, not during agent streaming. The dev notes acknowledge this gap and leave true concurrent input for a "Phase 2".

**Root Cause:** `ChatComposer.readInput()` blocks on `read()` system call. Making it cancellable requires refactoring to `select()` + timeout polling, which was correctly identified as high-risk.

**Impact:** The core AC "在 agent 还在执行时先输入" is not fully met. Users must wait for agent completion before queuing the next message.

**Recommendation:** Mark this as explicit tech debt. The Phase 2 implementation (non-blocking `KeyEventReader`) should be a dedicated story in a future epic.

### C2: Story 38.3 AC6 "快捷键无需 Enter" Not Implemented

Story 38.3 (审批中心 v2) AC6 specifies "用户按 y/a/p/d/Esc 直接选择对应决策，无需 Enter 确认". The implementation uses `readLine()` inside the `canUseTool` async callback, which requires Enter.

**Root Cause:** `canUseTool` is called from within the SDK's async streaming context. Using raw mode `read()` here would interfere with the agent streaming output. The review noted this as "已知限制".

**Impact:** Every approval requires typing a letter + pressing Enter. Two keystrokes instead of one.

**Recommendation:** This could be addressed by refactoring `canUseTool` to use the `ChatComposer`'s raw mode event loop (perhaps by queuing approval requests and processing them in the REPL main loop). This is a significant architectural change.

### C3: Story 38.7 rebuildAgent() Code Duplication (TD5 Not Addressed)

Story 38.7's review noted 4 places in `ChatCommand` where agent rebuild logic is duplicated (/new, /fork, /resume, initial setup). The review explicitly decided to defer extraction: "mutating func + 10+ 局部变量，提取风险高".

**Root Cause:** ChatCommand is a large imperative function with 10+ local variables referenced across the rebuild path. Extracting to a method requires either passing all variables as parameters or converting to a class with mutable properties.

**Impact:** Any change to agent rebuild logic (e.g., adding a new state variable) must be replicated in 4 places.

**Recommendation:** This debt is accumulating. Consider a `SessionContext` struct that bundles all rebuild-related state, making the rebuild method parameterizable.

## Lessons Learned

### L1: Pre-Defining Enum Cases Enables Parallel Story Implementation

Story 38.0 defined all 5 `ComposerMode` cases upfront. Subsequent stories (38.2, 38.4, 38.6) each implemented one mode independently. This pattern — "define all enum cases in the foundation story, implement in subsequent stories" — should be the default for state machine-based features.

**Key Insight:** The upfront cost of defining unused enum cases is minimal. The benefit of having a stable interface that multiple stories can independently implement against is enormous.

### L2: Half-Concurrency Is a Valid Delivery Strategy

Story 38.5's Phase 1 approach (Ctrl+Q queuing during readInput phase, not during streaming) is a pragmatic compromise. It delivers 70% of the user value at 20% of the implementation risk.

**Key Insight:** When full concurrency is architecturally challenging (blocking I/O, shared mutable state), a half-concurrency approach that explicitly documents the gap is better than either blocking indefinitely or taking on high-risk refactoring.

### L3: Story Spec Quality Directly Correlates With Implementation Quality

Story 38.4 (implemented by Claude Opus 4.8) had the most detailed dev notes (~400 lines of interaction flows, state machine diagrams, and error handling tables) and produced the fewest review issues. Stories with thinner specs (38.5, 38.7) had more issues.

**Key Insight:** The story spec is the "contract" between the planner and the implementer. More detailed specs → fewer implementation surprises → fewer review issues. The time invested in 38.4's spec paid for itself in implementation quality.

### L4: Codex Reference Architecture Accelerates Design Decisions

Every story included a "Codex 架构参考" table mapping Codex Rust files to Axion Swift adaptations. This gave implementers a concrete reference point for architecture decisions rather than designing from scratch.

**Key Insight:** Having an explicit reference architecture (even from a different language/platform) dramatically reduces design ambiguity. The "reference, don't copy" approach worked perfectly.

### L5: Pure Function Architecture Makes Refactoring Safe

When Story 38.2 needed to rework slash popup rendering, or 38.6 needed to add file search to the same event loop, the pure function pattern meant changes were localized. No mock state needed updating.

**Key Insight:** The pure function + DI pattern from Epic 37 (L4) proved its worth again. Every subdirectory (Composer/, Theme/, Approval/) can be modified independently without risk of cross-contamination.

### L6: AI Model Selection Matters for Complex Integration Stories

Claude Opus 4.8 produced Story 38.4 (the most complex ChatComposer integration — history search + external editor) with fewer issues than GLM-5.1 produced simpler stories. The model's superior code reasoning showed in the edge case handling.

**Key Insight:** For stories with complex state machine interactions and integration testing, using the highest-capability model available is worth the cost. For simpler CRUD-style stories (slash command metadata, session workflow), mid-tier models are sufficient.
