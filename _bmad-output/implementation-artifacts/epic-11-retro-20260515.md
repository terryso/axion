## Story 逐一分析

### Story 11.1: Agent 项目模板与脚手架 CLI

**亮点：**
- 18 个测试全部通过，覆盖参数解析、文件生成、模板替换、边界情况
- ScaffoldCLI 作为独立可执行目标的设计决策正确 — 不受 SPM Command Plugin 限制
- 模板内容嵌入为 Swift String 常量，零外部依赖
- 支持 basic 和 mcp-integration 两种模板类型，覆盖最小骨架和 MCP 集成场景
- 模板中的 HelloWorldTool 演示了 String 和 ToolExecuteResult 两种返回模式

**挑战：**
- **CRITICAL: `AnyTool` 类型不存在于 SDK** — 模板中使用了 `AnyTool` 而非 `ToolProtocol`，导致生成的项目编译失败
- **HIGH: `greetingTool` 的 inputSchema 与 Codable Input 类型不匹配** — schema 声明 `title` 字段但 Swift struct 用 `name`
- **HIGH: `greetingTool` 定义但未使用** — `createExampleTools()` 只返回 helloTool，greetingTool 是死代码
- Review 发现 6 个问题全部自动修复

**决策记录：**
- 脚手架放在 OpenAgentSDK 仓库（方案 A），随 SDK 分发
- 使用 `agent.prompt()` 而非 `agent.stream()` 用于模板简单演示
- 使用 `ToolProtocol` 作为工具数组类型（而非不存在的 `AnyTool`）

**亮点：**
- 29 个测试全部通过（新增 11 个），无回归

- **LOW: HelloWorldTool.swift 文件名有误导性** — 包含 5 个工具但文件名只提到 HelloWorld
- Review 发现 5 个问题，全部修复

**决策记录：**
- 工具注册示例覆盖 4 种 `defineTool` 重载，提供对比表
- Hooks 模板使用 `HookRegistry.register(.preToolUse)` + matcher 正则模式
- MCP 工具命名空间规则：`mcp__{serverName}__{toolName}`

## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "前序 Story 经验显式传递"模式**
Story 11.1 的 Dev Notes 中明确记录了 5 条关键经验（AnyTool 不存在、inputSchema 一致性、双重转义等），Story 11.2 在 Dev Notes 中引用并应用了这些经验。结果是 11.2 没有重复 11.1 的 `AnyTool` 错误。
**启示：** 每个 Story 的 Dev Notes 中的"关键经验"章节是防止知识流失的关键机制，值得在所有 Epic 中推广。

**2. "模板嵌入 + 零外部依赖"的脚手架策略**
ScaffoldCLI 将所有模板内容嵌入为 Swift String 常量，不依赖模板引擎或外部文件。这使得脚手架工具可以随 SDK 单一分发，用户无需额外安装任何工具。
**启示：** 工具/脚手架类项目应优先考虑嵌入式模板而非外部模板引擎，降低用户安装门槛。

**3. "资产审计 → 复用"的文档策略**
Story 11.3 在开始前系统审计了 OpenAgentSDK 的现有文档资产（README 471 行、30+ Examples、Story 11.2 模板），发现 5 个核心场景已有覆盖。这避免了重复造轮子，将工作聚焦在组织和索引上。
**启示：** 文档类 Story 应先做资产审计，复用已有内容而非从零开始。

**4. Review 一致性高，API 命名错误被系统性捕获**
三个 Story 的 Review 共发现 16 个问题，其中 6 个与 SDK API 命名准确性相关（AnyTool、canUseTool 签名、allowed/disallowed、Permission Modes）。Review 机制有效防止了不准确的 API 文档流入开发者手中。
**启示：** 涉及 SDK API 文档/模板的 Story，Review 应特别关注 API 签名和命名准确性。

**5. 跨仓库修改协调无冲突**
Epic 11 同时修改了 OpenAgentSDK 和 Axion 两个仓库的文件，三个 Story 之间的修改没有产生冲突。这得益于 Story 11.1/11.2 主要修改 OpenAgentSDK 模板，Story 11.3 同时修改 Axion 内联文档。
**启示：** 跨仓库 Epic 应在每个 Story 中明确"哪个仓库修改什么"，避免并行修改同一文件。

### 挑战模式（需要改进）

**1. SDK API 命名不一致在模板编写中反复出现**
三个 Story 中有 6 个问题与 SDK API 命名相关：`AnyTool`（不存在）、`canUseTool` 签名错误、`allowed`/`disallowed`（应为 `allowedTools`/`disallowedTools`）、Permission Modes 不完整。这表明 SDK API 文档或自动补全支持不够完善。
**根因：** 开发者在编写模板代码时依赖记忆而非 SDK 源码验证，SDK API 没有自动补全支持（SPM executable target 不像 Xcode 项目那样提供实时补全）。
**改进：** 编写 SDK API 相关代码（模板/文档/示例）前，必须先读取 SDK 源码中的类型定义进行验证。

**2. 模板字符串的双重转义增加心智负担**
Swift String 中的 `\()` 插值在模板源码中需要写成 `\\()`，嵌套多行字符串 `"""` 会与外层冲突。Story 11.2 专门记录了这个陷阱。
**根因：** 模板内容嵌入为 Swift String 常量的副作用 — 生成内容的转义字符需要在源码中双重转义。
**改进：** 考虑将模板内容移至外部文件（Bundle resource），避免转义地狱。短期解决方案是在 Dev Notes 中显式提醒此陷阱。

**3. NFR 超出但未提前预警**
Story 11.3 的文档总大小 61.4KB 超出 NFR 50KB 限制。虽然 Review 中接受了这个超出（"文档质量优先"），但 NFR 约束在设计阶段就应该被评估。
**根因：** NFR 约束没有在设计阶段转化为具体的文档大小分配计划（如"每个文档不超过 8KB"）。
**改进：** 当 NFR 有量化约束时，在设计阶段将其分解为每个组件的预算。

**4. 测试中的硬编码路径**
Story 11.3 的 DocumentationTests 使用了硬编码绝对路径，CI 环境不兼容。这类问题在 Epic 10 中也出现过（HotkeyConfigManagerTests 写入真实文件系统）。
**根因：** 测试实现时未考虑环境兼容性，直接使用了开发环境的路径。
**改进：** 测试中引用外部项目路径时，使用环境变量或 `ProcessInfo.processInfo.environment` 动态解析。

### 突破时刻

**1. "资产审计优先"的文档策略显著提高效率**
Story 11.3 开始前审计发现 30+ Examples 已覆盖核心场景，6 份文档内容可以从 README 和 11.2 模板提取。这使 Story 11.3 的工作量从"创建所有内容"降低到"组织和索引已有内容"。

**2. `defineTool` 4 种重载的对比表成为核心教学工具**
Story 11.2 将 `defineTool` 的 4 种重载整理为对比表（Input 类型 × 返回类型 × 用途），使第三方开发者能快速选择合适的重载。这种"对比表"模式适合所有多态 API 的文档化。

**3. 跨 Story 的经验传递机制有效运作**
Story 11.1 的 `AnyTool` 错误 → 11.2 显式记录避免 → 11.3 不再出现。Story 11.2 的 `allowedTools`/`disallowedTools` 修正 → 11.3 的 API 文档直接使用了正确命名。这种跨 Story 的经验传递防止了重复犯错。

## 关键经验

1. **SDK API 文档/模板中的命名错误是"最危险的 bug"** — 普通代码 bug 在编译或运行时暴露，但 SDK 文档和模板中的 API 命名错误（如 `AnyTool`、`canUseTool` 签名、`allowed` vs `allowedTools`）会误导所有使用 SDK 的第三方开发者，产生连锁错误。Epic 11 的 16 个 Review 问题中有 6 个是 API 命名相关，占比 37.5%。教训：涉及 SDK API 的文档和模板代码，必须从 SDK 源码验证每一个类型名和函数签名，不能依赖记忆。

2. **跨 Story 的经验传递机制是防止重复犯错的最后防线** — Epic 11 最成功的设计是 Story 11.1 的 Dev Notes 中明确记录了 5 条关键经验，11.2 在开始前引用并应用了这些经验。结果：11.1 的 `AnyTool` 错误在 11.2/11.3 中完全消除。这种"前序 Story 的教训显式传递给后续 Story"的机制应该成为所有 Epic 的标准实践。

3. **"资产审计优先"是文档类任务效率最大化的关键** — Story 11.3 在实施前先审计了现有资产（30+ Examples、README、11.2 模板），发现核心场景已覆盖。这使工作量从"创建所有内容"降低到"组织和索引"。教训：文档/内容类任务应先做资产盘点，明确"已有"和"需新建"的边界，避免重复造轮子。

4. **跨仓库 Epic 需要更明确的"谁改什么"约定** — Epic 11 横跨 OpenAgentSDK 和 Axion 两个仓库，虽然三个 Story 之间没有冲突，但 Dev Notes 中需要反复说明"这个 Story 修改哪个仓库的什么文件"。教训：跨仓库 Epic 应在 Story 规格中包含一个"仓库影响矩阵"，明确每个 Story 的仓库修改范围。

5. **Review 机制是 SDK 文档准确性的最终保障** — Epic 11 的 Review 发现并修复了所有 6 个 API 命名错误。如果没有 Review，这些错误将直接传递给使用脚手架创建项目的第三方开发者，造成"第一步就编译失败"的糟糕体验。教训：SDK 生态类工作的 Review 应特别关注 API 准确性，可能需要交叉验证 SDK 源码。
