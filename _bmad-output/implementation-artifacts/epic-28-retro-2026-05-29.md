**Impact:** Medium — any future change to the HTTP API execution pipeline must be applied in two places.

### C3: AC Violations Caught in Review — JSON Key Mismatch (28.4)

Story 28.4's review found that AC #4 specifies `"status"` and `"active_tasks"` as JSON keys, but implementation used `"state"` and `"active_task_count"`. This was fixed in review, but the deviation from spec should have been caught during implementation.

**Recommendation:** Before implementing Codable structs, cross-reference CodingKeys against AC-specified JSON field names.

### C4: Review Finding Rate Remains Elevated — ~27 Total Across 4 Stories

Epic 28 had ~27 review findings (9 HIGH, 10 MEDIUM, 4 LOW) across 4 stories (avg 6.75/story). This is higher than Epic 27 (4.5/story) and Epic 26 (5.0/story). Common patterns:

- **AC violations** (28.4 HIGH): JSON keys didn't match spec, missing log paths in output
- **Dead code left behind** (28.3 HIGH): `static let daemonLabel`, `GatewayNotImplementedError`
- **State corruption on error paths** (28.2 HIGH): `server.start()` throwing left state stuck at `.running`
- **Missing parameter forwarding** (28.4 HIGH): CodingKeys didn't map to AC-specified names

**Recommendation:** Add a pre-review self-check: "Does every Codable struct's CodingKeys match AC-specified JSON field names? Is any code no longer called after this change?"

### C5: Signal Handler Wiring Remains Untested

Story 28.2 Task 3.4 required "Test signal handler wiring via protocol mock" but the test just calls `stop()` directly. Story 28.4 noted the same limitation. Signal handler wiring requires integration tests (actual process + signals), which the unit test framework cannot verify.

**Impact:** Low — signal handling works in practice (manual testing), but there's no automated regression protection.

**Recommendation:** Accept this as an integration-level concern. Document in test plan for manual acceptance testing.

## Lessons Learned

### L1: Explicit Dev Notes Warnings About Anti-Patterns Are Read But Not Always Followed

Story 28.2's Dev Notes explicitly warned "should NOT copy-paste ServerCommand's code" for runHandler. The warning was read (the code review references it) but the implementation still duplicated the code. Warnings alone are insufficient — the story should have provided the extracted shared function as a required task.

**Key Insight:** When an anti-pattern is identified in Dev Notes, provide the correct implementation path as a task, not just a warning.

### L2: DaemonService Parameterization Is a Good Pattern for Service Reuse

Parameterizing DaemonService with init defaults rather than creating a new class proved clean and maintainable. The regression test for existing DaemonCommand behavior was crucial for confidence. This pattern should be used for other shared services in future epics.

**Key Insight:** Parameterize shared services with defaults rather than subclassing or duplicating.

### L3: AC-Specified JSON Field Names Need Explicit CodingKey Mapping

Story 28.4 had a mismatch between AC-specified JSON keys (`"status"`, `"active_tasks"`) and implementation keys (`"state"`, `"active_task_count"`). This was caught in review but should have been caught during implementation.

**Key Insight:** When ACs specify exact JSON field names, create the CodingKeys mapping before implementing the struct.

### L4: Gateway Status Fallback Pattern Is Reusable

The HTTP query → DaemonService fallback pattern in GatewayStatusCommand is a useful pattern for any CLI command that needs to query a potentially-running service. Future commands can follow this approach.

**Key Insight:** HTTP-first, launchd-fallback is the right pattern for querying running services from CLI.
