## Story 逐一分析

### Story 8.1: 多窗口状态追踪与上下文管理

**亮点：**
- 5 个新单元测试，全部通过（1189 total, 0 failures）
- 增量扩展设计精准：仅修改 7 个文件，无新文件创建
- WindowInfo.zOrder 使用 CGWindowList 天然 z-order（数组索引），实现简单可靠
- WindowState.appName 通过 NSWorkspace 查找，decodeIfPresent 向后兼容
- Trace 增强：tool_use 从 input JSON 提取 window_id/pid，tool_result 从 content JSON 提取 app_name/window_id
- Planner prompt 新增 Multi-Window Workflow 章节，覆盖窗口发现、切换、最小化恢复、剪贴板传递

**挑战：**
- 无 review 问题记录 — Story 8.1 可能缺少独立 review 环节（Dev Agent Record 无 Review 部分）

**决策记录：**
- zOrder 用 CGWindowList 返回顺序的索引实现，值越小越靠前
- 最小化窗口恢复通过 prompt 引导 LLM 调用 activate_window，无需新的自动恢复逻辑
- Trace 窗口上下文从工具调用参数和返回值中提取，不需要新的 trace 字段

### Story 8.2: 跨应用工作流编排

**亮点：**
- 3 个新测试，全部通过（1192 total, 0 failures）
- **零 Swift 代码变更** — 整个 Story 仅修改 planner-system.md 和测试文件
- Prompt 工程作为主要实现手段，充分利用已有架构能力
- 6 步跨应用工作流模板设计清晰：discover → source → verify → switch → target → verify
- 失败恢复策略完备：替代应用建议、AX tree fallback、pause_for_human 兜底

**挑战：**
- 无 review 问题记录 — 与 8.1 相同，可能缺少独立 review 环节
- AC5（端到端跨应用操作）依赖真实 macOS 环境，单元测试无法覆盖

**决策记录：**

- 剪贴板验证通过 get_window_state 确认焦点元素包含预期内容
- 应用不存在时 Planner 建议用户安装或选择替代应用

### Story 8.3: 窗口布局管理

**亮点：**
- 4+3 个新测试（初始 4 + review 补充 3），全部通过（936+ total, 0 failures）

## 关键经验

1. **"Prompt 工程优先"是编排类功能的最佳策略** — Epic 8 证明，当核心能力（Helper 工具、SDK Agent Loop）已经就绪时，编排类功能（多窗口协调、跨应用工作流）应优先通过 prompt 工程实现。这不仅更快、更安全，而且更容易迭代和回退。只有需要新能力时（如窗口布局）才创建新工具。

2. **LLM tool-use 循环是多窗口协调的理想架构** — SDK 的 Agent Loop 天然支持 LLM 在不同窗口间切换操作。不需要额外的编排层、状态跟踪器或窗口管理器。Planner 通过 prompt 指导编排，Executor 通过工具调用执行。这一架构选择在 Epic 8 得到了充分验证。

3. **分层 Prompt 架构类似代码模块化** — planner-system.md 的能力指导可以像代码模块一样分层构建：基础层（窗口发现）、编排层（工作流模式）、布局层（窗口排列）。每一层独立、可测试、可独立修改。

4. **Epic 8 的 Review 一致性问题值得警惕** — Story 8.3 的 review 发现了 7 个问题（包括 cascade 遗漏和 NSScreen.main! 崩溃风险），说明 review 是不可或缺的质量保障。8.1/8.2 虽然变更较小，但也应执行 review。

5. **Phase 3 实施策略验证成功** — Epic 8 作为 Phase 3 的第一个 Epic，延续了 Phase 2 的高质量交付。增量扩展 + prompt 优先的策略在 Phase 3 同样有效。
