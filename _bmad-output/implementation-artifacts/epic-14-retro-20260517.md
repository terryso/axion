## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "前一个 Story 经验"节建立跨 Story 知识传递**
三个 Story 都包含"前一个 Story 经验"节：14.2 引用了 14.1 的 APIRunStatus 扩展和 StandardTaskOutput 统一返回模式；14.3 引用了 14.2 的 Cache-Control 模式和 buildTestApplication 辅助方法。这种知识传递避免了重复犯同样的错误。
**启示：** 跨 Story 经验传递已成为标准实践，每个 Story 都能在前一个的基础上更高效地实现。

**2. API Model 一致性模式稳定化**
所有新 API model 遵循统一模式：`Codable + Equatable + Sendable + ResponseEncodable`，snake_case CodingKeys，`EditedResponse(headers:response:)` 响应模式，`AxionAPIError(status:error:)` 错误模式。Epic 14 的三个 Story 无一偏离这个模式。
**启示：** API 层的模式一致性已达到"肌肉记忆"水平，降低了认知负担和 Review 成本。

**3. Review 发现数量逐 Story 递减**
14.1: 6 findings → 14.2: 4 findings → 14.3: 2+3 findings。Review 质量随 Story 推进而提升，特别是 14.2 和 14.3 的 findings 主要是测试覆盖和细节优化，而非核心逻辑问题。
**启示：** 连续 Story 开发中的经验积累直接反映在代码质量提升上。

**4. 向后兼容策略有效**
AxionBar 使用 `decodeIfPresent` 兼容 StandardTaskOutput 的新字段；StatusBarController 同时检查 "done" 和 "completed" 状态字符串。两个客户端都在零修改的情况下保持了功能正确性。
**启示：** `decodeIfPresent` + 双值检查的向后兼容模式在 Axion 项目中已被验证。

### 挑战模式（需要改进）

**1. File List 与实际变更不完全一致**
14.1 的 File List 引用了不存在的文件（RunHistoryWindowTests.swift）和遗漏了实际修改的文件（SkillModelsTests.swift、SSEEventClientTests.swift）。这个模式在之前的 Epic 中也出现过。
**根因：** `.done → .completed` 迁移影响范围广（跨 9 个文件），开发者在修改后未更新 File List。
**改进：** 大范围重命名/migration 操作后，应使用 `git diff --name-only` 验证 File List 完整性。

**2. 测试断言质量不足**
14.2 的路由测试仅检查 count（`#expect(statuses.count == 8)`）而非实际值；14.3 的 POST round-trip 测试没有有效断言。这些"形式测试"通过了但未验证实际行为。
**根因：** 测试编写时的思维模型是"验证存在性"而非"验证正确性"。
**改进：** Review 应优先检查测试断言的质量，特别是枚举值和内容比较。

**3. registerRoutes 参数持续膨胀**
Epic 14 为 registerRoutes 添加了 `maxConcurrent`（14.2）和 `configDirectory`（14.3）参数。加上之前的 `authKey`、`concurrencyLimiter`、`runLockService`，签名已有 7 个参数。
**根因：** 每个新功能都需要注入新的依赖，但 registerRoutes 是静态方法，无法使用依赖注入。
**改进：** 考虑将 registerRoutes 改为实例方法或引入 `APIContext` 配置结构体统一管理参数。

### 突破时刻

**1. StandardTaskOutput 统一契约消除了 API 响应碎片化**
Epic 5 建立的 API 使用 `CreateRunResponse` 和 `RunStatusResponse` 两种不同响应格式。14.1 将所有 `/v1/runs` 端点统一为 `StandardTaskOutput`，消除了集成方需要处理多种响应格式的负担。这是一次有意义的 API 债务清理。

**2. inferResultKind 启发式为中英双语场景设计**
`inferResultKind` 同时支持中文关键词（"读取/查询/获取"）和英文关键词（"read/query/get"），并使用合理的默认值（confirmation）。这种双语启发式对 Axion 的中文用户场景特别有价值。

**3. Settings API 的 env/config/missing 三级检测**
`resolveApiKeySource` 清晰地区分了 API Key 的三种来源，并正确处理了 env 覆盖 config 的优先级。POST 永远写入 config.json，GET 反映实际生效的来源 — 这种"写入路径"和"读取路径"的分离是好的设计。

## 关键经验

1. **API 契约升级是"隐形基建" — 影响广泛但不易感知** — Epic 14 最大的价值是让 Axion API 从"能工作"变成"集成友好"。StandardTaskOutput 统一了 4 个 API 端点的响应格式，Capabilities 让集成方可以动态发现能力，Settings API 让运维不需要 CLI。这些改进对外部集成的价值远大于对核心功能的价值 — 它是"让 Axion 从一个好工具变成一个好平台"的关键一步。

2. **向后兼容策略需要在设计时而非事后考虑** — 14.1 的 AxionBar 兼容（decodeIfPresent + 双值检查 "done"/"completed"）是在 Dev Notes 中提前规划的，而非 Review 时发现修复的。这种前瞻性的兼容策略在 API 重构中至关重要 — 事后修复的成本远高于设计时考虑。

3. **连续 Story 开发中的质量提升是可量化的** — Review findings 从 6→4→2 递减，且 14.2 和 14.3 的 findings 主要是测试细节而非核心逻辑。这证明了"跨 Story 经验传递"机制的有效性 — 每个 Story 的"前一个 Story 经验"节直接转化为更少的错误。

4. **registerRoutes 参数膨胀是架构债务的信号** — 从 Epic 5 的 4 个参数增长到 Epic 14 的 7 个参数，每次新增功能都添加参数。这种"温水煮青蛙"式的接口退化比突然的架构问题更难发现。教训：当一个函数的参数超过 5 个时，应该主动重构为配置结构体，而不是等到问题严重。

5. **Epic 14 证明了"API 先"的策略有效** — 三个 Story 按正确顺序实施：先建立输出契约（14.1），再添加能力发现（14.2），最后补充配置管理（14.3）。每个 Story 都在前一个的基础上构建，没有返工。这种增量式的 API 规范化策略比"一次性重写所有 API"风险低得多。
