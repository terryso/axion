## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. Actor 隔离模式的一致应用**
四个 Story 全部使用 actor 隔离有状态服务：RunLockService、VisualDeltaTracker、CostTracker、SeatActivityMonitor。每个 actor 都遵循相同的模式：私有可变状态 + 公开异步方法 + Sendable 返回类型。这使得并发安全成为默认行为而非需要额外注意的事项。
**启示：** Actor 隔离已成为项目的"肌肉记忆"，这是 Swift 并发模型的最佳实践。

**2. Dev Notes 质量达到新高度**
每个 Story 的 Dev Notes 包含：核心设计决策（D1-D6）、现有代码修改清单、"不修改"文件清单、关键反模式提醒、OpenClick 参考映射表、集成点详解、测试策略。特别是"Previous Story Intelligence"节使得 Story 间经验传递系统化。
**启示：** 详尽的 Dev Notes 是实现质量和 Review 效率的关键投入。

**3. "不修改已有文件"原则持续有效**
四个 Story 都明确列出了"不修改"文件清单。RunEngine、AxionCore 模型层、Helper 端代码保持不变。新功能通过新增文件实现（4 个 Service 文件），修改集中在 RunCommand（消息流集成）和 TraceRecorder（事件类型）。
**启示：** 这种约束策略在 4 个 Story 的增量开发中避免了级联修改。

**4. OpenClick 参考映射加速设计但保持 Axion 特色**
每个 Story 都包含 OpenClick 映射表，但 Axion 做了有意义的差异化：actor 替代 class、原生 AppKit API 替代子进程调用、JSON 格式替代纯文本、Swift Testing 替代 Jest。这些不是照搬而是"有根据的改编"。
**启示：** 竞品参考的价值在于验证设计方向，而非决定实现细节。

### 挑战模式（需要改进）

**1. AgentRunner 集成是系统性弱点**
四个 Story 中有 3 个的 Review 发现 AgentRunner 集成问题：13.1（无集成问题）、13.3（CostTelemetry 硬编码错误、screenshotCount 为 0）、13.4（externallyModified 死代码、缺少 trace）。AgentRunner 是 API 模式的执行路径，但开发焦点始终在 CLI 模式（RunCommand），API 路径的集成经常被遗漏或敷衍。
**根因：** RunCommand 是主要开发路径，AgentRunner 作为"镜像"路径未被同等重视。开发时的思维模型是"先做 RunCommand，AgentRunner 应该差不多"。
**改进：** 每个 Story 的 Task 列表中应将 AgentRunner 集成作为显式的高优先级 Task，而非事后补充。Review 时应优先检查 AgentRunner 路径。

**2. Trace 事件定义与记录脱节**
13.1 定义了 lockAcquired/lockReleased 但未记录；13.4 定义了 seat_baseline/external_activity_detected 但 AgentRunner 中未记录。这个模式重复出现：开发者先在 TraceRecorder 中添加事件类型（容易），然后在集成时忘记实际调用（容易遗漏）。
**根因：** "添加事件类型"和"调用记录方法"分属不同代码区域，Review 时需要跨文件检查。
**改进：** 在 TraceRecorder 便捷方法中添加注释标记预期的调用点，或使用 `@available(*, noasync)` 等标记确保便捷方法不被遗忘。

**3. CLIOverrides/ConfigManager 漏配**
13.3 新增了 CLI flag（--max-model-calls、--max-screenshots）和 AxionConfig 字段，但漏掉了 CLIOverrides 结构体和 ConfigManager 环境变量支持。这个"配置层完整性"问题在之前的 Epic 中也出现过。
**根因：** Axion 的配置层有三层：CLI flag → CLIOverrides → AxionConfig → ConfigManager（环境变量）。开发者通常只处理前两层，忘记第四层。
**改进：** 当添加新的 CLI flag 时，应有 checklist 确认三层都覆盖：CLI flag 定义、CLIOverrides 结构体、ConfigManager 环境变量。

**4. RunCommand 复杂度持续增长**
RunCommand.swift 在 Epic 13 中被每个 Story 修改，新增了：run lock acquire/release、visual delta tracker、cost tracker、seat activity monitor。这个文件已成为"上帝对象"，承载了过多的关注点。
**根因：** 所有新功能都集成到消息流循环中，而消息流循环在 RunCommand。
**改进：** 考虑将消息流处理提取为独立的 `MessageStreamProcessor` 类，RunCommand 只负责参数解析和初始化。这是 Epic 14 或 Epic 15 之前应评估的重构。

### 突破时刻

**1. NSEvent/NSWorkspace 原生 API 替代子进程调用**
Story 13.4 的 SeatActivityMonitor 直接使用 `NSEvent.mouseLocation` 和 `NSWorkspace.shared.frontmostApplication`，而 OpenClick 通过 `runCuaDriverCapture` 和 `Bun.spawn(["swift", "-e", script])` 子进程调用。Axion 的原生方案零延迟、零额外进程、零 AX 权限需求。这是 Swift 原生桌面应用相对于 Electron 方案的显著优势。

**2. 两阶段成本策略是务实的工程决策**
CostTracker 在运行时仅做计数检查（不依赖 token 数据），最终汇总使用 SDK 的精确 totalCostUsd。这避免了在消息流中阻塞等待 token 数据的问题，同时保证了最终成本的精确性。

**3. 亮度差异法（Luminance Delta）比 RGB 逐通道比较更鲁棒**
Story 13.2 选择亮度差异法而非 RGB 三通道分别比较，对 JPEG 重压缩噪声和微小颜色抖动更鲁棒。1% 阈值在 256x256 分辨率下对应约 655 个像素差异，平衡了灵敏度和误报率。

## 关键经验

1. **AgentRunner 不是 RunCommand 的镜像 — 它是一条独立路径** — Epic 13 最系统的教训是 AgentRunner 集成质量。三个 Story 的 Review 都发现 AgentRunner 路径存在问题：hardcoded 值、死代码、缺失 trace。根因是开发思维模型把 AgentRunner 当作 RunCommand 的"自动副本"，但实际上两者的消息流处理、生命周期管理、状态传递都有差异。教训：API 路径和 CLI 路径应被视为两条独立但同等重要的路径，每条路径的集成都需要显式验证。

2. **Actor 隔离模式已成为项目 DNA** — RunLockService、VisualDeltaTracker、CostTracker、SeatActivityMonitor 四个 actor 的实现一致性极高：私有可变状态 + Sendable 返回类型 + 便捷记录方法。这降低了认知负担，也让 Review 更高效。每个 actor 都遵循相同的模式，Review 可以聚焦于业务逻辑而非并发安全。

3. **Dev Notes 的"反模式提醒"是最有价值的部分** — 每个 Story 的 Dev Notes 都包含明确的"不要做 X"清单（如"不要在 Helper 端做活动检测"、"不要用文件级 hash"）。这些约束比"应该做什么"更能防止实现偏航。13.4 的 9 条反模式提醒覆盖了所有主要的实现陷阱。

4. **两阶段策略（运行时粗检 + 最终精算）是处理异步数据约束的好方法** — CostTracker 在消息流中仅做计数检查（因为 .assistant 消息不含 token 数据），最终汇总使用 SDK 的精确数据。这种"先保证运行时约束，后补精确数据"的策略，适用于所有"中间过程缺少精确数据但需要实时控制"的场景。

5. **Epic 13 的四层防护形成完整的运行时安全体系** — 运行锁（互斥执行）、视觉增量（成本优化）、预算控制（成本上限）、活动检测（学习保护）四个维度覆盖了桌面自动化最关键的安全需求。每一层都独立工作，不依赖其他层的实现。这种正交的安全层设计是好的架构实践。
