## Story 逐一分析

### Story 6.1: 通过 SDK AgentMCPServer 暴露 Axion

**亮点：**
- 28 个新单元测试，全部通过（844 total, 0 failures）
- AgentMCPServer 集成模式清晰：SDK 自动处理 MCP 协议层，应用层专注工具注册和任务编排
- RunTaskTool 异步任务提交设计合理：立即返回 runId，后台串行执行
- TaskQueue actor 串行化 agent.prompt() 调用，解决 Helper 单连接约束
- 复用 Epic 5 的 AgentRunner 配置逻辑（API key、Helper path、prompt、memory、safety hooks）

**挑战：**
- Review 发现 2 HIGH + 3 MEDIUM + 1 LOW 问题：
  - **H1:** TaskQueue 无 graceful shutdown — AC5 要求等待运行中任务完成后退出
  - **H2:** MCPServerRunner cleanup 直接 agent.close() 不等待 TaskQueue
  - **M1:** QueryTaskStatusTool 手动 JSON 拼接不转义特殊字符
  - **M2:** 错误响应缺少 suggestion 字段
  - **M3:** RunTaskTool 手动 JSON 拼接
  - **L1:** TaskQueue.swift 导入未使用的 AxionCore
- SDK 变更：Agent.assembleFullToolPool() 需从 internal 改为 public
- Swift 6 Sendable 检查：inputSchema `[String:Any]` 需要 nonisolated(unsafe) 标注

**决策记录：**
- Helper 工具 + 自定义业务工具合并暴露模式 — 外部 Agent 既可直接调用原子操作，也可提交完整任务
- TaskQueue actor 串行化 — 因为 AxionHelper 是单进程 stdio MCP server，同一时间只能服务一个客户端
- 日志输出到 stderr，stdout 仅用于 MCP JSON-RPC — 保证协议纯净

### 挑战模式（需要改进）

**1. JSON 手动拼接再次出现**
Story 6.1 的 M1/M3 问题与 Epic 5 Story 5.2 的 SSE 编码问题类似：
- 手动字符串拼接 JSON → 特殊字符未转义
- **根因：** 快速原型时倾向用字符串拼接，review 才发现不安全
- **改进：** 项目中应建立规则：所有 JSON 输出必须使用 JSONEncoder + Codable struct，禁止手动拼接

**2. Review 发现的 HIGH 问题聚焦在「边界条件」**
两个 Story 的 4 个 HIGH 问题都与边界条件有关：
- 6.1 H1/H2: graceful shutdown（清理时序）
- 6.2 H1: 文档与代码不一致
- 6.2 H2: 测试验证的是字符串而非真实代码
- **根因：** 开发时聚焦于 happy path，边界场景靠 review 补漏
- **改进：** 开发 Story 时主动列出边界条件检查清单

**3. SDK visibility 变更**
Agent.assembleFullToolPool() 从 internal 改为 public：
- 需要修改 SDK 源码以满足 Axion 需求
- **根因：** SDK API 设计未完全考虑跨模块使用场景
- **改进：** 在 Story 规划阶段提前评估 SDK API visibility

### 突破时刻

**1. SDK createSession() 测试模式**
发现 SDK 的 `createSession()` 方法可用于 in-process MCP 协议测试：
- 无需真实 stdio 进程
- 测试速度更快
- 可直接验证 initialize、tools/list、tool_call 协议行为
- 9 个协议测试用此模式完成

**2. 三类工具暴露模式**
Epic 6 确立了 Axion 对外暴露工具的三种模式：
- Helper 原子工具（直接调用 list_apps、click 等）
- run_task（异步任务提交，立即返回 runId）
- query_task_status（状态查询）
- 加上 SDK 自动注册的 agent_prompt（同步任务执行）
- 外部 Agent 可根据场景灵活选择使用模式

## 关键经验

1. **Review 持续证明其价值** — Epic 6 的 11 个问题全部在 review 中发现。自动修复全部成功。自动化 review 流程应作为项目标准保持。

2. **SDK 边界设计影响开发效率** — Agent.assembleFullToolPool() 的 visibility 变更暴露了 SDK API 设计与实际使用场景的差距。建议 SDK 与应用层同步规划 API surface。

3. **JSON 手动拼接是反复出现的反模式** — 尽管在 Epic 5 中已发现，Epic 6 仍有同类问题。需要将「禁止手动 JSON 拼接」作为显式的代码规范。

4. **三层工具暴露模式确立** — 原子操作 + 异步任务 + 同步任务的分层为外部 Agent 提供了灵活的使用模式。这一架构决策应在后续 Epic 中保持。

5. **异步边界条件仍需更严格的开发纪律** — Epic 5 建议画异步流程图，但未严格执行导致 Epic 6 仍有 shutdown 时序问题。建议在涉及 actor + async 的 Story 中强制要求异步流程图。
