## Story 逐一分析

### Story 12.1: Memory Fact 模型升级

**亮点：**
- 46 个新测试，覆盖模型、生命周期、持久化和提取器
- AppMemoryFact 模型设计清晰：3 个枚举（Status/Source/Kind）+ struct，遵循 Codable+Equatable+Sendable
- MemoryFactStore 使用 actor 隔离，线程安全
- 惰性迁移策略正确：读旧 KnowledgeEntry 时自动升级为 AppMemoryFact，无强制迁移
- `factId()` 使用 djb2 确定性 hash（而非 Swift 的随机 hashValue），保证跨进程稳定

**挑战：**
- **CRITICAL: `factId()` 初始使用 `hashValue`** — Swift 的 hashValue 每次进程启动随机化，导致同一事实在不同运行中产生不同 ID。Review 捕获后替换为 djb2 确定性 hash。
- **HIGH: `mergeFact` 未更新 `updatedAt`** — 合并后 updatedAt 字段不正确，导致 30 天降级计算错误。
- **HIGH: 缺少 AC4（矛盾事实处理）测试** — 核心验收标准未被测试覆盖。
- 浮点精度陷阱（0.7 + 0.1 ≠ 0.8 in Double）在测试中暴露

**决策记录：**
- D1: AppMemoryFact 是 AxionCLI 层模型，不修改 SDK KnowledgeEntry
- D3: 生命周期状态机 candidate→active→retired→candidate（闭环）
- D4: 确定性 ID 使用 djb2 hash

### Story 12.2: 三类记忆分类

**亮点：**
- 自动化 affordance 检测：成功任务中直接操作占比高 → affordance（confidence=0.72）
- MemoryContextProvider 双模式：新方法 `buildFactMemoryContext()` 优先，旧方法 fallback
- Prompt 注入格式清晰：三类分 section 渲染，每类最多 5 条按 confidence 降序
- 软性建议声明："These are soft hints, not hard rules"
- MemoryListCommand 改用 MemoryFactStore，展示状态图标和分类标签

**挑战：**
- **HIGH: assembleFactContext 多行描述渲染缩进错误** — 长描述换行时未缩进 continuation lines
- **MEDIUM: affordance description 生成不够精炼** — 初始实现缺少简洁的 affordance 摘要行
- RunCommand 中存在 3 个 MemoryFactStore 实例（不同作用域），代码不够优雅但功能正确
- classifyKind 方法可以是 static func 但未标记

**决策记录：**
- D1: affordance 检测基于工具序列分析（直接操作 vs 探索操作占比），非 Takeover 触发
- D2: MemoryContextProvider 双模式，新旧方法并存
- D3: Prompt 注入格式参考 OpenClick renderRelevantMemoriesForPrompt

### Story 12.3: Memory 导入/导出

**亮点：**
- MemoryBundle 数据结构简洁：schema_version + exported_at + memories 数组
- 导入降级策略合理：source=imported, status=candidate, confidence 封顶 0.55
- 合并逻辑完备：max confidence、stronger status、local source 优先、evidenceCount +1
- CLI 命令 `axion memory export/import` 参数设计清晰
- 33 个测试覆盖 round-trip、降级、合并、错误处理、边界情况

**挑战：**
- **HIGH: 空 memories bundle 导入报错** — 有效 JSON 但 `memories: []` 导入失败，破坏 round-trip
- **MEDIUM: 无 schema_version 值验证** — 未来格式（v2+）会被静默接受为 v1
- **MEDIUM: 缺少 per-domain error isolation 测试** — AC5 声明逐 domain 处理但未测试
- AC 中 confidence 降为 0.4 与实际使用 0.55 不一致（最终采用 0.55 与 OpenClick 一致）

**决策记录：**
- D2: confidence 封顶 0.55（而非 AC 中的 0.4），与 OpenClick 一致
- D4: 不修改已有模型，AppMemoryFact 已有完整 Codable 支持

## 关键经验

1. **"不修改已有模型"是有约束的创新的基石** — Epic 12 最成功的设计决策是在每个 Story 中明确列出"不修改"文件清单。这使得每个 Story 的修改范围最小化，避免了级联修改和回归风险。三个 Story 的"不修改"清单被严格遵循，所有新功能通过新增文件实现。教训：在增量开发中，"不修改什么"比"修改什么"更重要。

2. **确定性是 ID 生成的最基本要求** — Story 12.1 的 `factId()` 初始使用 Swift `hashValue`，这个值每次进程启动都不同，导致同一事实在不同运行中产生不同 ID。Review 捕获了这个 CRITICAL 问题并修复为 djb2 hash。教训：任何用作标识符的 hash 值必须跨进程/跨运行稳定，这是不变量级别的需求，应在设计阶段就明确。

3. **竞品参考映射让设计决策有据可依** — 三个 Story 都包含详细的 OpenClick 参考映射表，标注了每个 Axion 组件对应的 OpenClick 函数和关键差异。这种"参考但不照搬"的方法让团队能快速做出有根据的设计决策（如 confidence 封顶 0.55 而非 0.4），而非凭空猜测。

4. **Review 是核心算法正确性的最后防线** — Epic 12 的 17 个 Review 问题中，3 个是 CRITICAL/HIGH 级别的核心算法问题（hashValue 随机化、mergeFact updatedAt、空 bundle 导入）。这些问题都在"正常路径"下不触发，只在边界条件下暴露。教训：核心算法的不变量（ID 稳定性、字段联动、空数据处理）应该在开发阶段通过专门的测试验证，而非依赖 Review 发现。

5. **Epic 12 为后续 Epic 奠定了 Memory 基础** — Epic 15（Takeover 学习）依赖 12.2 的 affordance/avoid 分类，Epic 13 的 Story 13.4（桌面活动检测）依赖 Memory 提取的禁用机制。Epic 12 的"不修改已有模型"策略确保了 Memory 系统的接口稳定性，为后续 Epic 提供了可靠的基础。
