## Story 逐一分析

### Story 18.1: 内置桌面技能

**亮点：**
- **AxionBuiltInSkills 遵循 SDK BuiltInSkills 同构模式** — caseless enum 命名空间 + static var 返回 Skill 实例，与 SDK 的 BuiltInSkills 模式完全一致，降低了学习成本
- **registerAll() 便捷方法消除 DRY 违反** — Review 发现 RunCommand 和 SkillListCommand 中重复注册三个技能，提取 `AxionBuiltInSkills.registerAll(into:)` 一行调用解决问题
- **内置技能不受文件系统约束** — 纯代码定义，无 SKILL.md 文件依赖，Axion 启动即可用
- **13 个单元测试覆盖全面** — 属性验证、registry 查找、alias 解析、promptTemplate 非空、argumentHint 非空

**挑战：**
- **[MEDIUM] 注册顺序与 Story 意图矛盾** — Story 声明"文件系统技能同名可覆盖内置技能"，但实现中内置技能注册在文件系统之后，导致内置覆盖文件系统。修复：交换顺序，内置先注册，文件系统后注册可覆盖
- **[MEDIUM] DRY 违反** — 三次 register() 调用在 RunCommand 和 SkillListCommand 中重复。修复：提取 `registerAll(into:)` 便捷方法
- **[LOW] 缺少 `screen` alias 测试** — screenshot-analyze 的 `screen` alias 未覆盖。修复：新增测试
- **[LOW] argumentHint 未测试** — 测试验证了 promptTemplate 内容但遗漏了 argumentHint。修复：新增 testArgumentHintNonEmpty()

**关键设计决策：**
- `toolRestrictions = nil` — MCP 工具不在 SDK ToolRestriction 枚举内，通过 promptTemplate 软限制
- `isAvailable = { true }` — 内置桌面技能是 Axion 核心功能
- 受 `--no-skills` 控制 — 尊重用户显式禁用意图

### Story 18.2: 技能 + Memory 联动

**亮点：**
- **scope 字段复用现有模型** — `AppMemoryFact.scope` 在 Story 12.1 已预留，本 Story 直接使用 `"skill:{skillName}"` 格式，零模型变更
- **三类 Memory 优先级排序** — affordance → avoid → observation，确保正向经验优先注入
- **录制技能也记录 Memory** — 18.2 不只为 prompt 技能，也为录制技能（RecordedSkillRunner）添加了 Memory 记录，统一了两种技能类型的经验沉淀
- **13 个新测试** — 覆盖 scope 过滤、kind 优先级、数量限制、空返回、noMemory 行为

**挑战：**
- **[HIGH] `--no-memory` 标志未在 scope 标记时检查** — `noMemory=true` 时 facts 仍被 scope-tagged 并保存。修复：添加 `!noMemory` guard
- **[HIGH] RecordedSkillRunner 只在成功时记录 Memory** — Dev Notes 4.2 明确要求失败时记录 avoid kind，但实现遗漏了错误分支。修复：添加 avoid fact 记录
- **[HIGH] Tasks 5.5/5.6 测试标记完成但实际未覆盖** — 声称测试了 RunCommand 集成和 `--no-memory` 行为，但测试文件缺少对应测试。修复：新增 noMemory scope-tagging 测试
- **[MEDIUM] 重复的 MemoryContextProvider/MemoryFactStore 实例** — 在 RunCommand 不同位置创建了重复实例。修复：重构为外层作用域复用

**Review 严重度分布：** 3 HIGH 是 Epic 18 中最严重的 Review 问题集中。三个 HIGH 的共同根因是"部分路径遗漏"——`--no-memory` 标志在 scope 标记路径未检查、失败路径的 Memory 记录遗漏、声称已测试但实际未覆盖。

### Story 18.3: HTTP API 支持 Skill 触发

**亮点：**
- **双路径分发架构清晰** — SkillRegistry 查找（prompt 技能）→ AgentRunner.runSkillAgent()，未命中则查 JSON 文件（录制技能）→ SkillAPIRunner，完全与 CLI 双轨查找一致
- **runSkillAgent 独立于 runAgent** — 不修改现有 runAgent()，新增方法避免回归风险
- **Memory 注入与 CLI 行为一致** — App 级 + skill-scoped Memory 完整复用
- **AxionBar 向后兼容** — type 字段使用 `decodeIfPresent`，现有客户端不受影响
- **SkillRegistry 在 daemon 启动时创建** — 明确的设计决策和权衡（技能变更需重启 server）

**挑战：**
- **[HIGH] runSkillAgent 使用空 HookRegistry，跳过安全检查** — shared seat 模式下 prompt 技能可执行前台工具（click/type），绕过安全限制。修复：改为 `buildSafetyHookRegistry()`
- **[HIGH] Prompt 技能运行绕过 RunLock 桌面级排他锁** — 并发 prompt 技能运行可能导致桌面操作冲突。修复：添加 runLockService.acquire/release
- **[MEDIUM] Prompt 技能运行绕过 ConcurrencyLimiter** — 无限制并发可能耗尽系统资源。修复：添加 concurrencyLimiter.tryAcquire() 检查
- **[LOW] 空 task 字段未验证** — 发送 `{"task": ""}` 导致 agent 收到空任务描述。修复：添加空字符串检查
- **[LOW, 未修复] runSkillAgent 缺少 TraceRecorder 和 SeatActivityMonitor** — 与 runAgent() 不一致，不影响功能正确性

**关键洞察：** 18.3 的两个 HIGH 和一个 MEDIUM 都指向同一个问题——新增执行路径（runSkillAgent）时未完全复制现有路径（runAgent）的安全/并发保障。这是一个系统性的模式问题。

## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "同构模式"降低学习成本**
AxionBuiltInSkills 完全遵循 SDK BuiltInSkills 的 caseless enum 命名空间模式。开发者看到 Axion 端代码时，可以立即关联到 SDK 端模式，无需学习新概念。
**启示：** 当 SDK 提供了可参考的设计模式时，集成端应尽量同构，而非创造新模式。

**2. "scope 复用"零模型变更**
技能 Memory 联动完全复用了 Story 12.1 预留的 `scope` 字段，没有引入任何模型变更。scope 值从 `"window-title:X"` 自然扩展到 `"skill:screenshot-analyze"`，体现了预留字段设计的远见。
**启示：** 通用可选字段（如 scope）比硬编码枚举更有弹性——不同子系统可以通过 scope 值约定各自使用模式，无需修改模型。

**3. "双路径分发"保持 API 一致性**
18.3 的 API 双路径分发与 17.2 的 CLI 双轨查找完全一致（prompt 优先 → recorded 回退 → 404），用户无论通过 CLI 还是 API 都得到一致的查找行为。
**启示：** 当同一功能需要多个入口（CLI、API、UI）时，核心查找逻辑必须统一，入口只做路由。

**4. "独立性原则"避免回归**
runSkillAgent() 独立于 runAgent()、PromptSkillRunRequest 独立于 SkillRunRequest、新增方法而非在现有方法中添加分支——每个 Story 都遵循了"新增而非修改"的原则。
**启示：** 在已有稳定逻辑上添加新能力时，独立路径 > 修改现有路径。即使有一些代码重复，也比引入回归风险安全。

### 挑战模式（需要改进）

**1. "新增执行路径未复制安全上下文"是系统性问题**
18.3 的 3 个 HIGH/MEDIUM issues（HookRegistry、RunLock、ConcurrencyLimiter）都源于同一个根因：新增 runSkillAgent() 路径时，只关注了功能实现，未复制 runAgent() 的安全保障。
类似问题在 18.2 也出现——新增 Memory 记录路径时，未在所有分支（成功/失败、noMemory/正常）一致实现。
**根因：** Dev Notes 和 Story 规格聚焦于功能 AC，缺少安全/并发/边界条件的检查清单。
**改进：** Story 规格中新增"安全保障继承"检查项——当新增执行路径时，必须列出参考路径的所有安全保障（RunLock、HookRegistry、ConcurrencyLimiter、Memory 标志）并逐一确认是否需要复制。

**2. "声称已完成但测试缺失"再次出现**
18.2 的 Tasks 5.5/5.6 被标记 `[x]` 但对应测试不存在。这与 Epic 17 中测试计数偏差是同类问题。
**根因：** 开发者在实现过程中基于预期标记任务完成，但最终输出与标记不一致。
**改进：** Completion Notes 应使用 `grep -c "@Test" Tests/...` 统计实际测试数量（Epic 17 行动项 3，仍未完全执行）。

**3. RunCommand 继续膨胀**
RunCommand.swift 在 Epic 18 中新增了：内置技能注册调用、显式技能 Memory 注入、技能 scope 标记、Memory provider/store 实例创建等逻辑。Epic 17 回顾标记的"RunCommand 提取 Skill 相关逻辑到独立服务层"高优先级技术债务再次未处理。
**根因：** 连续 6 个 Epic 都标记了 RunCommand 重构，但从未执行。
**改进：** 当技术债务连续 3 个 Epic 被标记但未执行时，应视为阻塞项（blocker），下一个 Epic 必须先处理。

**4. 18.2 Review 严重度较高（3 HIGH）**
Epic 18 的 Review findings 总数为 13 个（18.1: 4, 18.2: 4, 18.3: 5），其中 18.2 贡献了 3 个 HIGH。这是自 Epic 13 以来单 Story HIGH findings 最多的一次。
**根因：** 18.2 涉及修改多条执行路径（显式技能 Memory 注入、Memory 提取 scope 标记、RecordedSkillRunner Memory 记录），每条路径都有独立的边界条件，容易遗漏。
**改进：** 当 Story 涉及修改 N 条独立执行路径时，Dev Notes 应为每条路径列出独立的 AC/边界条件检查清单。

### 突破时刻

**1. ToolRestriction = nil 的设计决策**
Story 18.1 明确了"内置桌面技能不设置 toolRestrictions"的设计决策。这看似简单，实则解决了 SDK ToolRestriction 枚举只包含 SDK 内置工具（bash、read 等）、不包含 MCP 工具的根本矛盾。通过 promptTemplate 的"软限制"等效于硬限制，同时保持了对 MCP 工具生态的开放性。

**2. runSkillAgent 的"安全修复模式"**
18.3 Review 发现的安全问题（HookRegistry、RunLock、ConcurrencyLimiter）虽然严重，但修复方式干净——新增路径补充安全保障而非修改现有路径。这验证了"独立路径"设计原则的正确性：如果 runSkillAgent 被实现为 runAgent 的分支，安全问题会更难隔离和修复。

## 关键经验

1. **新增执行路径的安全保障继承是必需品** — Epic 18 最严重的 Review findings 全部来自"新增路径未复制安全上下文"（18.2: 3 HIGH, 18.3: 2 HIGH）。这不是个别遗漏，而是系统性问题。教训：当新增 Agent 执行路径时，安全保障（RunLock、HookRegistry、ConcurrencyLimiter、Memory 标志）不是可选的——它们是执行路径的一部分，必须与功能 AC 同等对待。

2. **ToolRestriction = nil 的"软限制"策略** — 内置桌面技能通过 promptTemplate 指导 LLM 使用哪些工具，而不是通过 SDK ToolRestriction 硬限制。这个决策的正确性在于：MCP 工具生态是开放的，SDK 枚举无法覆盖所有 MCP 工具名。教训：当集成层的工具集与 SDK 枚举不匹配时，prompt 级软限制比扩展 SDK 枚举更灵活、更低耦合。

3. **scope 字段的弹性设计** — `AppMemoryFact.scope` 从最初的 `"window-title:X"` 自然扩展到 `"skill:screenshot-analyze"`，无需任何模型变更。教训：通用可选字段 + 约定式值格式 > 硬编码枚举。当设计模型时，预留通用字段比预见所有用途更实际。
