## Lessons Learned

### L1: Event Loop Cleanup Is a Standard Requirement for AxionRuntime Consumers

Story 26.1 discovered (via HIGH review finding) that `runtime.execute()` throwing causes event loop resource leaks if not wrapped in do/catch. Story 26.2 correctly applied the pattern from the start. This should be documented as a mandatory pattern for all AxionRuntime consumers.

**Key Insight:** Any code that calls `runtime.startEventLoop()` MUST wrap `runtime.execute()` in do/catch and call `runtime.stopEventLoop()` in both success and error paths.

### L2: Dead Code Cleanup Requires Grep-Level Verification, Not Just Reading

Epic 20's lesson ("pre-deletion audit is the most valuable artifact") was correctly applied in Story 26.3. Every symbol was grep-verified before removal. The one thing that caught us was state variables that become always-false after code removal — this isn't visible from grep alone, it requires understanding the data flow.

**Key Insight:** After removing code that SETS a variable, check whether the variable's READERS still behave correctly with the default value.

### L3: Task Status Must Reflect Actual Implementation, Not Intent

Story 26.2 marked tasks as complete when the dev understood why they shouldn't be done, rather than reflecting actual code state. This created misleading story records that the code review had to correct.

**Key Insight:** Task checkboxes are implementation status, not rationale records. Use `[ ]` with a DEFERRED note for intentionally skipped work.

### L4: Pre-Existing Migration Simplifies Follow-Up Stories

Story 26.1 was almost entirely test-writing because Epic 25 had already migrated RunCommand. This is a pattern: when the original integration is thorough, follow-up stories that add tests or verify behavior are straightforward. The inverse is also true — if the original integration is incomplete, follow-up stories compound the complexity.

**Key Insight:** Doing the full migration in Epic 25 (including RunCommand) made Epic 26's CLI path trivially simple, freeing capacity for the harder API path and cleanup.
