## Story 逐一分析

### Story 7.1: 基于 SDK Pause Protocol 的用户接管机制

**亮点：**
- 26 个新单元测试，全部通过（889 total, 0 failures）
- TakeoverIO 模块设计干净：TakeoverAction 枚举 + TakeoverIO 类，职责分明
- SDK Pause Protocol 集成一次成功：只需注册 `createPauseForHumanTool()` + 监听 `.paused`/`.pausedTimeout` 消息
- JSON 模式下 TakeoverIO 写 stderr（stdout 保持纯净），正确处理双模式输出
- Trace 完整记录 takeover 生命周期：paused → resumed/aborted/timeout
- 前台模式集成零额外代码（SafetyChecker hook 自然覆盖）

**挑战：**
- Review 发现 2 HIGH + 3 MEDIUM + 2 LOW 问题：
  - **H1:** TakeoverIntegrationTests 两个同义反复测试（tautology tests）— 测试没有实际意义
  - **H2:** AC4 缺少 abort 步骤摘要 — 用户 abort 时不知道完成了多少步
  - **M1:** SDKJSONOutputHandler.pausedTimeout JSON 缺少 reason 字段
  - **M2:** JSON paused 事件缺少 sessionId
  - **M3:** SDKJSONOutputHandler 可选值隐式转换编译警告
  - **L1:** takeover_aborted trace 事件缺少 completedSteps payload
  - **L2:** 需要新增 3 个测试覆盖 abort 摘要和 JSON 新字段
- SDK Pause Protocol 文档需要仔细阅读（如 setPauseHandler 不应外部调用等边界条件）

**决策记录：**
- TakeoverIO 输出策略：Terminal 模式用 stdout，JSON 模式用 stderr + SDKJSONOutputHandler
- stdin 阻塞读取是预期行为 — readLine() 同步阻塞等待用户输入
- 超时由 SDK 管理，Axion 只需展示提示
- abort 时显示已完成步骤数 — 用户需要上下文来决策

### Story 7.2: `--fast` 模式

**亮点：**
- 24 个新测试（18 初始 + 6 review 补充），全部通过（916 total, 0 failures）
- 所有变更集中在 RunCommand.swift — 零新模块，最小化变更面
- `buildFullSystemPrompt()` fast 分支设计合理：fast prompt 在 dryrun 之前检查（fast 优先）
- 提取 `computeEffectiveMaxSteps()`、`computeEffectiveMaxTokens()`、`traceMode()` 为 internal static 方法，提升可测试性
- Fast mode prompt 设计精炼：4 条指令引导 LLM 生成最小计划

**挑战：**
- Review 发现 3 HIGH + 2 MEDIUM 问题：
  - **H1:** test_buildFullSystemPrompt_fastMode_includesFastInstructions 是虚假测试 — 测试硬编码字符串，不是真实方法
  - **H2:** test_buildFullSystemPrompt_fastMode_beforeDryrun 是虚假测试 — 测试本地拼接，不是实际代码
  - **H3:** test_fastMode_maxTurns_cappedAt5 只检查 flag，不验证 min(5) 逻辑
  - **M1:** runMode 变量缺少 dryrun case（与 trace 不一致）
  - **M2:** effectiveMaxTokens (2048) 无测试覆盖
- Completion Notes 中测试总数（939）与 review 中（916）不一致 — 文档准确性需注意

**决策记录：**
- Fast mode 不绕过 SDK — 在现有 Agent Loop 框架内调整参数（与 OpenClick 的 runTaskFast 不同）
- maxTurns = min(userSetting, 5)，maxTokens = 2048
- 失败不重规划，直接报告失败并建议去掉 `--fast`
- `--fast --dryrun` 组合时 fast 优先（prompt 追加顺序：fast → dryrun）
- `--fast --allow-foreground` 两个标志独立，不交互

**1. 虚假测试（Bogus/Tautology Tests）持续出现**
这是 Epic 7 最突出的问题。两个 Story 的 5 个 HIGH 问题中有 4 个与测试质量有关：
- Story 7.1 H1: 两个同义反复测试
- Story 7.2 H1/H2: 测试硬编码字符串而非真实方法
- Story 7.2 H3: 测试只检查 flag 不验证逻辑
- **根因：** 开发时倾向于先写"看起来通过"的测试，而不是测试真实行为
- **改进：** 建立「测试必须调用被测方法/函数，不允许测试纯字面量」的规则

**2. 完成记录中的数据不一致**
Story 7.2 Completion Notes 说 939 total tests，但实际运行结果和 review 部分都显示 916：
- **根因：** Completion Notes 在 review 之前写，review 修改了测试数量但未更新 Notes
- **改进：** Review 修改测试后应同步更新 Completion Notes 中的数字

**3. Review 发现的 HIGH 问题全部与测试质量有关**
两个 Story 的 5 个 HIGH 问题无一是生产代码 bug：
- **正面看：** 生产代码质量高，功能实现正确
- **反面看：** 测试质量成为薄弱环节，虚假测试给人虚假的安全感
- **改进：** 测试本身也需要质量门控 — 测试应验证真实行为而非通过字符串匹配

### 突破时刻

**1. SDK Pause Protocol 三步集成模式**
发现 SDK Pause Protocol 只需 3 步即可集成：
1. 注册 `createPauseForHumanTool()` 到工具池
2. 在 stream 循环中监听 `.paused`/`.pausedTimeout` 消息
3. 读取 stdin → 调用 `agent.resume()`/`agent.interrupt()`
- 这一模式比预期简单得多，SDK 抽象层设计优秀

**2. Fast Mode 参数化设计**
通过提取 `computeEffectiveMaxSteps()`、`computeEffectiveMaxTokens()`、`traceMode()` 为独立 static 方法：
- 提升了可测试性（不需要构建完整 RunCommand 即可测试逻辑）
- 代码意图更清晰（方法名即文档）
- Review 过程中这一重构帮助发现了 runMode 缺少 dryrun case 的 bug

## 关键经验

1. **虚假测试是最大的质量隐患** — Epic 7 的 5 个 HIGH 问题中有 4 个是测试质量问题。虚假测试通过但测试了错误的东西，给人虚假的安全感。「测试必须调用真实方法」应成为显式规则。

2. **SDK 集成模式已成熟** — Pause Protocol 的集成仅需 3 步，SDK 抽象层设计优秀。Epic 5/6/7 的 SDK 集成一次比一次顺畅，说明 SDK 的 API 设计和 Axion 的集成模式已经成熟。

3. **Dev Notes 中的 SDK API 文档价值巨大** — Story 7.1 在 Dev Notes 中详细记录了 SDK Pause Protocol 的全部 API（包括 setPauseHandler 陷阱），帮助避免了常见的集成错误。这一做法应成为 SDK 集成类 Story 的标准。

4. **Fast mode 的「参数化设计」策略值得推广** — 将 computeEffectiveMaxSteps()、computeEffectiveMaxTokens() 等逻辑提取为独立 static 方法，既提升可测试性又提升代码可读性。当逻辑分支超过 2 层时应考虑提取。

5. **Phase 2 实施顺序证明有效** — Epic 5（无 SDK 依赖）→ Epic 6（SDK 新能力）→ Epic 7（SDK 新能力）的顺序让团队逐步积累 SDK 集成经验，每个 Epic 的 SDK 集成都比上一个更顺畅。
