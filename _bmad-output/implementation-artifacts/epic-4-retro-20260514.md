## Story 逐一分析

### Story 4.1: 集成 SDK MemoryStore 与 App Memory 提取

**亮点：**
- 干净的 SDK 集成 — FileBasedMemoryStore 通过 `memoryDir` 自定义路径 `~/.axion/memory/`

- DoctorCommand 新增 Memory 状态检查，用户可见性良好

**挑战：**

- 中文/英文标签混用导致测试失败：content 使用 "成功/失败" 但测试断言 "success/failure"

**Review 发现：** 1 HIGH, 4 MEDIUM, 2 LOW（全部已修复）
- HIGH: XCTAssertNotNil 对非可选类型 `[KnowledgeEntry]` 永远不会失败
- MEDIUM: groupByAppDomain 静默丢弃工具（已修复为 attach 到第一个 domain）
- MEDIUM: extract() 中 ~80 行重复代码（已提取 buildEntry() helper）

### Story 4.2: App Profile 自动积累

**亮点：**
- AppProfileAnalyzer 设计为纯计算 struct（无副作用），测试友好
- 滑动窗口模式识别算法（2-4 步子序列，frequency >= 2）简洁有效
- AX tree JSON 解析从 `get_window_state` / `get_ax_tree` 中提取角色和标题控件
- 失败经验标记 + 修正路径推断（失败后有同类成功操作时自动关联）

**挑战：**

- 参数化工具名破坏模式匹配（如 `click(x:150,y:200)` vs `click`）
- Review 发现 totalRuns 错误统计了 profile 和 familiar 元数据条目

**Review 发现：** 4 Fixed, 2 Deferred + 3 Auto-Review patches
- Fixed: totalRuns 过滤、参数化工具名剥离、冗余失败标记、profile 条目累积
- Deferred: 关键词匹配的失败原因分类（脆弱但功能正常）、AX tree 遍历的无界递归（防御性问题）

## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. 纯计算服务模式**
AppProfileAnalyzer 和 MemoryContextProvider 都设计为纯计算 struct：
- AppProfileAnalyzer: 输入历史 KnowledgeEntry + 本次结果，输出 AppProfile
- MemoryContextProvider: 输入 task 描述 + MemoryStore，输出 prompt 文本
- **启示：** 将计算逻辑与 I/O/存储分离，测试时只需构造输入数据，无需 mock 文件系统

**2. 安全降级一致性**
三个 Story 统一遵循「Memory 失败不阻塞任务」原则：
- Memory 提取失败 → warning 日志，任务正常完成
- Memory 保存失败 → warning 日志，任务正常完成
- Memory 上下文获取失败 → 返回 nil，等同于无 Memory 模式
- **启示：** Memory 是增强功能，不是核心路径 — 这个定位在设计阶段就已明确并在实现中严格遵守

**3. Story 间增量构建**
每个 Story 精确地在前一个基础上构建：
- 4.1 → MemoryStore 集成 + AppMemoryExtractor + 基础 content 格式
- 4.2 → AppProfileAnalyzer + AX tree 提取 + 失败标记 + 熟悉度追踪（增强 content 格式）
- 4.3 → MemoryContextProvider + system prompt 注入 + CLI 管理命令
- **启示：** Story 拆分粒度恰当，依赖链清晰，每个 Story 都有独立可测试的交付物

**4. Review 安全审查有效**
Story 4.3 的 path traversal 漏洞在 review 中被捕获：
- domain 参数未验证，`../` 可突破 memory 目录
- 修复：添加 `isValidDomain()` 验证，拒绝 `/`、`\`、`..`
- **启示：** 涉及文件路径的用户输入必须经过验证 — 这是 review 流程的关键价值

### 挑战模式（需要改进）

**1. Force Unwrap 反复出现**
三个 Story 中都发现了 force unwrap 问题：
- 4.3: `latestProfile!` → guard let
- 4.3: `lastUsed!` → if let
- **根因：** 开发阶段为了快速验证使用 force unwrap，未在提交前清理
- **改进：** 开发完成后全局搜索 `!` 和 `as!`，强制替换为安全解包

**2. SDK API 参数顺序陷阱**
AgentOptions init 的参数顺序是隐式约束：
- Story 4.1 Debug Log: "memoryStore must precede hookRegistry"
- Story 4.2 Dev Notes: 再次提醒这个顺序
- **根因：** SDK 使用 positional parameters 而非 labeled parameters
- **改进：** 在 project-context.md 中标注此约束（已完成）

**3. 中英文标签混用**
Story 4.1 的 content 格式使用中文标签（"任务:"、"结果:"），但 success/failure 用英文：
- 导致测试失败：测试断言 "success" 但 content 包含 "成功"
- **根因：** content 格式设计时未统一语言
- **改进：** content 中的状态标签统一使用英文（success/failure），描述文本使用中文

**4. Profile 文本格式耦合**
MemoryContextProvider 直接解析 RunCommand.buildProfileContent() 生成的文本格式：
- 格式变化会导致解析静默失败（返回 nil）
- **根因：** 缺少共享的格式常量或解析器
- **改进：** 应提取 ProfileContentBuilder 为独立类型，provider 和 builder 共享格式定义

### 突破时刻

**1. 滑动窗口模式识别**
Story 4.2 的 AppProfileAnalyzer 使用滑动窗口提取 2-4 步子序列作为操作模式：
- 简洁高效，无需复杂的序列对齐算法
- frequency >= 2 的阈值在当前数据规模下足够
- 如果未来 Memory 数据量增大，可能需要升级为更智能的模式识别

**2. FamiliarityTracker 的简洁阈值**
3 次成功即标记为 "familiar"：
- 逻辑简单清晰，易于理解和测试
- 紧凑规划策略通过 system prompt 指令自然生效，不需要修改 Planner 代码

## 关键经验

1. **纯计算服务是可测试性的基石** — AppProfileAnalyzer 和 MemoryContextProvider 作为纯计算 struct，测试时无需 mock 任何外部依赖。这个模式应在所有新服务设计中推广。

2. **安全降级是增强功能的正确姿态** — Memory 是「锦上添花」而非「必不可少」，三个 Story 一致地将 Memory 操作放在 do/catch 中确保不阻塞核心路径。这个原则应适用于所有非核心能力。

3. **Review 再次证明是安全网** — Story 4.3 的 path traversal 漏洞是经典的「用户输入构造文件路径」问题。Review 流程有效捕获了此类安全缺陷。所有涉及文件路径操作的新代码都应在 review 中重点审查。

4. **SDK 参数顺序是隐性知识** — AgentOptions init 的参数顺序约束没有编译器提示，只能通过运行时错误发现。这类隐性知识应记录在 project-context.md 中（已完成）。

5. **文本格式耦合是隐性债务** — MemoryContextProvider 解析 RunCommand.buildProfileContent() 的文本输出，两者通过隐式约定而非共享接口耦合。这类耦合在短期内可行，但长期应提取为共享类型。
