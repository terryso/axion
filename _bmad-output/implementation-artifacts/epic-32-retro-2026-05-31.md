## What Went Well

### W1: Story Dependency Chain Was Well-Ordered

The story sequence 32.1 → 32.2 → 32.3 → 32.4 → 32.5 had a clean dependency chain. Each story's deliverables were consumed by subsequent stories without rework:
- 32.1's `TGAPIError` 4-case enum, `TGParseMode`, `editMessageText`, and `TGMessageFormatter` were all reused by 32.2
- 32.2's `TGStreamingController` was extended by 32.3's typing timer
- 32.3's config wiring pattern was followed by 32.4 and 32.5
- 32.4's registry pattern was the final piece before 32.5's interactive callbacks

### W2: Triple-Fallback Rendering Is Production-Grade

The MarkdownV2 → HTML → PlainText fallback chain (Story 32.1) proved robust across all subsequent stories. Every story that sends formatted messages benefits from this safety net. The `TGMessageFormatter.split()` operated on rendered length (not raw text), preventing the common Telegram bug where escaped characters inflate beyond the 4096-char limit.

### W3: Draft Preview Spike Gate Saved Wasted Effort

Story 32.3 correctly executed the spike-first approach. The technical research conclusively proved that Telegram Bot API has no `saveDraft` endpoint — drafts are client-side via MTProto. This avoided 2-3 tasks of dead-end implementation. The spike report (`TGDraftSpikeReport.md`) is a clear decision artifact for future reference.

### W4: TGStreamingController Actor Isolation Is Solid

The `TGStreamingController` actor design (Story 32.2) handled all concurrent state correctly: edit throttling, 429 Retry-After timing, transport degradation, buffer management. No race conditions were found across 3 stories of extension. The actor's closure injection pattern (`sendMessage`, `editMessage`, `sendChatAction`) kept it testable and decoupled.

### W5: Registry Pattern Eliminated Command Hardcoding

The `TGCommandRegistry` (Story 32.4) replaced a brittle 3-command switch with an extensible struct. Adding new commands is now a single `register()` call with metadata. The bot menu auto-syncs via `setMyCommands`. This is the first Telegram component that's genuinely extensible without modifying core files.

### W6: Inline Keyboard Approval/Clarify Bridge Works End-to-End

Story 32.5 delivered the most complex feature: `AgentPausedEvent` → `TGInteractiveSessionStore` → inline keyboard → callback query → `answerCallbackQuery` → agent resume. This 8-component pipeline works across SDK boundary (SDK message stream → custom EventBus event → TG adapter → SDK agent.resume). No other story in the project bridges this many layers.

## What Could Be Improved

### C1: CRITICAL: `sendMessage` Closure Returned Void Initially (Story 32.2)

The most dangerous bug of the epic: `sendMessage` closure returned `Void` throughout the entire chain (TGStreamingController → TGEventHandler → TaskSerialQueue → GatewayCommand → TelegramAdapter). This meant `previewMessageId` was never captured, and all edits silently fell to append mode. The streaming feature would have appeared to work (messages were sent) but would never have actually edited a single message.

**Root Cause:** The closure type was defined at GatewayCommand and propagated through 5 files. Nobody verified the return type matched the controller's expectation until adversarial review.

**Recommendation:** Define closure type aliases at the protocol/boundary level, not at the construction site. The controller should define `typealias SendMessageClosure = (String, Int64) async -> Int64?` and every wiring point should match it.

### C2: Double Formatting Bug — Controller and Adapter Both Formatted (Story 32.2)

The `handleAgentCompleted` called `TGMessageFormatter.format()` + `split()`, then the adapter's `sendFormatted`/`editMessage` called format() again, producing double-escaped output (e.g., `\\.` in messages).

**Root Cause:** The controller was designed to format before sending, but the adapter was also designed to format before sending. Two formatting layers were composed without understanding who owned the responsibility.

**Recommendation:** Establish a clear formatting ownership rule: **adapters own formatting; controllers produce raw text.** Document this in the project-context.md anti-patterns.

### C3: Config Fields Added but Not Wired (Story 32.3)

`AxionConfig.tgTypingEnabled` and `tgTypingInterval` were added to the model but never wired to `TGStreamingController`. The controller always used `TGStreamingConfig.default` (hardcoded `typingEnabled: true`). Config fields that exist but don't flow are worse than missing fields — they create a false sense of configurability.

**Recommendation:** When adding config fields, the story must include a wiring verification step: trace the field from model → convenience method → config construction → consumer. No field is "done" until end-to-end flow is verified.

### C4: Review Caught Multiple "Dead Code" Items

Across Stories 32.1–32.5, review found: `balanceCodeBlocks()` dead code (32.1), `escapeOutsideCodeSpans()` defined but never called (32.1), `setMyCommands` placed after blocking `adapter.start()` (32.4). These indicate the AI agent writes code that looks correct at first glance but has subtle dead paths.

**Recommendation:** Add a dead-code detection step to the review checklist: "For each new function/method, grep for callers. If zero callers found, flag for removal or document why it exists."

## Lessons Learned

### L1: Closure Type Aliases Must Be Defined at the Boundary, Not the Construction Site

The `sendMessage` returns `Void` vs `Int64?` bug (Story 32.2 CRITICAL) showed that when closures flow through 5+ files, the return type must be defined at the protocol/boundary level. If `TGStreamingController` had defined `typealias SendMessageClosure = (String, Int64) async -> Int64?`, the mismatch would have been a compile error, not a silent runtime bug.

**Key Insight:** For any closure that flows through 3+ files, define a `typealias` at the consumer and require providers to match it.

### L2: Formatting Ownership Must Be Explicit — Adapters Own Formatting, Controllers Own Raw Text

The double formatting bug (Story 32.2 CRITICAL #2) was caused by unclear ownership. The controller formatted text before sending, but the adapter also formatted it. Both were "correct" in isolation.

**Key Insight:** Establish a project-wide rule: presentation-layer adapters (TelegramAdapter, SDKTerminalOutputHandler) own formatting. Business logic and controllers produce raw text. No double formatting.

### L3: Spike-First Approach for Uncertain APIs Saves Time

The Draft Preview spike (Story 32.3) conclusively proved infeasibility in 1 task, saving 2-3 tasks of dead-end implementation. This validates the Epic's locked decision #4: "Draft Preview must pass technical spike gate before any implementation."

**Key Insight:** When an API's availability is uncertain, invest 15 minutes in a spike before committing implementation tasks. A documented "not feasible" finding is as valuable as a working feature.

### L4: SDK Pause/Resume Bridge Pattern Is Reusable

Story 32.5 created `AgentPausedEvent` as a custom bridge from SDK message stream to EventBus. This pattern (detect SDK-specific message → publish custom AgentEvent → consume in EventHandler) is generic and could be reused for other SDK message types that don't have EventBus equivalents.

**Key Insight:** When the SDK provides data through message streams but not EventBus events, create thin bridge events. Don't fork the SDK.

### L5: Deferred Wiring Pattern Scales Well

The pattern of declaring closures with no-op defaults at construction, then rewiring after adapter creation (introduced in Story 32.2 for `editHandler`), scaled cleanly through Stories 32.3–32.5 (`chatActionHandler`, resume handles). Each new closure follows the same pattern without increasing complexity.

**Key Insight:** The deferred wiring pattern is the right abstraction for actor-based dependency injection in Swift. It avoids initialization order problems while maintaining testability.

### L6: Review Is Essential — All 3 CRITICAL Bugs Were Caught by Adversarial Review

Every CRITICAL bug in this epic (sendMessage return type, double formatting, setMyCommands ordering) was caught by adversarial AI review, not by tests or manual inspection. Tests passed for all of them because they tested behavior, not integration.

**Key Insight:** Tests verify correctness of individual components. Review verifies correctness of component composition. Both are essential.
