## Story 逐一分析

### Story 15.1: Takeover 经验自动学习

**亮点：**
- TakeoverLearningService 纯 Swift struct，无状态，通过构造函数注入 MemoryFactStore 和 MemoryLifecycleService——设计简洁
- 延迟记录策略：不在 Takeover 恢复时立即记录，而是等任务完成后根据最终 outcome 决定 kind（affordance vs avoid）——比 OpenClick 的即时记录更准确
- inferDomain(from:) 辅助方法从 collectedPairs 提取 bundleId，复用已有数据
- MemoryLearnTakeoverCommand CLI 子命令，支持手动记录接管经验——与 OpenClick CLI 对齐
- AC4（Planner prompt 注入）和 AC6（Memory 生命周期集成）不需要额外代码——现有系统自动覆盖
- "反模式提醒"节明确禁止文件轮询、禁止在 TakeoverIO 中添加 Memory 逻辑——边界意识强

**挑战：**
- **HIGH: Takeover learning 与 Memory extraction 在同一 do/catch 中** — Memory extraction 失败会静默跳过 takeover learning。修复：将 takeover learning 移出 do/catch 块。
- **MEDIUM: inferDomain O(n²) 搜索** — 未直接使用 tuple 的 toolUse 字段，效率低。修复：使用 tuple 直接访问。
- **MEDIUM: Cancelled runs 生成错误的 avoid fact** — 任务被取消时也记录了 avoid learning。修复：添加 `runCompleted` 标志，非正常完成的 run 不记录学习。
- **LOW: TakeoverOutcome.cancelled 无测试** — 补充测试覆盖。
- **LOW: resolveMemoryDir() 路径逻辑重复** — 单用途，可接受。

### Story 15.2: Takeover 结构化标记

**亮点：**
- InterventionReason 枚举 15 个 case + classifyReason() 关键词映射器——完整覆盖 OpenClick 的 14 种干预原因 + unknown 兜底
- TakeoverMarker struct 使用 CodingKeys snake_case 映射——与 OpenClick trace JSON 格式一致
- TakeoverIO 提示语改进：引导用户输入反馈文本，feedback 分离逻辑清晰（action == .resume && 非空 → feedback 有值）
- duration 计时使用 ContinuousClock——高精度，不受系统时间调整影响
- 工厂方法 `TakeoverMarker.create(...)` 自动填充 schemaVersion 和 createdAt——减少调用者负担
- 31 个测试（24 TakeoverMarker + 7 TakeoverIO feedback + trace 事件）——覆盖率充分

**挑战：**
- **HIGH: Duration 计算时间范围错误** — 在运行结束后计算 duration，实际测量的是暂停到运行结束的总时间。修复：将 duration 计算移至用户恢复时（`.resume` case）。
- **MEDIUM: takeover_resumed trace 事件在 feedback 为 nil 时丢失** — `as Any` 导致 JSONSerialization 失败。修复：条件构建 payload，feedback 为 nil 时不包含该 key。
- **MEDIUM: Task 5.8 缺少 duration 测试** — 声称完成但无实际测试。修复：提取 `durationToSeconds()` 静态方法，新增 3 个测试。
- **LOW: 测试名误导** — `feedbackNilOnContinue` 实际测试 feedback == "continue"。修复：重命名。
- **LOW: takeoverEvent 元组 reason 字段冗余** — reason 和 issue 同为 pausedData.reason。记录为已知设计选择。

## 跨 Story 模式分析

### 成功模式（值得保持和推广）

**1. "延迟记录"策略优于即时记录**
OpenClick 在 Takeover 恢复时立即记录学习（此时不知道最终 outcome），Axion 延迟到任务完成后记录，能够根据 runSucceeded 准确判断 affordance vs avoid。这是一个经过深思熟虑的架构优化。
**启示：** 当下游数据依赖上游结果时，延迟处理比即时处理更准确。在 AI Agent 系统中，"最终状态"往往比"中间状态"更有价值。

**2. "反模式提醒"节防止架构退化**
两个 Story 都包含 6-7 条反模式提醒，明确禁止文件轮询、禁止新建存储层、禁止在 I/O 层添加业务逻辑。这些边界声明有效防止了 Agent 在实现时偏离既定架构。
**启示：** 在 AI 辅助开发中，"不要做什么"和"要做什么"同等重要。反模式提醒是架构守护的第一道防线。

**3. 现有系统自动覆盖的能力识别**
AC4（Planner prompt 注入）和 AC6（Memory 生命周期）通过分析现有系统 `MemoryContextProvider` 和 `MemoryLifecycleService` 确认无需额外代码。这种"不写代码"的决策避免了不必要的重复实现。
**启示：** 在增量开发中，先确认"已有系统能否自动覆盖"再写新代码，是避免代码膨胀的关键。

**4. Review findings 稳定在 5 个**
15.1 和 15.2 都有 5 个 review findings，且模式相似：一个 HIGH（逻辑错误）+ 两个 MEDIUM（效率/边界情况）+ 两个 LOW（命名/冗余）。一致性表明 Review 标准稳定。
**启示：** 稳定的 review 模式说明开发-审查循环已经成熟，可以预期每个 Story 的 review 结果。

### 挑战模式（需要改进）

**1. HIGH severity findings 反复出现逻辑/计时错误**
15.1 的 do/catch 范围错误和 15.2 的 duration 计算时间范围错误都是核心逻辑问题。两个 Story 的 HIGH finding 都需要重构核心实现路径。
**根因：** 复杂控制流中的边界条件分析不充分——do/catch 块的范围、计时起止点。
**改进：** 对于涉及"时间测量"和"错误传播范围"的代码，在 Dev Notes 中增加时序图和控制流分析。

**2. nil 值在 JSON 序列化中的处理**
15.2 的 `as Any` 导致 nil feedback 在 JSONSerialization 中失败。这是 Swift JSON 处理的常见陷阱。
**根因：** 可选值到 Dictionary 的转换缺乏条件构建。
**改进：** 建立标准模式：可选值在构建 JSON payload 时使用条件包含（`if let` / `compactMap`），避免 `as Any`。

**3. "声称完成但无实际测试"的 Task 仍出现**
15.2 的 Task 5.8（duration 测试）在 Tasks 列表中标记为 [x] 但实际未编写测试。这是从 Epic 14 延续的模式。
**根因：** Tasks checklist 的完成标记基于"意图"而非"验证"。
**改进：** Review 中增加 Tasks checklist 与实际测试文件的交叉验证步骤。

### 突破时刻

**1. InterventionReason 关键词映射的优先级设计**
classifyReason() 中 `2fa`/`login` 必须在 `verif` 之前检查，`modal`/`popup` 必须在 `dialog` 之前检查——这避免了关键词重叠导致的误分类。Debug Log 明确记录了这个顺序修复，说明开发者在实现后实际测试了边界情况。

**2. TakeoverIO 的 feedback 分离逻辑**
一行代码实现 feedback 分离：`let feedback = (action == .resume && userInput非空) ? userInput : nil`。这个设计既保持了 TakeoverIO 的 I/O 纯粹性（只返回输入），又在调用侧实现了业务逻辑分离。

**3. runCompleted 标志的引入**
15.1 的 MEDIUM fix 引入 `runCompleted` 标志，区分"任务完成（成功或失败）"和"任务取消"。这防止了取消的任务生成错误的 avoid 记忆，是一个对系统语义正确性的重要保护。

## 关键经验

1. **"延迟记录"是 AI Agent 系统中一个重要的设计模式** — Axion 的 takeover 学习在任务完成后记录而非恢复时记录，能够利用最终 outcome 做出更准确的分类。这个模式适用于所有"中间状态不确定，但最终状态可确定"的场景——在 AI Agent 中这几乎是所有学习场景的共同特征。

2. **"反模式提醒"是 AI 辅助开发中不可或缺的架构守护手段** — 两个 Story 共 13 条反模式提醒，每条都防止了一种可能的架构退化。在 AI Agent 自动编码的场景下，"告诉它不要做什么"比"告诉它要做什么"更有效地防止了偏离。

3. **Swift Optional → JSON 的转换是高频陷阱** — `as Any` 将 nil 转为 NSNull 导致 JSONSerialization 失败，这个模式在 15.2 中出现。Swift 的类型安全在 JSON 桥接层出现了"最后一公里"的断裂。教训：在构建 JSON Dictionary 时，应始终使用条件包含而非强制转换。

4. **RunCommand 作为"上帝对象"的技术债务正在累积** — 从 Epic 7 到 Epic 15，连续 6 个 Epic 修改 RunCommand，每次添加新的局部状态（takeoverEvent、runSucceeded、runCompleted、takeoverStartTime）。这个类已经超过了"可安全修改"的阈值。教训：当同一个文件在连续 3+ 个 Epic 中被修改时，应该主动重构而非继续堆积。

5. **Epic 15 证明了"Memory 先"的策略有效** — 先在 Epic 12 建立 Memory 生命周期（candidate→active→retired），再在 Epic 15 利用这个生命周期实现 takeover 学习。Epic 15 的 AC4 和 AC6 不需要额外代码，直接复用已有系统。这种"基础设施先于功能"的策略比"边做边补"效率高得多。
